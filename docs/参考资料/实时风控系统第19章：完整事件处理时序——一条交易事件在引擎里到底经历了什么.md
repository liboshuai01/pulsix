## 第19章：完整事件处理时序——一条交易事件在引擎里到底经历了什么

### 19.1 这一章解决什么问题

前面第 11 到第 18 章，我们已经分别把 Flink 在实时风控系统中的角色、快照接收与切换、特征系统、状态设计、规则与策略执行、表达式与 Groovy 的边界，以及执行器代码结构讲清楚了。

但这些内容大多还是“按模块拆开讲”的。

真正到了工程落地阶段，你最需要建立的是另一种能力：

> **把这些模块在一条真实事件上的协作过程，完整地串成一条可执行、可解释、可落地的时序链路。**

也就是说，你必须真正回答下面这些问题：

- 一条交易事件进入系统后，第一步到底做什么？
- Flink 是先读快照，还是先算特征？
- 流式特征、Redis lookup、派生特征、规则执行的顺序是什么？
- 哪一步在用 Broadcast State？
- 哪一步在用 Keyed State？
- 哪一步会用到本地编译缓存？
- 决策结果到底长什么样？
- 哪些数据在热路径里，哪些是异步输出？
- 如果规则命中、Redis 超时、脚本异常、版本切换，会发生什么？

所以这一章的目标，就是把整套 Flink 决策引擎的运行过程，压缩到**一条具体交易事件**上，从输入到输出完整走一遍。

你可以把这一章理解成：

> **这是前面 Flink 章节的“总装配图 + 总时序图”。**

如果这一章看懂了，你对整个脉流实时风控平台的运行理解，就会从“知道每个模块做什么”，升级成“知道它们是怎么一起工作的”。

---

### 19.2 先定义一个完整的示例场景

为了让这一章讲解足够具体，我们先固定一个例子，后面所有时序都围绕这个例子展开。

#### 场景

- `sceneCode = TRADE_RISK`
- 场景名称：交易风控

#### 当前线上生效版本

- `snapshotId = TRADE_RISK_v12`
- `version = 12`

#### 事件

```json
{
  "eventId": "E202603070001",
  "traceId": "T202603070001",
  "sceneCode": "TRADE_RISK",
  "eventType": "trade",
  "eventTime": "2026-03-07T10:00:00",
  "userId": "U1001",
  "deviceId": "D9001",
  "ip": "1.2.3.4",
  "amount": 6800,
  "currency": "CNY",
  "result": "SUCCESS",
  "channel": "APP"
}
```

#### 当前快照中定义的核心特征

##### 流式特征

- `user_trade_cnt_5m`
- `user_trade_amt_sum_30m`
- `device_bind_user_cnt_1h`

##### lookup 特征

- `device_in_blacklist`
- `user_risk_level`

##### 派生特征

- `high_amt_flag = amount >= 5000`
- `trade_burst_flag = user_trade_cnt_5m >= 3 && amount >= 5000`

#### 当前快照中的规则

- `R001: device_in_blacklist == true -> REJECT`
- `R002: trade_burst_flag == true -> REVIEW`
- `R003: device_bind_user_cnt_1h >= 4 && ['M','H'].contains(user_risk_level) -> REJECT`

#### 当前策略

- `decisionMode = FIRST_HIT`
- `ruleOrder = [R001, R003, R002]`
- `defaultAction = PASS`

在这个示例里，如果：

- 设备不在黑名单
- 但设备 1 小时关联账号数 >= 4
- 且用户风险等级 = H

那么最终就会命中 `R003`，产出 `REJECT`。

### 19.3 在事件真正到来之前，Flink 端已经处于什么状态

在讲 `processElement(event)` 之前，你必须先理解一个前提：

> **当业务事件真正进来时，Flink 引擎并不是“从零开始想怎么处理”，而是已经提前准备好了运行态环境。**

也就是说，在事件进来之前，通常已经发生过下面这些事：

#### 1）控制平台已完成发布

控制平台已经把设计态配置编译为 `TRADE_RISK_v12` 快照。

#### 2）Flink 已消费到配置流

Flink 的 `processBroadcastElement(snapshotEnvelope)` 已经执行过。

#### 3）Broadcast State 中已保存当前场景快照

例如：

- key = `TRADE_RISK`
- value = `SceneSnapshot(version=12, ...)`

#### 4）Task 本地已构建运行时缓存

例如：

- 已编译表达式对象
- 已编译 Groovy 脚本
- 特征执行计划
- rule 执行顺序
- policy runtime

#### 5）Keyed State 中已累积历史状态

例如：

- 用户近 5 分钟交易桶状态
- 用户近 30 分钟交易金额桶状态
- 设备近 1 小时关联用户状态

所以，一条事件真正到来时，Flink 面对的不是“空白世界”，而是：

> **一个已经加载了活跃快照、已经积累了历史状态、已经编译好执行逻辑的运行时系统。**

这也是为什么实时决策可以做到低延迟。

---

### 19.4 事件进入系统前的完整上游路径

我们先从 Flink 外部开始，看事件如何进入引擎。

#### 步骤 1：业务系统产生交易事件

例如：

- 前端 / App 埋点通过 `HTTP / Beacon -> Nginx -> pulsix-ingest`
- 业务后端通过 `Pulsix SDK(Netty) -> pulsix-ingest`

在 `pulsix-ingest` 中完成鉴权、标准化和公共字段补齐后，组装出标准事件。

#### 步骤 2：事件写入 Kafka 标准事件 Topic（`pulsix.event.standard`）

例如：

- `pulsix.event.standard`

此时，事件已经进入实时处理主链路。

#### 步骤 3：Flink Source 消费 Kafka 消息

Flink 从 Kafka 拿到这条消息，做：

- 反序列化
- 基础合法性检查
- 转换为标准 `RiskEvent`

如果失败：

- 进入死信 topic / 错误流
- 不进入正常决策主链路

这一阶段结束后，Flink 拿到的是一个标准事件对象，而不是原始字符串。

---

### 19.5 事件进入 Flink 后的完整主链路时序

下面开始进入真正的核心：

> **一条事件在 Flink 决策引擎内部到底经历了什么。**

为了便于理解，我把它拆成 12 个连续步骤。

---

## 19.5.1 第一步：事件进入 `processElement`

如果你的主链路采用 `KeyedBroadcastProcessFunction<SceneKey, RiskEvent, SnapshotEnvelope, DecisionResult>`，那么事件进入的是：

```java
processElement(RiskEvent event, ReadOnlyContext ctx, Collector<DecisionResult> out)
```

此时拿到的只有事件本身。第一件事不是执行规则，而是做运行时路由。

---

## 19.5.2 第二步：根据 sceneCode 找当前活跃快照

Flink 从事件中拿到：

- `sceneCode = TRADE_RISK`

然后在 Broadcast State 或本地运行时缓存中查找该场景当前活跃版本。

实际工程里通常是两级结构：

### 第一级：Broadcast State

保存可恢复的 `SceneSnapshot`。

### 第二级：本地 transient cache

保存不可序列化但高性能的：

- 编译表达式
- 编译 Groovy
- 执行计划对象

所以此时通常会得到一个对象，例如：

```java
CompiledSceneRuntime runtime = runtimeManager.getActiveRuntime("TRADE_RISK");
```

如果拿不到：

- 可以打错误日志
- 可以把事件打到异常 topic
- 不建议直接让任务失败

这一阶段的关键点是：

> **事件处理首先必须绑定到某个明确的运行版本。**

否则后面所有特征、规则、策略都没有上下文。

---

## 19.5.3 第三步：构建基础上下文 `EvalContext`

接下来，引擎会把事件中的基础字段装进一个统一上下文对象。

例如：

```java
EvalContext ctx = new EvalContext();
ctx.put("eventId", event.getEventId());
ctx.put("traceId", event.getTraceId());
ctx.put("sceneCode", event.getSceneCode());
ctx.put("eventType", event.getEventType());
ctx.put("eventTime", event.getEventTime());
ctx.put("userId", event.getUserId());
ctx.put("deviceId", event.getDeviceId());
ctx.put("ip", event.getIp());
ctx.put("amount", event.getAmount());
ctx.put("result", event.getResult());
ctx.put("channel", event.getChannel());
```

这一步非常基础，但非常重要。因为：

- 后面的特征、规则、派生表达式都依赖这个上下文
- 它也是命中原因模板渲染的输入之一
- 仿真测试与线上执行，最终都应该围绕同一个 `EvalContext` 模型

你可以把 `EvalContext` 理解为：

> **当前这条事件在某个版本规则体系下的运行时变量容器。**

---

## 19.5.4 第四步：根据快照中的执行计划，识别当前需要哪些特征

并不是所有场景的所有特征，都需要对每条事件计算。

举例来说，这条交易事件只需要本次策略真正依赖的特征，例如：

### stream feature

- `user_trade_cnt_5m`
- `user_trade_amt_sum_30m`
- `device_bind_user_cnt_1h`

### lookup feature

- `device_in_blacklist`
- `user_risk_level`

### derived feature

- `high_amt_flag`
- `trade_burst_flag`

这些信息不是临时猜出来的，而应该来自快照中的 `runtimeHints / featurePlan / dependsOn`。

这一阶段本质上是在做：

> **基于当前快照，确定当前事件的最小执行闭包。**

这样做的好处是：

- 不会多算无关特征
- 不会多查无关 Redis key
- 性能更稳定
- 执行链路更可解释

---

## 19.5.5 第五步：计算流式特征

这是 Flink 引擎最核心的一步之一。

对于当前示例，假设需要计算：

- `user_trade_cnt_5m`
- `user_trade_amt_sum_30m`
- `device_bind_user_cnt_1h`

### 1）计算 `user_trade_cnt_5m`

引擎会根据特征定义：

- entityKeyExpr = `userId`
- aggType = `COUNT`
- window = `5m`
- filterExpr = `result == 'SUCCESS'`

得到当前 key：

- `U1001`

然后在对应的 Keyed State 中：

- 找到按分钟切桶的 bucket 状态
- 更新当前分钟桶
- 清理过期桶
- 汇总窗口范围内值

例如最终得到：

```latex
user_trade_cnt_5m = 3
```

### 2）计算 `user_trade_amt_sum_30m`

同理：

- entityKey = `U1001`
- 读取 30 分钟窗口相关桶
- 当前事件 amount=6800 进入当前桶
- 聚合求和

例如得到：

```latex
user_trade_amt_sum_30m = 18800
```

### 3）计算 `device_bind_user_cnt_1h`

这类特征通常更复杂一些，因为是：

- 维度 = device
- 聚合 = distinct\_count(userId)

如果当前设备 `D9001` 最近 1 小时已出现多个不同 userId，则会读取对应状态结构，更新当前 userId，最终得到：

```latex
device_bind_user_cnt_1h = 4
```

### 为什么这一步必须在规则执行之前完成

因为规则本身并不会去做“流式状态演化”，它只消费结果。

也就是说：

> **Flink 在这里做的是“先构建状态型上下文，再把它交给规则层”。**

### 这一阶段的数据存放位置

- 中间态：Flink Keyed State
- 当前事件计算结果：写入 `EvalContext`

例如：

```java
ctx.put("user_trade_cnt_5m", 3L);
ctx.put("user_trade_amt_sum_30m", new BigDecimal("18800"));
ctx.put("device_bind_user_cnt_1h", 4L);
```

---

## 19.5.6 第六步：执行 lookup，查询 Redis 名单 / 画像 / 热点信息

流式特征算完以后，下一步通常是 lookup 特征。

当前示例需要查：

- `device_in_blacklist`
- `user_risk_level`

### 1）查询设备黑名单

根据快照定义：

- lookupType = `REDIS_SET` / `REDIS_STRING`
- keyExpr = `deviceId`
- redisKeyPrefix = `blacklist:device:`

引擎拼出 key，例如：

```latex
blacklist:device:D9001
```

假设查出来结果为：

```latex
device_in_blacklist = false
```

### 2）查询用户风险等级

根据快照：

- lookupType = `REDIS_HASH` / `REDIS_STRING`
- keyExpr = `userId`
- prefix = `profile:user:risk:`

拼出 key：

```latex
profile:user:risk:U1001
```

查得：

```latex
user_risk_level = H
```

然后同样写入 `EvalContext`：

```java
ctx.put("device_in_blacklist", false);
ctx.put("user_risk_level", "H");
```

### 这一阶段的关键点

#### 1）lookup 是上下文补充，不是规则判断

也就是说，Redis 查出来的结果也是“变量”，不是“结果”。

#### 2）lookup 失败必须有兜底

例如：

- 超时用默认值
- 查询异常打监控
- 严格场景可做降级策略

#### 3）不要在规则里直接写 Redis 操作

规则层应该只消费变量，不应该承担 IO。

---

## 19.5.7 第七步：计算派生特征

此时，基础字段、流式特征、lookup 特征都已经准备好了。于是进入派生特征阶段。

当前示例要算两个派生特征。

### 1）计算 `high_amt_flag`

表达式：

```latex
amount >= 5000
```

根据上下文：

- amount = 6800

结果：

```latex
high_amt_flag = true
```

### 2）计算 `trade_burst_flag`

表达式：

```latex
user_trade_cnt_5m >= 3 && amount >= 5000
```

根据上下文：

- user\_trade\_cnt\_5m = 3
- amount = 6800

结果：

```latex
trade_burst_flag = true
```

然后继续写入 `EvalContext`：

```java
ctx.put("high_amt_flag", true);
ctx.put("trade_burst_flag", true);
```

### 为什么派生特征要单独成为一层

因为它的作用是：

- 把复杂规则拆成中间语义变量
- 提升规则可读性
- 支持复用
- 降低规则层表达式复杂度

很多成熟系统都会把它放在“规则执行前的一层预计算逻辑”中。

---

## 19.5.8 第八步：执行规则

现在终于进入用户最直观理解的“规则执行”阶段。

但你要非常清楚：

> **规则执行并不是整条链路的开始，而是上下文准备完之后才发生的一步。**

当前示例规则为：

### R001

```latex
device_in_blacklist == true
```

代入上下文：

- device\_in\_blacklist = false

结果：

- `hit = false`

### R003

```groovy
return device_bind_user_cnt_1h >= 4 && ['M','H'].contains(user_risk_level)
```

代入上下文：

- device\_bind\_user\_cnt\_1h = 4
- user\_risk\_level = H

结果：

- `hit = true`

### R002

```latex
trade_burst_flag == true
```

代入上下文：

- trade\_burst\_flag = true

结果：

- `hit = true`

### 命中信息通常会形成什么结构

每条规则执行后，通常会生成一个 `RuleHitDetail`：

```json
{
  "ruleCode": "R003",
  "hit": true,
  "action": "REJECT",
  "score": 80,
  "reason": "设备1小时关联用户数=4, 用户风险等级=H"
}
```

### 执行顺序的重要性

规则执行并不只是“全部算一遍”。
它的顺序、是否短路、是否保留全部命中结果，都由策略层决定。

在当前示例里，策略顺序是：

- R001
- R003
- R002

所以虽然 R002 也能命中，但如果策略是 `FIRST_HIT`，则最终只会采用最先命中的高优先级规则。

---

## 19.5.9 第九步：执行策略收敛

规则执行得到的是“命中集合”，而最终业务需要的是“一个结果”。

这就需要策略层来完成收敛。

### 当前示例策略：FIRST\_HIT

顺序：

- R001
- R003
- R002

执行结果：

- R001 未命中
- R003 命中
- 因为是 FIRST\_HIT，所以到这里直接结束

最终动作：

```latex
REJECT
```

最终原因：

```latex
设备1小时关联用户数=4, 用户风险等级=H
```

如果当前策略不是 `FIRST_HIT`，而是 `SCORE_CARD`，则逻辑会变成：

- R003 得 80 分
- R002 得 60 分
- 总分 = 140
- 最终按分段规则得 `REJECT`

也就是说：

> **规则层提供局部判断，策略层决定最终收敛方式。**

这是必须分层的。

---

## 19.5.10 第十步：组装最终 `DecisionResult`

当前事件的最终输出，应该是一个结构化对象，而不是简单字符串。

例如：

```json
{
  "eventId": "E202603070001",
  "traceId": "T202603070001",
  "sceneCode": "TRADE_RISK",
  "version": 12,
  "snapshotId": "TRADE_RISK_v12",
  "finalAction": "REJECT",
  "finalScore": 80,
  "hitRules": ["R003"],
  "allEvaluatedRules": [
    {"ruleCode": "R001", "hit": false},
    {"ruleCode": "R003", "hit": true},
    {"ruleCode": "R002", "hit": true}
  ],
  "reason": "设备1小时关联用户数=4, 用户风险等级=H",
  "latencyMs": 37,
  "decisionTime": "2026-03-07T10:00:00.037"
}
```

很多系统还会同时保留：

- 输入事件摘要
- feature snapshot
- policyCode
- engine diagnostics

你后面做日志查询页时，这个结构会非常重要。

---

## 19.5.11 第十一步：输出决策结果到下游

决策结果形成以后，热路径通常会做两类输出。

### 输出 1：业务结果流

发到：

- `pulsix.decision.result`

供：

- 下游业务服务
- 审核系统
- 标签系统
- 风险告警系统

### 输出 2：日志流

发到：

- `pulsix.decision.log`

供：

- 查询
- 分析
- 追溯
- 监控

### 为什么建议异步输出

因为热路径最重要的是：

- 低延迟
- 稳定
- 不被日志落库阻塞

所以一般不建议在热路径里同步写多张 MySQL 表。

---

## 19.5.12 第十二步：输出指标与异常信息

除了业务结果，Flink 通常还会输出或累计以下指标：

- 场景维度 TPS
- 决策成功数
- 决策失败数
- Redis lookup 超时数
- 规则执行异常数
- 脚本执行异常数
- latency P50/P95
- 当前快照版本号标签

这些指标供：

- Prometheus
- Grafana
- 告警系统

也就是说，一条事件处理结束后，不只是得到了一个业务结果，整个系统还获得了：

- 可追溯日志
- 可观测指标
- 可分析行为轨迹

---

### 19.6 用一张“上下文演进图”理解整条链路

很多人读完上面的步骤后，还是会觉得信息很多。这里我把它简化成一个上下文逐步充实的过程。

#### 阶段 1：原始事件

```json
{
  "eventId": "E202603070001",
  "sceneCode": "TRADE_RISK",
  "userId": "U1001",
  "deviceId": "D9001",
  "amount": 6800,
  "result": "SUCCESS"
}
```

#### 阶段 2：加入流式特征

```json
{
  "user_trade_cnt_5m": 3,
  "user_trade_amt_sum_30m": 18800,
  "device_bind_user_cnt_1h": 4
}
```

#### 阶段 3：加入 lookup 特征

```json
{
  "device_in_blacklist": false,
  "user_risk_level": "H"
}
```

#### 阶段 4：加入派生特征

```json
{
  "high_amt_flag": true,
  "trade_burst_flag": true
}
```

#### 阶段 5：执行规则

```json
[
  {"ruleCode": "R001", "hit": false},
  {"ruleCode": "R003", "hit": true},
  {"ruleCode": "R002", "hit": true}
]
```

#### 阶段 6：策略收敛

```json
{
  "finalAction": "REJECT",
  "hitRule": "R003"
}
```

这其实就是整个实时决策链路最本质的过程：

> **事件 -> 上下文 -> 规则命中集 -> 最终决策**

### 19.7 一段接近真实工程的伪代码

下面给你一段很贴近真实引擎实现的伪代码，帮助你把上述时序落到代码脑图里。

```java
public void processElement(RiskEvent event, ReadOnlyContext ctx, Collector<DecisionResult> out) throws Exception {
    long start = System.currentTimeMillis();

    // 1. 获取场景运行时
    CompiledSceneRuntime runtime = runtimeManager.getActiveRuntime(event.getSceneCode());
    if (runtime == null) {
        emitError(event, "NO_ACTIVE_RUNTIME");
        return;
    }

    // 2. 构建基础上下文
    EvalContext evalCtx = EvalContext.fromEvent(event);

    // 3. 计算 stream features
    for (StreamFeatureExecutor executor : runtime.getFeaturePlan().getStreamExecutors()) {
        Object value = executor.compute(event, ctx, stateStore);
        evalCtx.put(executor.featureCode(), value);
    }

    // 4. lookup features
    for (LookupFeatureExecutor executor : runtime.getFeaturePlan().getLookupExecutors()) {
        Object value = executor.lookup(evalCtx);
        evalCtx.put(executor.featureCode(), value);
    }

    // 5. derived features
    for (DerivedFeatureInvoker invoker : runtime.getFeaturePlan().getDerivedInvokersInTopoOrder()) {
        Object value = invoker.eval(evalCtx);
        evalCtx.put(invoker.featureCode(), value);
    }

    // 6. 执行规则
    List<RuleHitDetail> hits = new ArrayList<>();
    for (RuleInvoker ruleInvoker : runtime.getPolicyRuntime().orderedRules()) {
        RuleEvalResult r = ruleInvoker.eval(evalCtx);
        hits.add(r.toHitDetail());
        if (runtime.getPolicyRuntime().isFirstHitMode() && r.isHit()) {
            break;
        }
    }

    // 7. 策略收敛
    DecisionResult result = runtime.getPolicyRuntime().decide(event, evalCtx, hits);
    result.setVersion(runtime.getSnapshot().getVersion());
    result.setLatencyMs(System.currentTimeMillis() - start);

    // 8. 输出结果
    out.collect(result);

    // 9. 异步输出日志/指标
    decisionLogSink.emit(result, evalCtx, hits);
    metrics.record(result);
}
```

你会发现，这段代码其实就是前面整章内容的程序化表达。

### 19.8 如果当前版本切换发生在事件处理中，会怎样？

这是一个很实际的问题。

在真实系统中，配置流和事件流是并行到达的。那如果：

- 事件正在处理
- 新版本快照也刚好到达

会发生什么？

推荐的原则是：

> **单条事件在开始处理时绑定一个运行时版本，整个处理过程保持一致；新版本只影响后续事件。**

也就是说：

- 当前事件开始时拿到 `TRADE_RISK_v12`
- 即使处理过程中收到 `v13`
- 当前事件仍按 `v12` 跑完
- 后续新事件再切到 `v13`

这样做的好处是：

- 单条事件结果可解释
- 不会出现半路切版本
- 日志里的 version 清晰一致

### 19.9 如果某一步失败，会怎么处理？

在完整时序中，失败处理也是必须设计的。

---

## 19.9.1 快照不存在

情况：

- sceneCode 没有活跃版本

处理建议：

- 输出错误日志
- 事件进入异常流
- 不建议让任务直接 fail

---

## 19.9.2 Redis lookup 超时

情况：

- 名单查询超时
- 画像查询失败

处理建议：

- 使用默认值
- 记录超时指标
- 对极端高风险场景可触发降级动作

---

## 19.9.3 表达式执行异常

例如：

- 类型不匹配
- 变量缺失

处理建议：

- 当前规则判定为失败或不命中
- 记录规则异常指标
- 保证整个事件处理不被一条规则拖垮

---

## 19.9.4 Groovy 执行异常

例如：

- 脚本 bug
- 非法访问
- 运行时异常

处理建议：

- 当前脚本规则失败
- 记录错误日志与版本号
- 根据场景决定是否降级为 REVIEW / PASS / 拒绝执行结果

---

## 19.9.5 决策结果输出失败

如果输出 Kafka / sink 失败：

- 通常依赖 Flink sink 的重试和 checkpoint 机制
- 业务侧不要同步阻塞

### 19.10 这条时序里，Flink State、Redis、Kafka、MySQL 各自出现在哪

这一节专门把存储职责和时序绑定起来。

#### Kafka

出现两次：

- 事件输入
- 决策结果/日志输出

#### Broadcast State

出现于：

- 当前活跃快照读取

#### Keyed State

出现于：

- stream feature 计算
- bucket 更新
- distinct 计数
- timer 清理

#### Redis

出现于：

- 名单 lookup
- 画像 lookup
- 热点补充信息 lookup

#### MySQL

不在热路径主线中直接参与逐条决策。
主要参与：

- 控制平台配置管理
- 发布持久化
- 日志异步查询存储

这就是为什么我们一直强调：

> **MySQL 属于设计态/查询态存储，不属于实时决策热路径。**

### 19.12 本章小结

这一章的核心目标，是把前面所有 Flink 相关章节串成“一条真实事件时序”。

你需要记住的最关键内容有：

#### 1）事件处理前，运行时环境已经准备好

- 活跃 snapshot 已加载
- 本地编译缓存已准备
- 历史状态已存在

#### 2）一条事件进入后，不是直接执行规则

而是先经历：

- 版本绑定
- 基础上下文构建
- stream feature 计算
- lookup 特征查询
- 派生特征计算

#### 3）规则层负责局部判断，策略层负责最终收敛

- Rule = 单点判断
- Policy = 规则组织与结果收敛

#### 4）热路径和日志分析路径要分离

- 决策结果尽量异步输出
- 日志不要阻塞主链路

#### 5）单条事件应绑定单一版本执行到底

不要中途切版本。

#### 6）这条时序就是你后面代码实现的总骨架

你写 Flink 引擎时，所有核心模块都应该围绕这条时序组织。
