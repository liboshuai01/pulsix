## 第23章：测试体系——单元测试、仿真测试、回放测试、联调测试

### 23.1 这一章解决什么问题

前面我们已经把这套实时风控平台从系统认知、控制平台建模、发布快照、Flink 执行链路、特征体系、规则策略执行、代码模块拆分等方面讲得比较完整了。到这里，很多人会自然进入一个误区：

> 架构讲清楚了，代码也写出来了，项目应该就差不多完成了。

但真实工程里， **没有测试体系的实时风控系统，几乎等于没有真正交付能力**。

原因很简单：

- 你做的是风控，不是普通 CRUD；
- 一条规则写错，可能会误杀大量正常用户；
- 一次发布处理不当，可能导致线上延迟飙升或规则全部失效；
- 一次特征窗口变更，可能让历史状态解释发生偏差；
- 一个看似无害的表达式改动，可能让决策结果整体漂移。

所以这一章要解决的是：

1. 这套实时风控平台应该建立什么样的测试体系；
2. 单元测试、组件测试、仿真测试、回放测试、联调测试分别测什么；
3. Flink 有状态逻辑怎么测；
4. 控制平台与引擎的一致性怎么测；
5. 怎样设计测试数据、测试场景、回归机制；
6. 怎样把测试能力真正变成项目交付质量的一部分。

你可以把这一章理解为：

> 它不是在讲“测试补充”，而是在讲“如何证明这套系统真的可靠”。

---

### 23.2 为什么实时风控平台的测试难度远高于普通业务系统

普通后台系统的测试难点，通常在于：

- 接口逻辑是否正确；
- 数据库读写是否符合预期；
- 权限和状态流转是否正确；
- 页面操作是否闭环。

而实时风控平台除了这些问题，还多了下面这些难点：

#### 1）规则是动态配置的

不是代码里写死的 if-else，而是运行时热切换的逻辑。

#### 2）Flink 是有状态的

测试不能只看输入输出，还要看：

- 状态是否正确累积；
- timer 是否正确触发；
- checkpoint 恢复后是否一致。

#### 3）同一条事件的结果依赖上下文

一条交易是否命中规则，不只取决于当前金额，还取决于：

- 过去几分钟发生了什么；
- Redis 画像是什么；
- 当前活跃快照是哪一版。

#### 4）控制平台和运行时引擎必须一致

仿真页如果算出来是 PASS，线上却是 REJECT，这个平台就失去可信度。

#### 5）版本切换会改变系统行为

测试不仅要测“当前版本正确不正确”，还要测：

- 新旧版本差异是否符合预期；
- 回滚后是否恢复正确；
- 快照切换时是否平滑。

所以这类系统必须建立**分层测试体系**，而不能只靠几个接口测试或者手工页面点点看。

---

### 23.3 实时风控平台推荐的测试分层模型

我建议你把测试体系按下面 6 层来理解：

1. **单元测试**：验证最小逻辑单元；
2. **组件测试**：验证模块内协作，例如编译器、执行器、特征计算器；
3. **仿真测试**：验证某个场景/版本下的决策结果；
4. **回放测试**：验证新旧版本在历史数据上的差异；
5. **联调测试**：验证控制平台、Kafka、Redis、Flink、日志链路是否真正打通；
6. **非功能测试**：验证性能、延迟、恢复、容错、观测能力。

你也可以把它理解成一棵从下到上的测试金字塔：

- 底层是大量稳定的单元测试；
- 中层是组件测试和仿真测试；
- 上层是少量但关键的联调测试和回放测试；
- 旁边再补充性能与恢复测试。

这比“全部靠端到端测试”要更稳健，也更适合个人项目持续演进。

---

### 23.4 第一层：单元测试——验证最小逻辑单元

单元测试的目标不是证明系统全通，而是保证**最核心的逻辑积木是可靠的**。

对于你的项目，我建议重点覆盖下面这些单元。

#### 23.4.1 表达式执行器

例如：

- Aviator 表达式是否能正确执行；
- null 值处理是否符合预期；
- 数值类型比较是否稳定；
- 布尔短路逻辑是否正确；
- 错误表达式是否能抛出明确异常。

示例测试点：

- `amount >= 5000 && user_trade_cnt_5m >= 3`
- `device_in_blacklist == true`
- `user_risk_level in ['M','H']`

#### 23.4.2 Groovy 执行器

要验证：

- 编译是否成功；
- Binding 注入是否正确；
- 返回值类型是否正确；
- 脚本异常是否被包装；
- 不安全 API 是否被禁止。

#### 23.4.3 规则执行器

要验证：

- 命中 / 未命中是否正确；
- hitReason 是否正确渲染；
- score 是否正确返回；
- priority 字段是否不影响单条 rule eval 结果。

#### 23.4.4 策略执行器

分别测试：

- FIRST\_HIT 模式；
- SCORE\_CARD 模式；
- 默认动作；
- 规则全未命中的情况；
- 多规则同时命中时的顺序和收敛逻辑。

#### 23.4.5 Snapshot 编译器

要验证：

- 设计态对象能否被正确编译为 SceneSnapshot；
- 依赖关系是否正确收集；
- 缺失 feature/rule 时是否抛出错误；
- checksum 是否稳定。

#### 23.4.6 依赖分析器

要验证：

- rule 依赖哪些 feature；
- derived feature 是否存在循环依赖；
- 基础字段、feature、list 引用是否能区分。

#### 23.4.7 Feature 计算器

你后面为 Flink 抽象出来的 COUNT、SUM、MAX、LATEST、DISTINCT 计算器，建议都做纯 Java 层的单元测试。

例如：

- 连续输入 3 条事件后 count 是否为 3；
- 时间桶滚动后旧 bucket 是否被清理；
- MAX 是否只保留最大值；
- DISTINCT 的去重是否正确。

#### 23.4.8 命中原因模板渲染器

例如模板：

```latex
用户5分钟交易次数={user_trade_cnt_5m}, amount={amount}
```

要验证：

- 占位符替换；
- 缺失变量时的容错；
- 特殊字符处理；
- null 值渲染策略。

---

### 23.5 第二层：组件测试——验证模块内部协作

单元测试保证的是“砖块没问题”，但还不能证明“墙没问题”。
所以还需要组件测试。

组件测试通常针对一个较完整的模块，例如：

- 发布编译模块；
- 仿真执行模块；
- Flink 本地 runtime 编译模块；
- Redis lookup 模块；
- 决策日志组装模块。

#### 23.5.1 发布编译组件测试

给定一套设计态配置：

- scene
- features
- rules
- policy

验证：

- 能否成功生成 snapshot；
- 是否得到正确的 requiredStreamFeatures / requiredLookupFeatures；
- ruleOrder 是否正确；
- 错误配置是否被正确拦截。

#### 23.5.2 仿真执行组件测试

输入：

- SceneSnapshot
- 模拟事件
- mock lookup 结果
- mock stream feature 值

输出：

- final action
- hit rules
- feature snapshot
- hit reasons

这一层测试非常重要，因为后面线上 Flink 也应该尽可能复用这套 rule-core / policy-core。

#### 23.5.3 Redis lookup 组件测试

验证：

- key 拼接是否正确；
- defaultValue 是否正确回退；
- 超时是否兜底；
- 类型转换是否稳定。

#### 23.5.4 本地编译缓存组件测试

验证：

- SceneSnapshot 是否能编译出 CompiledSceneRuntime；
- 新版本加载后旧版本是否还能按策略释放；
- 编译失败是否不会污染当前可用版本；
- 本地缓存与 Broadcast State 的版本一致性。

### 23.6 第三层：仿真测试——控制平台最关键的“可验证能力”

仿真测试不是普通测试辅助工具，而是风控平台非常核心的一项能力。

它的本质是：

> 在不依赖真实线上流量的情况下，基于某个版本快照，对一条模拟事件进行完整决策，并返回全链路解释结果。

#### 23.6.1 仿真测试要解决什么问题

- 规则改完以后是否符合预期；
- 新版本发布前是否安全；
- 命中逻辑是否可解释；
- 仿真结果是否与运行时一致；
- Demo 演示时能否清晰展示平台价值。

#### 23.6.2 仿真测试的输入

至少包括：

- `sceneCode`
- `versionNo` 或“当前草稿快照”
- `eventJson`
- 可选的 mock lookup 数据
- 可选的 mock stream feature 值

#### 23.6.3 仿真测试的输出

至少包括：

- `finalAction`
- `finalScore`
- `hitRules`
- `featureSnapshot`
- `hitReasons`
- `usedVersion`
- `durationMs`

#### 23.6.4 仿真测试最重要的原则

> **仿真与线上必须尽量共用同一套规则执行内核。**

也就是说：

- rule-core 不能线上一套、后台一套；
- hit reason 渲染不能线上一个逻辑、仿真另一个逻辑；
- policy 执行模式必须一致。

否则很容易出现：

- 仿真 PASS，线上 REJECT；
- 仿真命中 2 条，线上命中 1 条；
- 仿真分值 80，线上分值 60。

这类问题会严重伤害平台可信度。

#### 23.6.5 仿真测试和 Flink 的关系

仿真测试通常不直接跑完整 Flink 流。它更多是：

- 复用同样的快照结构；
- 复用同样的表达式/Groovy 执行器；
- 复用同样的规则与策略执行器；
- 流式特征值由 mock 或历史取值注入。

也就是说，仿真本质上是：

- **控制平台的“离线决策执行器”** ，
- 而不是一个假的页面演示。

### 23.7 第四层：回放测试——比较新旧版本在历史数据上的差异

回放测试是平台“从能用到像生产系统”的重要分水岭。

它解决的问题是：

> 新版本规则上线前，我们能不能先拿历史事件流跑一遍，看看结果会发生什么变化？

#### 23.7.1 为什么回放测试很重要

因为很多规则变更，肉眼看表达式很简单，但真实影响可能很大。
例如：

- 金额阈值从 5000 改成 3000；
- user\_trade\_cnt\_5m >= 3 改成 >= 2；
- 新增一条黑名单优先级更高的规则。

这些改动可能导致：

- 拒绝率明显上升；
- 审核量翻倍；
- 某类用户被误杀；
- 原本通过的交易变成拒绝。

回放测试的价值就是在上线前先看到这些差异。

#### 23.7.2 回放测试的典型输入

- 历史事件文件 / Kafka 历史分区数据；
- old snapshot version；
- new snapshot version；
- 可选历史 lookup 快照或当前 lookup 数据。

#### 23.7.3 回放测试的典型输出

- old version 命中结果；
- new version 命中结果；
- 差异事件列表；
- 差异规则明细；
- 拒绝率 / 审核率 / 命中率对比；
- 差异摘要报告。

#### 23.7.4 回放测试的难点

- 要不要按 event time 回放；
- 流式特征是否需要按原时间顺序恢复；
- lookup 画像是否需要历史版本；
- 历史流量是否完整；
- 如何去重与保持事件顺序。

#### 23.7.5 你项目里的建议做法

个人项目不一定要一开始就做全量历史 Kafka 回放。可以先做一个相对务实的版本：

- 从文件读取一批标准事件 JSON；
- 按时间排序；
- 依次用 old/new 两个 snapshot 执行；
- 输出 diff 报告。

这样已经非常有展示价值。

### 23.8 第五层：Flink 有状态逻辑如何测试

这是你这类项目和普通后端项目最大的不同点之一。

Flink 不是普通同步方法调用，它有：

- 状态；
- timer；
- watermark；
- broadcast config；
- checkpoint 恢复。

所以你需要针对 Flink 设计专门测试方法。

#### 23.8.1 流式特征测试的核心关注点

例如对于 `user_trade_cnt_5m`，你要验证：

- 同一个 user 连续 3 条交易，count 是否按预期累加；
- 超出窗口后旧 bucket 是否过期；
- 不同 user 是否隔离；
- result != SUCCESS 的事件是否被 filter 掉；
- includeCurrentEvent 配置是否生效。

#### 23.8.2 Keyed State 测试

可以将聚合器逻辑抽成纯 Java 状态演化类，先做单元测试；
再在 Flink 层做更少量但关键的集成测试。

重点验证：

- state key 是否正确；
- timer 注册与删除；
- 过期清理；
- state snapshot/restore 一致性。

#### 23.8.3 Broadcast State 测试

重点验证：

- 新快照是否正确进入广播状态；
- 本地编译缓存是否正确构建；
- 旧版本是否保留或淘汰；
- 快照编译失败时是否继续使用旧版本。

#### 23.8.4 事件时间与乱序测试

如果后面你做 event-time 语义，需要测试：

- 乱序事件进入后窗口结果是否符合预期；
- watermark 推进后是否触发清理；
- 延迟到达数据是否被接纳或丢弃。

一期如果先做 processing-time 或 bucket 近似模型，也建议把边界写清楚并覆盖对应测试。

#### 23.8.5 Flink MiniCluster / TestHarness

你可以考虑两类方式：

##### 方式 A：逻辑单元抽离

把绝大部分状态演化逻辑抽到纯 Java 类里测试。
优点：

- 快；
- 稳；
- 不依赖 MiniCluster。

##### 方式 B：关键算子用 Flink 测试工具验证

对于：

- BroadcastProcessFunction；
- KeyedProcessFunction；
- timer 触发；
- state 清理；

可以使用 Flink 的测试 harness 或 MiniCluster 做更贴近真实运行的测试。

对于个人项目，我建议两者结合：

- 大量逻辑下沉做普通 JUnit；
- 少量关键链路做 Flink 集成测试。

### 23.9 第六层：联调测试——真正把控制平台、Kafka、Redis、Flink 串起来

联调测试的目标是：

> 验证整套系统的关键模块是否真的能连起来工作。

这和单元测试、组件测试完全不同。它不关注某个方法是否正确，而关注：

- 发布是否真的能让 Flink 收到快照；
- Redis lookup 是否真的能查到值；
- Kafka 事件是否真的能触发决策；
- 决策日志是否真的落库可查；
- 前端页面能否真正展示结果。

#### 23.9.1 最小联调链路建议

至少要覆盖下面这条主链路：

1. 在控制平台创建场景 / 特征 / 规则 / 策略；
2. 点击发布，生成 snapshot；
3. Flink 成功接收该 snapshot；
4. 向 Kafka 发一条事件；
5. Redis 中预先准备名单或画像；
6. Flink 计算后输出 decision result；
7. 日志落库；
8. 后台页面可查询该条 decision log。

如果这条链路能稳定跑通，你的平台主骨架就成立了。

#### 23.9.2 联调测试环境建议

推荐使用：

- Docker Compose
- MySQL
- Redis
- Kafka
- Flink
- rdp-admin
- rdp-ui

如果你后面做自动化，可以考虑 testcontainers；
但对个人项目来说，先把 Compose 联调环境打磨好已经很实用。

---

### 23.10 非功能测试：性能、恢复、容错、稳定性

风控系统除了“结果对不对”，还必须考虑“跑得稳不稳”。

所以你至少要建立几个非功能测试方向。

#### 23.10.1 性能测试

关注：

- 吞吐量（EPS/TPS）；
- 平均延迟；
- P95 / P99 延迟；
- Redis lookup 开销；
- 规则数量增加后的性能变化；
- 快照切换时的抖动。

#### 23.10.2 恢复测试

重点看：

- Flink 重启后是否能恢复状态；
- Broadcast State 是否正确恢复；
- 恢复后是否继续使用正确版本快照；
- 状态恢复后决策结果是否一致。

#### 23.10.3 容错测试

例如：

- Redis 临时不可用时怎么降级；
- 某条 Groovy 脚本执行异常怎么兜底；
- 配置新版本编译失败时是否保留旧版本；
- Kafka 某分区积压时行为如何。

#### 23.10.4 稳定性测试

例如：

- 长时间持续压测下内存是否稳定；
- Groovy classloader 是否泄漏；
- 热更新多次后本地缓存是否膨胀；
- 决策日志异步输出是否积压。

这些能力哪怕你不做成特别正式的“测试平台”，也建议至少形成脚本与测试报告。

---

### 23.11 测试数据应该怎么构造

这类项目很容易卡在：

> 我知道要测，但不知道怎么造数据。

我建议你的测试数据分成三类：

#### 23.11.1 规则级小样本数据

用于单元测试和仿真测试。
特点：

- 条数少；
- 手工可读；
- 明确知道预期结果。

例如：

- 一条 amount = 6800 的 trade 事件；
- 三条同 user 的连续交易事件；
- 一条 device 命中 blacklist 的事件。

#### 23.11.2 场景级用例集

用于仿真回归。
特点：

- 按场景组织；
- 每个场景有若干典型样例；
- 每条都有 expected action / expected hit rules。

例如：

- 登录风控 10 条；
- 注册反作弊 10 条；
- 交易风控 20 条。

#### 23.11.3 历史回放级数据

用于 diff 测试和压测。
特点：

- 数量较多；
- 时间顺序明确；
- 尽量接近真实事件模型；
- 可从 mock generator 或历史样本文件中构造。

### 23.12 如何保证“仿真结果”和“线上运行结果”一致

这是你整个项目非常关键的一条测试原则。

#### 核心原则

> **同一条规则、同一份快照、同一份上下文，在仿真和线上应尽可能得到相同结果。**

要实现这一点，建议做到：

#### 1）共用同一套 rule-core / policy-core

不要后台一套判断逻辑，Flink 一套判断逻辑。

#### 2）共用同一套表达式/Groovy 执行接口

执行器可以有 runtime 差异，但接口和语义要一致。

#### 3）仿真要使用和线上相同的快照结构

不是后台临时拼一些对象就算仿真。

#### 4）上下文字段命名必须一致

例如：

- `user_trade_cnt_5m`
- `device_in_blacklist`
- `high_amt_flag`

不能线上叫一个名字，仿真叫另一个。

#### 5）命中原因模板复用

线上和仿真都通过同一模板渲染器生成说明。

做到这些以后，你的平台可信度会明显提升。

如果你希望这个开源项目更像一个“可持续演进”的项目，建议至少做一层基础自动化。

### 23.15 一个人做这个项目，测试应该优先投入在哪些地方

如果资源有限，不可能所有测试一步到位。那就要讲优先级。

#### 第一优先级

- 表达式执行器；
- 规则执行器；
- 策略执行器；
- Snapshot 编译器；
- 仿真测试；
- 一个完整主链路联调测试。

这是最值钱的部分。

#### 第二优先级

- Flink 特征状态逻辑测试；
- Broadcast State 切换测试；
- Redis lookup 组件测试；
- 命中原因渲染测试。

#### 第三优先级

- 回放测试；
- 压测脚本；
- 恢复测试；
- 长稳测试。

也就是说，你不用一开始就做满全套，但一定要把最关键的“逻辑正确性 + 主链路一致性”测扎实。

### 23.16 常见测试误区

最后我总结几个这类项目特别容易踩的坑。

#### 误区 1：只测接口，不测规则内核

风控平台最核心的是规则和策略执行逻辑，不是 Controller。

#### 误区 2：仿真和线上两套逻辑

这是最危险的误区之一。

#### 误区 3：只测当前版本，不测版本切换

你的平台核心价值之一就是热更新，所以必须测版本切换。

#### 误区 4：忽略 Flink 状态边界

状态清理、窗口过期、timer 触发这些不测，线上很容易出问题。

#### 误区 5：没有固定样例集

没有标准 case，后面规则演进时很难做回归。

#### 误区 6：把所有测试都压到端到端

这样会导致测试慢、脆、定位难。一定要做分层。
