## 7.1 这一章解决什么问题

前面第 1\~6 章，我们已经把这套实时风控平台的定位、领域模型、能力边界、总体架构以及核心数据流讲清楚了。

接下来要进入一个真正开始“能落代码”的阶段：

> 控制平台的数据模型到底该怎么设计？

这个问题非常关键，因为控制平台不是一个随便存几段 JSON 的后台，而是整个平台的**设计态中心**。它决定了：

- 场景如何组织
- 事件模型如何统一
- 特征如何配置
- 规则如何表达
- 策略如何编排
- 发布如何产出运行时快照
- 审计、仿真、日志如何支撑

如果这一层的数据模型设计得不好，后面会出现很多问题：

- 页面难做，字段语义混乱
- 规则和特征的边界不清
- 多张表关系紊乱
- 发布时很难编译出可执行快照
- Flink 运行端很难消费配置
- 仿真与线上执行难以保持一致

所以这一章的目标，是把控制平台的设计态模型真正“落到数据库和工程结构上”。

这一章会重点解决：

1. 控制平台到底要管理哪些核心对象
2. 为什么设计态必须结构化存储，而不是一股脑存大 JSON
3. 多张业务表之间应该如何组织
4. 什么数据属于设计态，什么属于发布态，什么属于运行态
5. 后续如何从这些表编译出 Snapshot

---

## 7.2 为什么控制平台不能只存一坨 JSON

这是很多人做平台时最容易踩的一个坑。

表面看，把场景、特征、规则、策略全部塞进一份 JSON，好像很省事：

- 表少
- 开发快
- 页面保存简单
- 发布也方便

但只要项目稍微复杂一点，这种做法很快就会失控。

### 7.2.1 全 JSON 方案会带来的问题

#### 问题 1：无法做结构化校验

比如你想检查：

- 规则引用的 feature 是否存在
- scene 下 feature code 是否重复
- policy 是否引用了不存在的 rule
- derived feature 是否循环依赖

如果所有东西都在大 JSON 里，这些校验会变得很难维护。

#### 问题 2：无法做高效管理与查询

例如你想在后台实现：

- 规则列表筛选
- 按场景查询特征
- 查询某个 feature 被哪些规则引用
- 查看某条规则的历史版本

全 JSON 会让这类查询非常别扭。

#### 问题 3：难以做审计和差异对比

如果某个版本只改了一条规则阈值，你希望知道到底改了什么。

如果全是大 JSON：

- diff 粒度粗
- 很难做可视化对比
- 审计体验很差

#### 问题 4：页面模型会越来越混乱

一个成熟控制平台通常是多个页面协同：

- 场景中心
- 事件中心
- 特征中心
- 规则中心
- 策略中心
- 发布中心

如果底层只有一个大 JSON，那么每个页面都在改同一坨结构，最终会很难维护。

#### 问题 5：不利于“设计态 -> 发布态 -> 运行态”分层

设计态本来就应该是结构化、多对象、多关系的；
发布态才适合编译成一份自包含快照。

如果一开始就把设计态也做成一坨 JSON，实际上是把两个阶段混在了一起。

---

## 7.3 为什么设计态必须结构化，而运行态适合 JSON 快照

这是你后面所有建模工作的总原则。

### 7.3.1 设计态的本质

设计态是供人编辑和平台管理的，因此它需要：

- 结构化
- 可校验
- 可筛选
- 可审计
- 可分页查询
- 可增量修改
- 可做关联分析

所以设计态更适合：

- MySQL 关系表
- 明确字段语义
- 显式外键或逻辑关联

### 7.3.2 运行态的本质

运行态是给 Flink 用的，因此它需要：

- 自包含
- 去关系化
- 可序列化
- 可广播
- 可恢复
- 可版本化

所以运行态更适合：

- 一份 Snapshot JSON
- 一组 Java POJO
- 一个完整可执行计划

一句话概括：

> 设计态追求“好管理”，运行态追求“好执行”。

---

## 7.4 控制平台设计态模型的总体原则

我建议你在设计控制平台的数据模型时，坚持下面几个原则。

### 原则 1：以 Scene 为一级组织维度

所有核心对象尽量归属某个场景：

- 规则属于某个场景
- 特征属于某个场景
- 策略属于某个场景
- 发布版本属于某个场景

这样管理最清晰。

### 原则 2：对象边界要清楚

要明确区分：

- Event Schema 是输入模型
- Feature 是变量定义
- Rule 是判断逻辑
- Policy 是规则组织与收敛
- Release 是发布产物

不要让一个对象承担太多职责。

### 原则 3：设计态允许关系化，运行态再去关系化

不要怕设计态表多，真正关键的是：

- 结构要清楚
- 发布时能编译展开

### 原则 4：支持审计和版本演进

控制平台的数据模型一定要考虑：

- 谁改了
- 什么时候改的
- 改了什么
- 当前线上是什么版本
- 是否能回滚

### 原则 5：先满足“能编译成快照”

任何设计态表结构，最终都要回答一个问题：

> 这张表的数据，未来如何进入 Snapshot？

如果回答不出来，说明模型设计还不够工程化。

---

## 7.5 我建议的控制平台核心对象分层

控制平台的数据模型，建议分成五层：

1. **基础管理层**
2. **场景与事件层**
3. **决策对象层**
4. **发布与运行桥接层**
5. **审计与分析支撑层**

下面分别展开。

## 7.7 场景与事件层

这一层定义“系统处理什么事件”。

### 7.7.1 场景表 `biz_scene`

这是整个控制平台最核心的一级对象。

建议字段：

- `id`
- `scene_code`
- `scene_name`
- `scene_type`
- `description`
- `status`
- `owner_id`
- `created_at`
- `updated_at`

示例：

- `LOGIN_RISK`
- `REGISTER_ANTI_FRAUD`
- `TRADE_RISK`

#### 为什么 scene 要单独建表

因为 scene 是所有设计态对象的组织根：

- feature 归属 scene
- rule 归属 scene
- policy 归属 scene
- release 归属 scene

所以它必须是一级实体。

---

### 7.7.2 事件模型表 `event_schema`

用于定义某个场景下的事件类型。

建议字段：

- `id`
- `scene_code`
- `event_code`
- `event_name`
- `event_type`
- `source_type`
- `topic_name`
- `version`
- `status`
- `description`
- `created_at`
- `updated_at`

示例：

- `login`
- `register`
- `trade`

### 7.7.3 事件字段表 `event_field`

用于定义事件有哪些字段。

建议字段：

- `id`
- `event_schema_id`
- `field_name`
- `field_type`
- `required_flag`
- `default_value`
- `sample_value`
- `description`
- `sort_no`

#### 为什么要拆 `event_schema` 和 `event_field`

因为事件模型本质是“主表 + 字段明细”。

这样做的好处是：

- 页面好做
- 字段校验好做
- 变量提示好做
- 仿真输入生成好做
- 后续表达式变量合法性检查更方便

---

## 7.8 实体维度层（可选但非常建议有）

严格来说，Entity 可以只是逻辑概念，不一定非要单独建表。 但我建议你至少做一张实体类型表，方便特征和规则配置统一化。

### 7.8.1 实体类型表 `entity_type_def`

建议字段：

- `id`
- `entity_type`
- `entity_name`
- `key_field_name`
- `description`

示例：

- `user` -> `userId`
- `device` -> `deviceId`
- `ip` -> `ip`
- `mobile` -> `mobile`

#### 为什么建议有这张表

因为很多特征配置都要回答：

- 这个特征是按什么实体聚合的？
- entity key 从哪个字段取？

如果有这张表，平台表达会更统一。

- `list_type`（black/white/tag）
- `status`

### 7.9.2 名单条目表 `list_item`

建议字段：

- `id`

- `match_key`
- `ext_json`
- `expire_at`
- `status`

#### `ext_json` 可以存什么

## 7.10 特征中心模型设计

这一层是平台最核心的设计态模型之一。

建议把特征显式分成三类：

1. Stream Feature
2. Lookup Feature
3. Derived Feature

这样后面：

- 页面会清晰
- 发布编译会清晰
- Flink 执行边界会清晰

---

### 7.10.1 特征定义主表 `feature_def`

建议所有特征共用一个主表，用 `feature_type` 区分类别。

建议字段：

- `id`
- `scene_code`
- `feature_code`
- `feature_name`
- `feature_type`（STREAM / LOOKUP / DERIVED）
- `value_type`
- `status`
- `description`
- `version`
- `created_at`
- `updated_at`

#### 为什么建议有统一主表

因为：

- 规则依赖的统一对象是 Feature
- 不同类型特征需要共享：code、name、scene、value\_type、status
- 页面可以统一展示，再按类型展开配置

---

### 7.10.2 流式特征配置表 `feature_stream_spec`

建议字段：

- `id`
- `feature_id`
- `source_event_types`
- `entity_type`
- `entity_key_expr`
- `agg_type`（COUNT / SUM / MAX / LATEST / DISTINCT\_COUNT）
- `value_expr`
- `filter_expr`
- `window_type`
- `window_size`
- `window_slide`
- `include_current_event`
- `state_ttl`
- `warmup_mode`
- `remark`

#### 为什么 stream feature 要单独一张 spec 表

因为它和 Flink 状态执行强相关，有很多专属字段：

- aggType
- windowSize
- entityKeyExpr
- ttl

这些字段如果硬塞到通用表里，会变得很难看。

---

### 7.10.3 Lookup 特征配置表 `feature_lookup_spec`

建议字段：

- `id`
- `feature_id`
- `lookup_type`（REDIS\_SET / REDIS\_HASH / KV / CONST）
- `key_expr`
- `storage_namespace`
- `default_value`
- `cache_ttl`
- `timeout_ms`
- `remark`

#### 说明

这张表定义的是：

- 到哪里查
- 用什么 key 查
- 查不到默认值是什么

它与运行态 Redis 查询边界非常清晰。

---

### 7.10.4 派生特征配置表 `feature_derived_spec`

建议字段：

- `id`
- `feature_id`
- `engine_type`（AVIATOR / GROOVY）
- `expr_content`
- `depends_on_json`
- `remark`

#### 为什么派生特征要独立出来

因为派生特征的本质不是状态计算，而是“已有上下文上的二次推导”。
它最适合表达式引擎或 Groovy。

---

## 7.11 规则中心模型设计

规则的本质是“基于上下文做条件判断”。

### 7.11.1 规则定义表 `rule_def`

建议字段：

- `id`
- `scene_code`
- `rule_code`
- `rule_name`
- `engine_type`（AVIATOR / GROOVY）
- `when_expr`
- `priority`
- `hit_action`
- `risk_score`
- `hit_reason_template`
- `status`
- `version`
- `description`
- `created_at`
- `updated_at`

#### 为什么规则表不再拆主子表

规则的核心主要就是：

- 用什么引擎
- 表达式是什么
- 命中后是什么动作
- 分值和优先级如何

它不像事件字段或特征 spec 那样需要那么多结构化专属字段。

当然，如果你后面想更强，可以再拆：

- rule condition
- rule action

但对你现在的项目，一张 `rule_def` 已经足够清晰。

---

### 7.11.2 规则依赖关系表（可选） `rule_dependency`

这张表不是必须，但如果你想让平台更强，可以保留一张编译期依赖表。

建议字段：

- `id`
- `scene_code`
- `rule_code`
- `dependency_type`（FIELD / FEATURE / LIST）
- `dependency_code`

#### 作用

- 做可视化依赖分析
- 查询某个 feature 被哪些 rule 使用
- 发布时辅助校验

如果你不想前期加表，也可以把依赖分析结果只放在发布时临时计算，不持久化。

---

## 7.12 策略中心模型设计

规则不是最终决策，策略才是“如何把规则组织起来”。

### 7.12.1 策略定义表 `policy_def`

建议字段：

- `id`
- `scene_code`
- `policy_code`
- `policy_name`
- `decision_mode`（FIRST\_HIT / SCORE\_CARD）
- `default_action`
- `status`
- `description`
- `version`
- `created_at`
- `updated_at`

### 7.12.2 策略规则关联表 `policy_rule_ref`

建议字段：

- `id`
- `policy_id`
- `rule_id`
- `order_no`
- `enabled_flag`
- `score_weight`
- `branch_expr`
- `created_at`

#### 为什么 policy 和 rule 必须分表

因为：

- 一条规则是可复用对象
- 一个策略是规则的组织方式
- 策略关注的是：顺序、模式、收敛
- 规则关注的是：条件判断本身

如果两者不分开，后面很容易变成“规则里写满所有决策流程”，平台会变脆弱。

---

## 7.13 场景绑定当前策略的模型设计

一个 scene 通常在某个时刻对应一套主策略。

### 7.13.1 场景策略绑定表 `scene_policy_bind`

建议字段：

- `id`
- `scene_code`
- `policy_id`
- `bind_status`
- `created_at`

#### 为什么建议单独建这张表

因为：

- 场景和策略可以解耦
- 后续可以支持切换主策略
- 发布时更清楚当前 scene 需要编译哪套 policy

如果你项目规模较小，也可以把 `policy_def` 直接按 scene 唯一约束处理，不单独建绑定表。 但从扩展性上，我更建议单独建表。

---

## 7.14 发布与运行桥接层模型设计

这一层是控制平台最关键的一层。 它决定了设计态如何进入运行态。

### 7.14.1 发布记录表 `scene_release`

这张表非常关键。

建议字段：

- `id`
- `scene_code`
- `version_no`
- `snapshot_json`
- `checksum`
- `publish_status`
- `effective_from`
- `published_by`
- `published_at`
- `rollback_from_version`
- `remark`
- `created_at`

#### 这张表的意义

它不是普通配置表，而是：

> 某个场景某个版本的完整运行态产物。

Flink 不应该直接读几十张设计态表；
Flink 应该消费这张表对应的快照。

#### `publish_status` 可以有哪些状态

例如：

- `DRAFT`
- `PUBLISHED`
- `ACTIVE`
- `ROLLED_BACK`
- `FAILED`

#### `checksum` 有什么用

用于：

- 防止重复发布
- 检查配置是否被篡改
- 方便运行端对版本内容做一致性判断

---

### 7.14.2 发布明细/变更记录表（可选） `scene_release_change`

建议字段：

- `id`
- `release_id`
- `change_type`
- `target_type`（FEATURE / RULE / POLICY / LIST）
- `target_code`
- `before_json`
- `after_json`

#### 作用

用于增强：

- 发布 diff 展示
- 审计可读性
- 回滚辅助

前期如果想简化，可以先不做这张表，把变更 diff 逻辑留到应用层实现。

---

## 7.16 决策日志与命中日志模型设计

严格来说，这部分更偏分析面。 但控制平台通常也会提供查询入口，因此模型上可以先一并设计。

### 7.16.1 决策日志表 `decision_log`

建议字段：

- `id`
- `trace_id`
- `scene_code`
- `event_id`
- `event_type`
- `entity_id`
- `input_json`
- `feature_snapshot_json`
- `hit_rules_json`
- `final_action`
- `final_score`
- `version_no`
- `latency_ms`
- `event_time`
- `created_at`

### 7.16.2 规则命中日志表 `rule_hit_log`

建议字段：

- `id`
- `decision_id`
- `rule_code`
- `hit_flag`
- `score`
- `hit_value_json`
- `created_at`

#### 为什么建议把 `decision_log` 和 `rule_hit_log` 分开

因为：

- 决策是一条主记录
- 规则命中是多条子记录
- 便于页面查询和统计
- 后面若要做 OLAP 同步也更方便

如果项目初期想简化，也可以先把命中规则直接放到 `decision_log.hit_rules_json` 中。

## 7.17 多张业务表之间的关系应该怎么理解

这里我们把前面讲的表关系用逻辑方式串起来。

### 7.17.1 Scene 是根对象

- 一个 Scene 下面有多个 Event Schema
- 一个 Scene 下面有多个 Feature
- 一个 Scene 下面有多个 Rule
- 一个 Scene 下面有多个 Policy
- 一个 Scene 下面有多个 Release

### 7.17.2 Event Schema 描述输入

- 一个 Event Schema 有多个 Event Field

### 7.17.3 Feature 分类型扩展

- 一个 FeatureDef 对应一种 FeatureType
- 根据类型再落到不同 spec 表中

### 7.17.4 Policy 组织 Rule

- 一个 Policy 通过 `policy_rule_ref` 关联多条 Rule

### 7.17.5 Release 编译 Scene 下当前有效对象

- 发布时，读取 scene 下当前启用的 feature / rule / policy / list 引用
- 编译为 Snapshot
- 落到 `scene_release`

### 7.17.6 Decision 与 Release 对应

- 每次 Decision 都应该记录使用的 `version_no`
- 这样未来才能追溯到当时到底用了哪份快照

## 7.18 为什么不要让“规则、策略、特征”混成一张万能表

这也是很多系统早期容易犯的错。

比如有人会想：

- 反正都是表达式
- feature 也有 expr
- rule 也有 expr
- policy 也可以存 expr
- 那干脆搞一张万能节点表

我不建议你这么做。

原因是这几个对象虽然都可能有“表达式”，但职责完全不同：

- Feature：定义变量
- Rule：基于变量做判断
- Policy：组织规则并收敛结果

如果用一张万能表去承载：

- 页面会变复杂
- 校验规则会变复杂
- 发布编译会变复杂
- 后续代码结构会变得不清晰

所以：

> 不要因为“都带表达式”，就把它们建成同一种对象。

## 7.20 控制平台的数据模型最终要支持什么能力

设计这些表，不是为了“表设计好看”，而是为了支撑平台能力。

一套合格的控制平台数据模型，至少应该支持下面这些能力：

### 1）可管理

- 页面能查、能改、能分页
- 配置结构清楚

### 2）可校验

- 表达式检查
- 依赖检查
- 完整性检查

### 3）可编译

- 能从设计态编译出 Snapshot

### 4）可发布

- 有 version
- 有状态
- 有历史记录

### 5）可回滚

- 能定位旧版本
- 能快速切回

### 6）可追溯

- 决策结果能追到具体版本
- 配置变更能追到人

### 7）可扩展

- 后续增加新的 feature 类型
- 增加新的 engine type
- 增加新的策略模式

如果你设计的表结构不能自然支撑这些能力，说明模型还需要继续调整。

## 7.22 这一章对“快照设计”的直接影响

这一章讲完以后，你后面再看第 8、9 章会更容易理解。

因为控制平台的数据模型，最终不是为了存数据本身， 而是为了支撑下面这件事：

> 从多张设计态表，编译出一份运行态 Snapshot。

也就是说：

- `biz_scene` 会变成 snapshot 的 scene 信息
- `feature_*` 会编译成 streamFeature / lookupFeature / derivedFeature
- `rule_def` 会编译成 rule specs
- `policy_def + policy_rule_ref` 会编译成 policy plan
- `scene_release` 会承载最终产物

你后面会发现：

> 设计态表像“源码”，Snapshot 像“编译产物”。

这个类比是非常准确的。

## 7.23 本章小结

这一章本质上解决的是：

> 控制平台的数据模型到底该怎么落地。

重点可以收成下面几条。

### 1）设计态必须结构化存储

不能一上来把所有东西都塞进大 JSON。

### 2）运行态才适合 JSON 快照

设计态是为管理服务，运行态是为执行服务。

### 3）控制平台核心对象应清晰分层

建议围绕：

- scene
- event schema
- feature
- list
- rule
- policy
- release
- simulation
- log

来设计。

### 4）Feature 应按类型拆 spec

- stream
- lookup
- derived

这样后面 Flink 执行边界才会清楚。

### 5）Release 是设计态和运行态之间的桥梁

`scene_release` 是后续整个快照和 Flink 执行设计的关键表。

### 6）好的设计态模型，不只是能 CRUD

还必须能支撑：

- 校验
- 依赖分析
- 发布
- 回滚
- 仿真
- 追溯
