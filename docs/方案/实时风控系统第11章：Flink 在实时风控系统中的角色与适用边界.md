## 11.1 这一章解决什么问题

到第 10 章为止，我们已经把控制平台这一侧的事情讲到了一个比较完整的程度：

- 你已经知道设计态数据应该怎么建模
- 你已经知道为什么要发布成运行时快照
- 你已经知道快照应该长什么样
- 你也知道发布前为什么必须做表达式/Groovy 校验与依赖分析

但是到这里为止，系统仍然停留在“控制面已经准备好了”的阶段。
接下来必须回答一个更关键的问题：

> 控制平台已经生成了运行时快照，那为什么是 Flink 来执行？
> Flink 到底在这个系统中扮演什么角色？
> 它能做什么，不能做什么？
> 哪些逻辑应该放进 Flink，哪些逻辑不应该强行塞给 Flink？

这几个问题如果不先讲透，后面你在看第 12 章到第 19 章时，就很容易陷入一种误区：

- 觉得 Flink 只是一个“窗口统计工具”
- 或者反过来，觉得 Flink 什么都应该做
- 再或者，把 Flink 当成一个可以无限动态执行任意脚本的平台

这三种理解都不对。

所以这一章的目标就是：

1. 解释为什么实时风控非常适合用 Flink
2. 解释 Flink 在这个系统里的准确定位
3. 说明 Flink 能做什么，不能做什么
4. 划清 Flink、控制平台、Redis、Kafka 之间的边界
5. 建立一个后面所有引擎设计都会反复用到的核心认知：

> **Flink 在实时风控系统中，本质上不是“窗口工具”，而是“状态型实时决策执行引擎”。**

---

## 11.2 先说结论：为什么实时风控天然适合 Flink

如果用一句话概括：

> **实时风控的核心难点，是在事件发生的那一刻，基于事件本身、历史状态、外部上下文和当前策略版本，做低延迟、可恢复、可热更新的实时决策。**

而 Flink 恰好擅长处理这类问题。

因为它同时具备下面这些能力：

- 持续消费实时事件流
- 面向 Key 做状态维护
- 支持事件时间与处理时间
- 支持定时器和状态清理
- 支持广播配置流
- 支持异步查询外部系统
- 支持 checkpoint 恢复
- 支持高吞吐、低延迟的数据处理

你把这些能力和风控需求一对照，就会发现高度匹配：

| 风控需求                 | Flink 能力                    |
| :------------------- | :-------------------------- |
| 实时接收登录/注册/交易事件       | 流式消费 Kafka / CDC            |
| 维护用户、设备、IP 的行为状态     | Keyed State                 |
| 统计 5 分钟、30 分钟、1 小时特征 | 状态 + 时间 + Timer             |
| 根据新发布版本动态切换逻辑        | Broadcast State             |
| 查询黑白名单、画像、标签         | Async I/O / Redis Lookup    |
| 崩溃后恢复状态与配置           | Checkpoint + State Recovery |
| 高并发连续处理              | 流式执行模型                      |

所以你后面要始终记住：

> **Flink 适合实时风控，不是因为它能做窗口，而是因为它能做“状态驱动的实时上下文构建和决策执行”。**

---

## 11.3 如果不用 Flink，会遇到什么问题

为了理解 Flink 的价值，先反过来看。
如果你不用 Flink，而是用普通 Spring Boot 服务 + Redis + 数据库硬做，也不是完全不可能，但很快就会遇到问题。

### 11.3.1 问题 1：状态维护会越来越难

风控里最常见的逻辑是：

- 用户 5 分钟交易次数
- IP 10 分钟注册次数
- 设备 1 小时关联账号数
- 最近一次登录时间
- 一段时间内 distinct 数量

如果用普通服务自己维护，你需要自己解决：

- 多实例并发一致性
- key 分区
- 状态生命周期
- 过期清理
- 恢复
- 窗口重算
- 容错

这些本质上就是一个流处理引擎要解决的问题。自己手写，成本很高。

---

### 11.3.2 问题 2：配置热更新和状态恢复难以统一

风控系统不是写死代码发布一次就完事，而是要：

- 发布新规则
- 回滚旧版本
- 保持状态不丢
- 保持逻辑可恢复

如果是普通服务，很容易出现：

- 配置更新了，但内存状态和版本关系混乱
- 服务重启后状态丢失或恢复不一致
- 配置和状态无法一起恢复到某个稳定点

而 Flink 的 checkpoint 能把：

- keyed state
- broadcast state
- timer

统一纳入恢复体系。

---

### 11.3.3 问题 3：流式时间语义处理很麻烦

风控里很多逻辑天然是“时间相关”的：

- 最近 5 分钟
- 最近 30 分钟
- 1 小时内 distinct
- 某个事件后 10 分钟内是否又发生某行为

如果没有流引擎，你会发现：

- 时间窗口难管理
- 事件乱序难处理
- 清理过期状态难做
- 定时器逻辑散落一地

而这些恰好是 Flink 的基础能力。

---

### 11.3.4 问题 4：可扩展性会很差

随着场景增加，你会越来越想支持：

- 不同 entity 的状态
- 更多规则
- 多场景并发执行
- 多版本配置切换
- 更多指标和日志

如果一开始不用流式状态引擎，后面系统很容易演变成：

- 大量 Redis 操作拼起来的准实时系统
- 很多“补丁式 if-else + 定时清理任务”
- 难以解释，也难以恢复

所以不是说不用 Flink 完全做不了，而是：

> **一旦目标是做“平台级实时决策系统”，Flink 会让很多本来要你手工实现的底层能力变成现成能力。**

---

## 11.4 Flink 在实时风控系统中的准确定位

这里给一个非常重要的定义。

> **Flink 在实时风控系统中的定位，不是规则配置平台，也不是最终的风控平台本身，而是“运行态实时决策执行引擎”。**

这句话可以拆成三层理解。

### 第一层：它不是控制平台

Flink 不负责：

- 配规则页面
- 策略管理页面
- 发布审批
- 用户权限
- 审计日志管理
- 设计态依赖分析

这些都属于控制平台。

---

### 第二层：它不是单纯的规则执行器

如果只是把一个上下文 Map 丢进去算表达式，那只是表达式引擎或规则引擎的事。

Flink 除了执行规则，还要负责：

- 消费实时事件
- 维护状态
- 计算特征
- 查询外部上下文
- 动态接收配置
- 输出决策结果

所以它是一个更大的执行框架。

---

### 第三层：它是“运行态上下文构建 + 决策执行”的核心

你可以把 Flink 在系统中的职责总结成：

1. 接收事件
2. 找到当前生效快照
3. 构建实时上下文
4. 执行规则/策略
5. 输出结果与日志

所以它既是：

- 实时特征引擎
- 规则执行承载层
- 决策结果输出层

也是整个运行态系统的中心。

---

## 11.5 你应该如何看待 Flink：不是窗口工具，而是状态计算引擎

这是我特别希望你建立起来的认知。

很多人学 Flink 时的第一印象是：

- 窗口统计
- SQL 聚合
- 流批一体

这些当然没错，但对于风控平台来说，这种理解是不够的。

对于你这个项目，更准确的理解应该是：

> **Flink 是一个面向事件流、具备状态、时间、定时器、广播配置、恢复能力的状态型计算引擎。**

为什么要强调“状态型”？
因为风控的大部分价值，都不是来自“事件本身”，而是来自“事件 + 状态”的组合。

例如：

- 当前金额 6800：只是事件字段
- 最近 5 分钟交易 4 次：这是状态计算结果
- 设备 1 小时关联 6 个账号：这是状态计算结果
- 最近 30 分钟交易和 18000：这是状态计算结果

真正的风险判断，往往建立在后者之上。

所以你看 Flink 时，不要想：

- 它只是帮我做几个窗口

而要想：

- 它帮我维护“线上实时决策所需的行为状态”

这会直接影响你后面如何设计特征系统、状态存储和引擎执行链路。

---

## 11.6 Flink 在这个系统里最重要的 6 项能力

下面把 Flink 在风控系统里的关键能力系统讲一下。

### 11.6.1 状态能力（State）

这是 Flink 对风控最重要的价值。

风控很多逻辑都不是一次事件的静态判断，而是围绕 entity 的历史行为累积：

- user\_trade\_cnt\_5m
- user\_trade\_amt\_sum\_30m
- device\_bind\_user\_cnt\_1h
- ip\_login\_fail\_cnt\_10m

这些都要靠 state 维护。

在 Flink 里，你通常会用：

- `ValueState`
- `MapState`
- `ListState`（谨慎）
- `ReducingState` / `AggregatingState`（某些场景）

而这背后的本质是：

> **把“某个实体到当前为止的行为摘要”稳定地保留在引擎里。**

---

### 11.6.2 时间能力（Time）

风控逻辑天然和时间相关。

例如：

- 最近 5 分钟
- 1 小时内
- 24 小时内
- 某次失败后 10 分钟内

Flink 能很好地处理：

- 处理时间
- 事件时间
- Watermark
- 时间窗口语义

不过在你的项目里，一期不一定需要把时间语义做得非常复杂；
但至少要理解：

> **风控的窗口不是 SQL 语法层面的概念，而是状态生命周期的一部分。**

---

### 11.6.3 定时器能力（Timer）

很多状态不是“永远保留”的，而是有生命周期。

比如：

- 5 分钟窗口桶过期要清理
- 某个 event 聚合等待超时要兜底
- 某些临时上下文要及时释放

这就需要 timer。

Flink 的 timer 能让你：

- 在未来某个时间点触发清理
- 做迟到/超时处理
- 做状态生命周期维护

所以 timer 在风控系统里不是可有可无的，而是状态设计的一部分。

---

### 11.6.4 广播配置能力（Broadcast State）

这是你这个项目最关键的能力之一。

风控平台需要支持：

- 新规则发布后不停机生效
- 新策略版本切换
- 新快照动态加载

这就需要把配置流广播给所有并行任务。

Flink 的 Broadcast State 正适合做：

- 场景快照下发
- 活跃版本维护
- 规则/策略运行时配置广播

但要注意：

> Broadcast State 适合存“可恢复配置”，不适合直接存 Groovy 编译结果等不可序列化对象。

这个边界后面第 13、18 章还会详细讲。

---

### 11.6.5 流式 Join / Lookup / Async I/O

风控决策不能只看事件和状态，通常还要补充外部上下文：

- 黑名单
- 白名单
- 风险等级画像
- 设备标签
- IP 风险信息

这些常常存在 Redis 或其他外部 KV 中。

Flink 可以通过：

- 同步查询 + 本地缓存
- 异步 I/O
- 维表 Join（某些场景）

把这些上下文补进来。

但要注意，外部查询虽然常见，却不能无限制滥用。因为：

- 查询会带来延迟
- 查询会带来超时风险
- 查询过多会让热路径不稳定

所以你后面做引擎设计时，必须控制 lookup 的数量和边界。

---

### 11.6.6 恢复能力（Checkpoint / Recovery）

实时风控系统不是 demo，它必须考虑：

- 任务挂了怎么办
- 发布版本切换后重启怎么办
- 状态如何恢复
- 当前活跃配置如何恢复

Flink 的 checkpoint 机制使得：

- key state 可恢复
- broadcast state 可恢复
- timer 可恢复

这对于风控这样对一致性、解释性有要求的系统非常重要。

因为你不希望出现：

- 规则版本恢复错乱
- 状态丢失导致风控能力瞬间失效
- 恢复后特征值与发布版本不匹配

---

## 11.7 Flink 能做什么，不能做什么

这一节特别重要，因为很多系统就是边界没划清，最后设计失控。

### 11.7.1 Flink 非常适合做的事情

#### 1）实时事件消费

这是基础能力。

#### 2）流式特征计算

例如：

- count
- sum
- max
- latest
- distinct\_count（谨慎做）

#### 3）基于运行时快照动态执行逻辑

例如：

- 当前场景有哪些规则
- 哪些 feature 要参与执行
- 策略模式是什么

#### 4）状态型上下文构建

例如：

- 结合 event + state + lookup + derived

#### 5）低延迟规则执行和结果输出

例如：

- PASS / REVIEW / REJECT

#### 6）输出运行指标和日志

例如：

- latency
- error count
- hit count

---

### 11.7.2 Flink 不适合直接做的事情

#### 1）设计态配置管理

它不适合做规则编辑后台。

#### 2）多表关系拼装和复杂发布校验

这些应该在控制平台完成。

#### 3）大规模管理型查询

例如：

- 列出所有规则
- 查审计日志
- 查版本对比

#### 4）无限制执行任意脚本式流处理逻辑

尤其不适合让用户直接用脚本动态定义：

- 任意 keyBy
- 任意 state schema
- 任意 timer 逻辑
- 任意窗口语义

#### 5）保存大规模主数据

例如：

- 百万级黑名单主数据
- 大量画像主数据

Flink 可以做短期缓存或状态，但不是主数据系统。

---

## 11.8 什么逻辑适合动态化到 Flink，什么不适合

这节要承接你前面很关心的一个核心问题：

> 控制平台生成的逻辑，Flink 到底能承载到什么程度？

答案是：

> **Flink 适合执行“受约束的声明式运行计划”，不适合执行“无限制的任意脚本式流处理定义”。**

下面分类说明。

### 11.8.1 很适合动态化到 Flink 的

#### 1）规则条件表达式

例如：

- `amount >= 5000 && user_trade_cnt_5m >= 3`

#### 2）策略顺序与策略模式

例如：

- FIRST\_HIT
- SCORE\_CARD

#### 3）派生特征表达式

例如：

- `high_amt_flag = amount >= 5000`

#### 4）受控模板化的流式特征定义

例如：

- COUNT / SUM / MAX / LATEST
- 指定 entityKeyExpr、filterExpr、windowSize

这些都是适合做成平台动态配置的。

---

### 11.8.2 只适合“受控动态化”的

#### 1）lookup feature

例如：

- Redis Set/Hash 查询
- keyExpr
- 默认值

#### 2）状态型特征配置

例如：

- aggType
- windowSize
- valueExpr

这些可以开放配置，但必须限制在平台支持的模板能力内。

---

### 11.8.3 不适合开放成任意动态脚本的

#### 1）底层 state 结构任意定义

#### 2）任意 keyBy 逻辑

#### 3）任意 timer/watermark/window 语义

#### 4）任意脚本访问文件、网络、数据库

#### 5）任意脚本创建复杂对象图并长期持有

这些如果全部开放，平台复杂度和风险会急剧上升。

所以你后面设计时，一定要坚持：

> **动态化应该集中在“业务逻辑表达层”，而不是“引擎底层执行语义层”。**

---

## 11.9 为什么 Flink 适合执行“运行时计划”，而不是执行“无限制任意脚本”

这部分和你前面一直问的表达式/Groovy 边界是紧密相关的。

### 11.9.1 什么叫“运行时计划”

比如控制平台发布出来的快照里，会明确描述：

- 哪些 stream feature 需要计算
- 每个 feature 的 aggType、window、entity key 是什么
- 哪些 lookup feature 需要查 Redis
- 哪些 derived feature 用表达式计算
- 哪些 rule 用 Aviator / Groovy 执行
- policy 的顺序是什么

这其实是一份“声明式执行计划”。

Flink 很适合拿着这份计划去执行。

---

### 11.9.2 什么叫“无限制任意脚本”

例如让平台用户直接写脚本决定：

- 用哪个字段 keyBy
- 创建什么 state
- 如何清理 state
- 如何注册 timer
- 如何处理迟到数据
- 如何做任意外部调用

这就已经不是“规则表达”了，而是在让用户直接控制流处理引擎。

这种做法的问题在于：

- 难校验
- 难恢复
- 难监控
- 难优化
- 难保障安全
- 难解释执行行为

所以你后面一定要有一个非常清晰的判断：

> **Flink 可以执行动态化规则和特征计划，但不应该变成一个无限制脚本型流处理宿主。**

---

## 11.10 Flink 与控制平台、Redis、Kafka 的边界关系

### 11.10.1 Flink 与控制平台

控制平台负责：

- 设计态配置管理
- 校验
- 依赖分析
- 快照编译
- 发布/回滚

Flink 负责：

- 接收快照
- 执行快照
- 处理实时事件

所以两者关系是：

> 控制平台产出“可执行计划”，Flink 承担“运行时执行”。

---

### 11.10.2 Flink 与 Redis

Redis 负责：

- 名单
- 画像
- 热点特征 materialization
- 辅助查询数据

Flink 负责：

- 在热路径中按需读取这些上下文
- 但不把 Redis 当成所有实时状态的替代品

所以边界是：

> **实时聚合状态在 Flink，在线补充上下文在 Redis。**

---

### 11.10.3 Flink 与 Kafka

Kafka 负责承载流：

- 原始事件流
- 配置快照流
- 决策结果流
- 日志流

Flink 负责消费与产出这些流。

所以 Kafka 是数据总线，Flink 是流的处理者。

---

## 11.11 你这个项目里，Flink 最合理的能力边界

如果结合你的项目目标，我建议你把 Flink 的边界定成下面这样。

### Flink 负责

- 消费事件流
- 消费快照配置流
- 维护 Keyed State / Broadcast State
- 执行 stream feature 模板
- 调 Redis 做 lookup enrichment
- 执行 derived feature
- 执行 rule / policy
- 输出 decision result / decision log / metrics

---

### Flink 不负责

- 后台页面与 CRUD
- 设计态规则校验
- 规则依赖分析前置校验
- 管理型查询
- 大名单主存储
- 任意脚本定义底层状态机

这个边界如果坚持住，你后面的系统会清晰很多。

---

## 11.12 对你来说，Flink 端最值得重点做的不是“多”，而是“对”

这一点我想单独强调一下。

很多人做开源项目时，会想在 Flink 端展示很多“高级功能”：

- 任意 Groovy
- 自定义窗口
- 自定义算子 DSL
- 任意 CEP 编排
- 任意状态脚本

这些当然炫，但很容易让系统失控。

对你而言，更有价值的做法是：

### 一期把下面这些做扎实

- Broadcast State 动态加载快照
- Keyed State 计算基础实时特征
- Redis lookup 补上下文
- 统一 EvalContext 构建
- 规则表达式执行
- 策略收敛
- 决策日志和版本追溯

只要这条主链路做扎实，已经足够证明：

- 你懂 Flink
- 你懂风控平台
- 你懂控制面和计算面的边界
- 你懂热更新和状态恢复

这比“堆很多炫技特性”更有说服力。

## 11.13 本章小结

这一章最核心的是帮你建立一个正确的 Flink 认知。

### 1）实时风控为什么适合用 Flink

因为它天然需要：

- 实时事件消费
- 状态维护
- 时间语义
- 配置广播
- 外部 lookup
- 恢复能力

### 2）Flink 在系统中的定位

它不是控制平台，也不是单纯规则引擎，
而是：

> **运行态实时决策执行引擎**

### 3）你应该如何看待 Flink

不是窗口工具，而是状态计算引擎。

### 4）Flink 最核心的能力

- State
- Time
- Timer
- Broadcast State
- Async I/O / Lookup
- Checkpoint / Recovery

### 5）Flink 能做什么，不能做什么

它适合执行声明式运行计划，
不适合变成无限制脚本型流处理宿主。

### 6）你项目里最重要的边界

- 动态化业务逻辑可以放进 Flink
- 底层状态执行语义不要无限制开放给脚本
