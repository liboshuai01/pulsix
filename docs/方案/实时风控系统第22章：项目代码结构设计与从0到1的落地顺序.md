## 22.1 这一章解决什么问题

前面的章节，我们已经把这套实时风控系统从理论、架构、快照、Flink 引擎、规则执行、表达式与 Groovy 边界，基本都讲清楚了。

但真正开始做项目时，很多人还是会卡在一个非常现实的问题上：

> 我现在知道系统应该长什么样了，但代码到底该怎么组织？先写什么？后写什么？哪些模块先稳定？一个人怎么把这套系统从 0 推到可运行？

这就是这一章要解决的问题。

这一章不再只是讲“概念应该怎么设计”，而是进入真正的工程落地层面，重点回答下面这些问题：

- 整个项目应该拆成几个仓库或几个子模块
- `pulsix-server`、 `pulsix-module-risk`、 `pulsix-engine`、 `pulsix-framework/pulsix-kernel`、 `pulsix-framework/pulsix-common` 分别应该放什么代码
- 控制平台和 Flink 引擎之间，哪些代码应该复用，哪些不应该耦合
- 包结构怎么组织，才能保证后面扩展时不乱
- 哪些接口和数据结构应该先稳定下来
- 一个人做这个项目，最合理的编码顺序是什么
- 前 1 个月、前 2 个月、前 3 个月，最应该交付哪些里程碑

如果说前面几章是在搭“设计蓝图”，那么这一章就是在回答：

> **蓝图已经有了，工程施工应该怎么排计划、怎么分工、怎么分层、怎么避免一上来就把自己写乱。**

---

## 22.2 先明确一个总原则：逻辑上分层，物理上适度精简

对于个人开源项目来说，最容易犯的一个错误是两头走极端：

### 极端 1：一股脑做成一个大单体

所有代码都放在一个工程里：

- 后台 API
- 规则管理
- 发布编译
- Flink 逻辑
- 表达式执行器
- Redis 查询
- 仿真
- 前端静态资源

短期看似乎简单，但很快会出现：

- 模块边界不清
- 代码依赖乱
- 后台逻辑和引擎逻辑互相污染
- 后面很难梳理“哪部分是控制面，哪部分是运行面”

### 极端 2：一上来拆很多微服务

比如：

- user-service
- scene-service
- feature-service
- rule-service
- policy-service
- release-service
- log-service
- gateway
- auth-service
- engine-service

对个人项目来说，这通常收益远小于成本。你会花很多时间在：

- 服务拆分
- 接口联调
- 网关
- 鉴权传递
- Docker 编排
- 服务注册发现

而不是花在真正的风控核心能力上。

所以最合理的原则是：

> **逻辑上一定要分层清楚，但物理上不要过度拆分。**

也就是说：

- 代码结构上，你要明确区分控制面、运行时内核、Flink 引擎、前端
- 部署形态上，前期尽量收敛为少量独立进程

对你来说，最合适的是：

1. 一个控制平台启动容器： `pulsix-server`
2. 一组平台通用业务模块： `pulsix-module-system`、 `pulsix-module-infra`
3. 一个风控控制面核心业务模块： `pulsix-module-risk`
4. 一个 Flink 决策引擎： `pulsix-engine`
5. 一个前端： `pulsix-ui`
6. 一组共享核心模块： `pulsix-framework/pulsix-common`、 `pulsix-framework/pulsix-kernel`
7. `pulsix-framework` 下继续保留各类 `pulsix-spring-boot-starter-*` 作为基础组件层

这已经非常合理，而且足够专业。

## 22.3 推荐的项目总目录结构

我建议你把整个项目组织成一个多模块仓库。可以是 Maven 多模块，也可以是 Gradle 多模块。结合当前 `pulsix` 的实际工程，建议继续走 **Maven 多模块**，并尽量贴合已有目录。

推荐目录如下：

```latex
pulsix/
├── README.md
├── pom.xml
├── docs/
├── deploy/
├── pulsix-dependencies/
├── pulsix-framework/
│   ├── pulsix-common/
│   ├── pulsix-kernel/
│   └── pulsix-spring-boot-starter-*/
├── pulsix-server/
├── pulsix-module-system/
├── pulsix-module-infra/
├── pulsix-module-risk/
├── pulsix-engine/
└── pulsix-ui/
```

这个结构的思路是：

- `pulsix-dependencies`：BOM、版本对齐
- `pulsix-framework/pulsix-common`：通用工具、错误码、公共枚举、共享 DTO / CommonApi
- `pulsix-framework/pulsix-kernel`：统一执行内核，控制平台仿真和 Flink 引擎都依赖它
- `pulsix-server`：Spring Boot 启动器，负责聚合 `system / infra / risk`
- `pulsix-module-system`：用户、角色、租户、菜单、审计等平台通用系统域
- `pulsix-module-infra`：配置、文件、任务、监控、基础日志等基础设施域
- `pulsix-module-risk`：场景、事件模型、特征、名单、规则、策略、发布、仿真、日志查询等风控控制面主业务
- `pulsix-engine`：Flink 实时引擎
- `pulsix-ui`：前端控制台

这里最关键的一点是：

> **一定要把“运行时核心能力”和“后台管理逻辑”拆开，但不要为了拆分而额外增加不必要模块。**

这样你既能保证边界清楚，也能保持当前工程足够精简。

## 22.4 各模块的职责与边界

下面我们逐个讲清楚这些模块分别应该放什么。

---

## 22.4.1 `pulsix-framework/pulsix-common`：通用基础与共享模型模块

这个模块不要做得太大，但一定要有。它的职责应该尽量基础且稳定。

建议放：

- 通用常量

- 错误码定义

- 枚举类型
  - `SceneCode`
  - `FeatureType`
  - `DecisionAction`
  - `EngineType`
  - `DecisionMode`

- 基础工具类
  - JSON 工具
  - 时间工具
  - 字符串工具
  - 校验工具

- 通用异常
  - `BizException`
  - `CompileException`
  - `EngineRuntimeException`

- 通用返回体（仅控制平台可用，避免侵入引擎）

### 不建议放什么

- 业务 Service
- 数据库 Mapper
- Spring Bean
- Flink 依赖较重的类

也就是说， `pulsix-common` 应保持“轻、纯、稳定”。

---

## 22.4.2 `pulsix-framework/pulsix-common` 中的 `biz/risk`：共享 DTO 与轻量运行时模型

考虑到当前项目已经明确“多模块共享 DTO 放到 `pulsix-framework/pulsix-common/biz`”，这里不再单独拆 `domain` 模块，而是把风险域共享模型统一收敛到 `pulsix-common` 中。

建议放：

### 设计态 / 发布态共享 DTO

- `SceneDef`
- `EventSchema`
- `EventFieldDef`
- `FeatureDef`
- `RuleDef`
- `PolicyDef`
- `ListSet`
- `ListItem`
- `SceneRelease`

### 运行态轻量模型

- `SceneSnapshot`
- `StreamFeatureSpec`
- `LookupFeatureSpec`
- `DerivedFeatureSpec`
- `RuleSpec`
- `PolicySpec`
- `RuntimeHints`

### 执行态共享模型

- `RiskEvent`
- `EvalContext`
- `DecisionResult`
- `RuleHitResult`
- `FeatureValue`

### CommonApi / 查询接口

- `SceneReleaseCommonApi`
- `DecisionLogCommonApi`
- 以及控制面与引擎之间需要共享的轻量查询接口

### 为什么不再单独做独立 domain 模块

因为：

- 当前工程已经有 `pulsix-common/biz` 这套共享约定
- 再增加独立 `domain` 模块会和现有约定重复
- 控制面与计算面共享轻量模型的诉求，用 `pulsix-common/biz/risk` 足以承载

建议目录：

```latex
cn.liboshuai.pulsix.framework.common.biz.risk
├── scene/
├── feature/
├── list/
├── rule/
├── policy/
├── release/
├── simulation/
├── runtime/
└── log/
```

## 22.4.3 `pulsix-framework/pulsix-kernel`：规则执行与上下文执行内核

这个模块是整个项目中**最值得你投入精力抽象**的地方之一。

它应该承载的是：

> **给定一个运行时快照和一个** `EvalContext`**，执行派生特征、规则、策略并返回** `DecisionResult`\*\* 的那套核心逻辑。\*\*

建议包括：

### 核心接口

- `ExpressionInvoker`
- `GroovyInvoker`
- `RuleInvoker`
- `DerivedFeatureInvoker`
- `PolicyExecutor`
- `DecisionEngine`

### 运行时模型

- `CompiledSceneRuntime`
- `CompiledRuleRuntime`
- `CompiledDerivedFeatureRuntime`
- `PolicyRuntime`

### 上下文相关

- `EvalContextBuilder`
- `ContextValueResolver`
- `HitReasonRenderer`

### 执行相关

- `DefaultDecisionEngine`
- `FirstHitPolicyExecutor`
- `ScoreCardPolicyExecutor`

### 为什么这个模块特别重要

因为它应该被下面两个地方共同复用：

1. **控制平台仿真测试**
2. **Flink 线上运行时执行**

你一定要坚持这个原则：

> **仿真与线上尽量共用一套规则执行内核。**

否则很容易出现：

- 仿真命中，线上不命中
- 表达式上下文不一致
- 命中原因模板不一致

这是平台可信度的大坑。

---

## 22.4.4 发布编译器：先内聚在 `pulsix-module-risk` 的 `release` 子域

当前阶段我不建议单独拆 `snapshot-compiler` 模块。

更合适的做法是：

> **把发布编译器作为 `pulsix-module-risk` 中 `release` 子域的一等能力来组织，而不是额外新增物理模块。**

建议包括：

### 核心协作者

- `SceneConfigLoader`
- `SceneConfigValidator`
- `DependencyAnalyzer`
- `SceneSnapshotCompiler`
- `ReleasePublisher`

### 关键能力

- 加载设计态对象
- 规则依赖分析
- 特征依赖分析
- 表达式校验
- Groovy 语法校验
- 运行时顺序展开
- 快照组装
- checksum 生成

### 为什么先不单独拆模块

因为：

- 当前发布编译只服务于控制平台发布中心
- 单独拆模块的收益暂时低于维护成本
- 把它组织成 `release` 子域内部的清晰协作者，同样能保持编译器边界

这样既能保持“发布是一等能力”的架构思想，也不会破坏当前工程的精简性。

## 22.4.5 `pulsix-server` + `pulsix-module-risk`：控制平台后端

控制平台后端不再建议抽象成一个笼统的 `admin` 模块，而是拆成：

- `pulsix-server`：标准 Spring Boot 启动器，负责聚合模块与启动进程
- `pulsix-module-risk`：控制面核心业务模块，负责风控平台主业务闭环

其中 `pulsix-module-risk` 的职责包括：

### 系统管理依赖

- 借助 `pulsix-module-system` 提供登录鉴权、用户/角色/菜单、操作审计

### 配置管理

- 场景管理
- 事件模型管理
- 特征中心
- 名单中心
- 规则中心
- 策略中心

### 发布管理

- 快照编译
- 版本生成
- 发布
- 回滚
- 发布记录查询

### 仿真测试

- 单条事件仿真
- 用例管理
- 结果对比

### 查询分析

- 决策日志查询
- 命中明细查询
- Dashboard 基础统计

### 建议的包结构

```latex
cn.liboshuai.pulsix.module.risk
├── controller/
│   └── admin/
├── service/
│   ├── scene/
│   ├── event/
│   ├── feature/
│   ├── list/
│   ├── rule/
│   ├── policy/
│   ├── release/
│   ├── simulation/
│   └── log/
├── dal/
│   ├── dataobject/
│   └── mysql/
├── convert/
├── api/
├── mq/
├── enums/
└── framework/
```

### 为什么推荐这种结构

因为你这个项目虽然不需要搞很重的 DDD，但也不应该所有东西都塞在 `service.impl`，同时又要和当前 `system / infra` 模块风格保持一致。

## 22.4.6 `pulsix-engine`：Flink 决策引擎

这是另一个核心模块。

它应专注于：

- 消费事件流
- 消费快照流
- 维护广播状态和 keyed state
- 计算 stream feature
- 做 Redis lookup
- 执行 `pulsix-kernel`
- 输出决策结果与日志

### 建议包结构

```latex
cn.liboshuai.pulsix.engine
├── job/
├── source/
├── sink/
├── process/
├── broadcast/
├── feature/
│   ├── stream/
│   ├── lookup/
│   └── derived/
├── runtime/
├── state/
├── metrics/
├── serializer/
└── config/
```

### 核心子模块建议

#### `broadcast/`

- 处理快照广播流
- 维护 Broadcast State
- 本地 runtime cache 构建

#### `feature/stream/`

- COUNT/SUM/MAX/LATEST 等执行器
- bucket 状态处理
- TTL 与 timer

#### `feature/lookup/`

- Redis 查询封装
- 本地 cache
- fallback 逻辑

#### `runtime/`

- `CompiledSceneRuntimeHolder`
- `SceneRuntimeManager`
- 版本切换逻辑

#### `process/`

- 主链路 `KeyedBroadcastProcessFunction`
- 事件进入后的上下文组装与执行

### 一个非常关键的要求

`pulsix-engine` 不应该依赖 `pulsix-server`。

它只应该依赖：

- `pulsix-framework/pulsix-common`
- `pulsix-framework/pulsix-kernel`

最多再依赖少量快照/编译相关模型，但不应依赖后台 Controller/Service/Mapper。

## 22.4.7 造数、压测与演示脚本：当前阶段先不单独拆模块

这一类能力很有价值，但当前阶段我不建议单独新增 Maven 模块。

更务实的做法是先放到：

- `docs/` 中的示例数据与演示说明
- `deploy/` 或 `scripts/` 中的联调脚本
- 测试工程中的回放 / 造数辅助代码

### 为什么先不单独拆模块

因为这样可以：

- 降低模块数量和维护成本
- 让主链路先聚焦在控制面、执行内核、Flink 引擎三者打通
- 等真正出现独立 CLI / 压测工具诉求时再抽离

## 22.5 模块依赖关系应该怎么控制

一个多模块项目最怕的不是模块多，而是依赖乱。

我建议你用下面这个依赖方向：

```latex
pulsix-framework/pulsix-common
   ↑
   ├── pulsix-module-system
   ├── pulsix-module-infra
   ├── pulsix-module-risk
   └── pulsix-framework/pulsix-kernel

pulsix-framework/pulsix-kernel
   ↑
   ├── pulsix-module-risk
   └── pulsix-engine

pulsix-server
   └── 聚合 pulsix-module-system + pulsix-module-infra + pulsix-module-risk
```

也就是说：

- `pulsix-server` 和 `pulsix-engine` 共享底层核心模块
- `pulsix-engine` 不依赖 `pulsix-server`
- `pulsix-kernel` 不依赖 Spring Boot Web 层
- 发布编译能力先内聚在 `pulsix-module-risk` 的 `release` 子域，不额外增加物理模块

### 这套依赖方向的意义

它保证了：

- 引擎独立
- 风控主业务边界清楚
- 规则执行逻辑可复用
- 平台后台不把运行时逻辑写死在自己里面

这对于后续维护至关重要。

## 22.6 哪些数据结构和接口应该优先稳定下来

一个项目从 0 到 1 开发时，不是所有东西都要同时稳定。你需要先稳定“骨架接口”。

我建议你优先稳定下面这几类：

### 1）标准事件模型 `RiskEvent`

至少明确：

- eventId
- sceneCode
- eventType
- eventTime
- traceId
- entity fields
- ext

因为：

- Flink 输入靠它
- 仿真输入靠它
- 日志输出也要能追溯到它

---

### 2）运行时快照模型 `SceneSnapshot`

至少明确：

- scene
- features
- rules
- policy
- runtimeHints
- version
- checksum

因为：

- 发布机制靠它
- Flink 广播靠它
- 本地编译缓存靠它

---

### 3）执行上下文 `EvalContext`

至少明确：

- base fields
- feature values
- lookup values
- derived values
- metadata

因为：

- 表达式执行依赖它
- Groovy 执行依赖它
- 仿真和线上共用它

---

### 4）决策结果 `DecisionResult`

至少明确：

- sceneCode
- version
- finalAction
- score
- hitRules
- hitReasons
- latencyMs
- traceId

因为：

- 下游输出靠它
- 日志记录靠它
- 查询分析靠它

---

### 5）发布接口与仿真接口

建议尽早固定：

- `/release/publish`
- `/release/rollback`
- `/simulation/evaluate`

因为这三类接口能尽快打通“控制面 -> 快照 -> 执行”这条主线。

## 22.7 从 0 到 1 的推荐开发顺序

这一部分是本章最实战的部分。

我不建议你按“页面模块一个个做完”这种方式推进。更好的方法是：

> **按主链路打通顺序推进。**

下面给你一版非常适合个人项目的编码顺序。

---

## 22.7.1 第一阶段：先打骨架，不碰复杂业务

目标：

- 工程跑起来
- 模块边界搭好
- 基础环境可启动

建议顺序：

### 第 1 步：初始化多模块工程

- 父工程 `pom.xml`
- 子模块结构
- 基础依赖管理
- Spotless/Checkstyle（可后置）

### 第 2 步：搭建 `pulsix-server`

- Spring Boot 启动类
- MySQL 连接
- MyBatis-Plus / JPA（建议 MyBatis-Plus）
- 基础鉴权
- 健康检查接口

### 第 3 步：搭建 `pulsix-engine`

- Flink 本地 main
- Kafka Source 占位
- 简单 print sink

### 第 4 步：搭建 `pulsix-ui`

- 登录页
- 基础 layout
- 菜单框架

### 第 5 步：准备 `docker-compose`

- MySQL
- Redis
- Kafka
- Zookeeper / Kraft 模式
- 可选 Flink mini cluster

这一阶段不要急着做规则。先把系统骨架稳定下来。

---

## 22.7.2 第二阶段：先把“设计态对象管理”做出来

目标：

- 控制平台具备基础配置能力

建议先做：

- scene
- event schema
- feature
- rule
- policy

注意：
这阶段先把 CRUD 做完整，但不要急着做花哨页面。重点是：

- 表结构定下来
- API 稳下来
- 模型边界稳下来

这一步完成后，你的平台开始有“控制面”的雏形。

## 22.7.3 第三阶段：做快照编译器

这一阶段是项目第一次进入“平台化”阶段。

建议顺序：

### 1）定义 `SceneSnapshot` 模型

### 2）实现 `SceneSnapshotCompiler`

### 3）实现依赖分析器

### 4）实现表达式基础校验

### 5）实现发布接口

### 6）把 snapshot\_json 写入 `scene_release`

到这里，你应该已经能做到：

- 后台维护规则
- 点击发布
- 生成一份结构化 snapshot

即使此时 Flink 还没完全接上，也已经很关键了。

## 22.7.4 第四阶段：先打通“最小执行链路”

建议不要一上来就全量特征、多种规则引擎。先做一个最小可运行链路。

### 最小链路建议

- 一个场景： `TRADE_RISK`
- 一个 stream feature： `user_trade_cnt_5m`
- 一个 lookup feature： `device_in_blacklist`
- 一个 derived feature： `high_amt_flag`
- 两条规则
- 一个 FIRST\_HIT policy

### 目标

让下面这条链路先真正跑起来：

- 后台发布 snapshot
- Flink 收到 snapshot
- Kafka 收到事件
- Flink 构建上下文
- 执行规则
- 输出决策结果

这是项目里第一个“里程碑级胜利”。

---

## 22.7.5 第五阶段：补齐仿真能力

这一步非常建议紧跟在最小执行链路之后做。

原因是：

- 它能逼你抽出统一的规则执行内核
- 能尽早验证控制面和引擎是否一致
- 能大幅增强平台展示效果

建议做法：

- `pulsix-server` 调用 `pulsix-kernel`
- 读取某个已发布 snapshot
- 输入一条测试事件
- 返回命中规则、最终结果、命中原因

这个功能做出来以后，你的项目平台感会非常明显。

## 22.7.6 第六阶段：扩充特征与规则能力

当主链路跑通后，再逐步扩展：

### Stream Feature

- COUNT
- SUM
- MAX
- LATEST
- DISTINCT\_COUNT（先简单版）

### Lookup

- Redis Set
- Redis Hash
- 默认值

### Rule Engine

- Aviator / DSL
- 命中原因模板

### Policy

- FIRST\_HIT

这一步不要跳过，因为很多项目跑通一个 Demo 后就停住了。你需要把它稍微平台化一些。

## 22.7.7 第七阶段：补齐日志、回滚、监控

当执行主链路稳了以后，再做：

- 决策日志落库

## 22.9 哪些地方最容易把项目写乱

我提前帮你指出几个高风险点。

---

### 风险点 1：后台 Service 里直接写快照执行逻辑

比如你在 `pulsix-server` 里直接写：

- 表达式执行
- 规则命中
- 命中原因拼装
- 仿真逻辑

短期看没问题，但后面 Flink 也需要一样逻辑，最后会复制两份。

正确做法：

- 提前抽到 `pulsix-kernel`

---

### 风险点 2：Flink 直接依赖后台数据库表结构做运行逻辑

这会破坏“设计态 / 运行态分离”。

正确做法：

- Flink 只认 snapshot

---

### 风险点 3：所有模块都放 DTO/VO/PO 混在一起

时间一久会非常乱。

正确做法：

- `pulsix-framework/pulsix-common/biz/risk` 统一放共享领域模型与 DTO
- `pulsix-module-risk` 放控制平台页面请求响应对象
- `pulsix-engine` 放引擎内部 process 结构

---

### 风险点 4：先做很多页面，再考虑执行内核

这样最后容易变成“好看后台 + 假执行链路”。

正确做法：

- 先打通发布 -> Flink -> 决策 -> 日志
- 页面随着主链路补齐

---

### 风险点 5：一开始就全量上 Groovy

这很容易把系统复杂度炸起来。

正确做法：

- 一期先 Aviator / DSL
- Groovy 作为二期高级能力

## 22.11 你后面写代码时应坚持的工程原则

最后，我再给你几条非常实用的工程原则。

### 原则 1：所有“执行语义”尽量集中在 `pulsix-kernel`

不要让规则执行逻辑分散到：

- admin Service
- Flink ProcessFunction
- simulation Controller

### 原则 2：所有“发布语义”尽量集中在 `pulsix-module-risk` 的 `release` 子域

不要一半在 feature service、一半在 rule service、一半在 policy service。

### 原则 3：Flink 只认运行时快照，不认设计态表

这条原则必须非常坚定。

### 原则 4：先让链路成立，再追求抽象优雅

一期先求可跑、可讲、可验证，不要一开始过度设计。

### 原则 5：仿真和线上一致性优先级极高

只要出现“仿真结果和线上结果不一致”，平台可信度会大幅下降。

### 原则 6：尽量让文档和代码同步成长

每完成一个关键模块，就补：

- README
- DDL
- 架构图
- 示例配置

这会极大降低后期整理成本。

## 22.12 本章小结

这一章本质上在回答：

> **你已经知道系统该怎么设计了，现在代码到底该怎么组织，项目应该怎么一步一步落下去。**

核心结论可以总结为：

1. 采用 **多模块单仓库** 是最适合当前 `pulsix` 的工程形态

2. 模块边界建议为：
   - `pulsix-framework/pulsix-common`
   - `pulsix-framework/pulsix-kernel`
   - `pulsix-server`
   - `pulsix-module-system`
   - `pulsix-module-infra`
   - `pulsix-module-risk`
   - `pulsix-engine`
   - `pulsix-ui`

3. 最关键的抽象有两个：
   - **统一执行内核**
   - **发布编译能力**（先内聚在 `pulsix-module-risk/release`）

4. 开发顺序必须围绕主链路：
   - 控制面建模
   - 快照发布
   - Flink 执行
   - 仿真与日志

5. 共享 DTO 与轻量运行时模型统一收敛到 `pulsix-framework/pulsix-common/biz/risk`
