## 第 9 章：运行时快照的结构设计

---

### 9.1 这一章解决什么问题

上一章我们已经把一个非常关键的问题讲清楚了：

> **为什么实时风控平台必须有发布机制，而且发布结果必须是一份运行时快照。**

现在你已经知道：

- 设计态多表适合人编辑
- 运行态配置适合 Flink 执行
- 发布的本质是“编译”
- Flink 不应该直接读取几十张设计态表

但接下来马上会出现一个更具体、也更关键的问题：

> **那这份运行时快照到底应该长什么样？**

这不是一个“JSON 随便设计一下”的小问题，而是整个系统工程化的核心问题之一。

因为快照结构一旦设计得不合理，后面很多地方都会难受：

- Flink 不知道怎么消费
- 表达式/Groovy 不知道依赖哪些变量
- 规则和策略执行顺序不清楚
- 配置广播不好做
- 本地编译缓存不好组织
- 回滚和追溯不方便
- 仿真和线上很难统一

所以这一章要解决的是：

1. 运行时快照到底应该承载哪些信息
2. 快照结构为什么必须自包含、去关系化、可恢复
3. scene、feature、rule、policy 在快照中该怎么表示
4. 变量依赖关系该如何表达
5. 快照里的“执行计划”应该体现到什么程度
6. Java POJO 应该怎么设计，才能既适合控制平台编译，又适合 Flink 消费

你可以把这一章理解为：

> **这一章要定义“Flink 真正要吃下去的那份配置数据模型”。**

也就是：

- 第 7 章定义设计态模型
- 第 8 章定义发布机制
- **第 9 章定义运行态模型**

这是三者之间最关键的连接点。

---

### 9.2 先说核心结论：快照不是“配置导出”，而是“运行时执行计划”

很多人第一次设计快照时，容易这么想：

- 把数据库里的 rule、feature、policy 查出来
- 原样转成 JSON
- 存一下
- 发给 Flink

这听起来好像也能工作，但它其实只完成了“导出”，并没有真正完成“运行态建模”。

更准确的理解应该是：

> **快照不是设计态配置的简单 JSON 化，而是面向运行时执行的一份结构化执行计划。**

这里的关键词有两个：

### 1）结构化

也就是快照中的对象必须清晰可识别：

- scene
- stream features
- lookup features
- derived features
- rules
- policy
- runtime hints

而不是一坨散乱字段。

### 2）执行计划

也就是快照不只是“记录你配了什么”，而是要表达：

- 这些配置该按什么顺序执行
- 哪些变量依赖哪些特征
- 哪些规则依赖哪些特征
- 哪些特征要查 Redis
- 哪些特征要先算
- 哪些配置会影响 Flink State 行为

所以你可以把快照理解成：

> **Flink 引擎的场景级运行脚本说明书**

只是它不是任意脚本，而是一份受控的、结构化的、可恢复的运行计划描述。

---

### 9.3 一个好的快照必须满足哪些原则

我建议你用下面 7 条原则来约束快照设计。

---

## 9.3.1 原则 1：自包含

所谓自包含，就是：

> Flink 拿到这份快照之后，不需要再回头查设计态多表，才能理解怎么执行。

也就是说，快照中至少要包含：

- 这个场景是什么
- 需要哪些特征
- 哪些规则参与执行
- 规则依赖什么
- 策略模式是什么
- 默认动作是什么

如果你的快照还需要 Flink 再去查：

- 哪条 rule 对应哪条 policy
- 这个 feature 的配置在哪张表
- 这个 derived feature 依赖哪个字段

那它就不算真正自包含。

---

## 9.3.2 原则 2：去关系化

设计态表天生是关系型的：

- policy 和 rule 是关联表
- feature 和配置是主子表
- list 和 scene 也是关联关系

但运行态不应该再让 Flink 临时理解这些关系。

所以发布时必须把关系展开。

例如：

- `policy_rule_ref` 最终应该展开成快照里明确的 ruleOrder
- `feature_def + feature_stream_conf` 最终应该展开成一个完整的 streamFeatureSpec
- 名单引用最终应变成明确的 lookupFeatureSpec

也就是说：

> **设计态关系，在发布时被摊平；运行态快照只保留执行所需结构。**

---

## 9.3.3 原则 3：可恢复

快照最终是要进入 Flink Broadcast State 的。 这意味着它必须：

- 可以序列化
- 可以 checkpoint
- 可以恢复
- 不依赖不可恢复的瞬时对象

因此：

- 原始 JSON / POJO 可以进 Broadcast State
- Groovy 编译结果不应该直接进 Broadcast State
- 表达式编译对象也不应该直接进 Broadcast State

换句话说：

> **Broadcast State 中放的是“配置事实”，不是“不可序列化运行时对象”。**

---

## 9.3.4 原则 4：稳定可追溯

快照一定要能回答：

- 这是谁发布的
- 是哪个版本
- 什么时间生效
- 内容摘要是什么
- 某次决策用了哪个版本

所以元信息是必须的，而不是可选项。

例如：

- snapshotId
- sceneCode
- version
- checksum
- publishedAt
- effectiveFrom

这些字段不是为了“好看”，而是为了：

- 追溯
- 回滚
- 排障
- 对账

---

## 9.3.5 原则 5：面向执行，而不是面向页面

快照不是前端页面 DTO，也不是管理表导出结果。 它是给运行时引擎用的。

所以你设计快照时，优先考虑的是：

- Flink 怎么读取
- 引擎怎么定位依赖
- 表达式怎么拿变量
- 本地缓存怎么构建

而不是：

- 页面怎么直接显示得漂亮

页面如果要看快照，可以额外转展示格式； 但快照本身应该优先服务执行端。

---

## 9.3.6 原则 6：边界清晰

快照里应该明确区分：

- stream feature
- lookup feature
- derived feature
- rule
- policy

不要做成所有对象都在一个大数组里，再靠 type 来区分一切。 那会让消费端逻辑越来越难维护。

当然，底层 Java 类可以有抽象父类， 但在快照结构上，建议显式分层。

---

## 9.3.7 原则 7：为本地编译缓存留出足够信息

Flink 运行时通常不会直接逐条解释 JSON。 更合理的方式是：

- Broadcast State 保存原始快照
- Task 本地构建 `CompiledSceneRuntime`

那么快照必须提供足够信息，让引擎能生成：

- 已编译表达式
- 已编译 Groovy invoker
- 特征执行计划
- 规则执行顺序
- 依赖图

所以快照里最好有：

- dependsOn
- engineType
- valueType
- priority / ruleOrder
- runtimeHints

这些字段会大幅简化引擎实现。

---

### 9.4 推荐的快照整体结构

下面我先给你一版推荐的整体结构，再逐段解释。

```json
{
  "snapshotId": "TRADE_RISK_v12",
  "sceneCode": "TRADE_RISK",
  "sceneName": "交易风控",
  "version": 12,
  "status": "ACTIVE",
  "checksum": "9d8c1abf...",
  "publishedAt": "2026-03-07T20:00:00Z",
  "effectiveFrom": "2026-03-07T20:00:10Z",

  "eventSchema": {
    "eventType": "trade",
    "requiredFields": ["eventId", "eventTime", "userId", "deviceId", "ip", "amount"]
  },

  "variables": {
    "baseFields": [
      "eventId", "eventTime", "traceId", "userId", "deviceId", "ip", "amount", "result"
    ]
  },

  "streamFeatures": [
    {
      "code": "user_trade_cnt_5m",
      "name": "用户5分钟交易次数",
      "sourceEventTypes": ["trade"],
      "entityType": "USER",
      "entityKeyExpr": "userId",
      "aggType": "COUNT",
      "valueExpr": "1",
      "filterExpr": "result == 'SUCCESS'",
      "windowType": "SLIDING",
      "windowSize": "5m",
      "windowSlide": "1m",
      "includeCurrentEvent": true,
      "ttl": "10m",
      "valueType": "LONG"
    }
  ],

  "lookupFeatures": [
    {
      "code": "device_in_blacklist",
      "name": "设备是否在黑名单",
      "lookupType": "REDIS_SET",
      "keyExpr": "deviceId",
      "sourceRef": "blacklist:device",
      "defaultValue": false,
      "valueType": "BOOLEAN"
    }
  ],

  "derivedFeatures": [
    {
      "code": "high_amt_flag",
      "name": "高金额标记",
      "engineType": "AVIATOR",
      "expr": "amount >= 5000",
      "dependsOn": ["amount"],
      "valueType": "BOOLEAN"
    }
  ],

  "rules": [
    {
      "code": "R001",
      "name": "黑名单设备直接拒绝",
      "engineType": "AVIATOR",
      "priority": 100,
      "whenExpr": "device_in_blacklist == true",
      "dependsOn": ["device_in_blacklist"],
      "hitAction": "REJECT",
      "score": 100,
      "hitReasonTemplate": "设备命中黑名单"
    }
  ],

  "policy": {
    "policyCode": "TRADE_RISK_POLICY",
    "decisionMode": "FIRST_HIT",
    "defaultAction": "PASS",
    "ruleOrder": ["R001"]
  },

  "runtimeHints": {
    "requiredStreamFeatures": ["user_trade_cnt_5m"],
    "requiredLookupFeatures": ["device_in_blacklist"],
    "requiredDerivedFeatures": ["high_amt_flag"]
  }
}
```

这只是示意版。 但它已经体现了快照设计的几个关键思想：

- 有元信息
- 有场景信息
- 有变量边界
- 有三类 feature
- 有 rule
- 有 policy
- 有 runtime hints

下面我们逐段拆开。

---

### 9.5 快照的元信息部分应该怎么设计

这一段是所有快照的“头部”。

建议至少有这些字段：

- `snapshotId`
- `sceneCode`
- `sceneName`
- `version`
- `status`
- `checksum`
- `publishedAt`
- `effectiveFrom`

---

## 9.5.1 `snapshotId`

建议可读性强一些，例如：

- `TRADE_RISK_v12`
- `LOGIN_RISK_v6`

它的作用是：

- 日志展示方便
- 人工排查方便
- 本地缓存 key 也方便

底层你也可以只用 `sceneCode + version`， 但展示上有个 snapshotId 会更友好。

---

## 9.5.2 `sceneCode`

这个字段是场景执行边界。 Flink 后面通常就是按它来查当前快照。

---

## 9.5.3 `version`

数值型版本号更适合比较和排序。 例如：

- 11
- 12
- 13

---

## 9.5.4 `checksum`

建议基于 snapshot 内容计算。 它的作用包括：

- 去重发布
- 校验一致性
- 排查“版本号一样但内容不对”类问题

---

## 9.5.5 `publishedAt` 与 `effectiveFrom`

这两个字段帮助你区分：

- 什么时候发布成功
- 什么时候开始生效

后续如果你支持定时生效、灰度发布、延迟切换，这两个字段会非常有用。

---

### 9.6 scene 信息和 event schema 应该怎么表示

很多人快照里只放 rule 和 feature，忽略 scene 和 event schema。 这其实不够完整。

因为运行时除了知道“怎么算”，还要知道：

- 这是哪个场景
- 输入事件的大致边界是什么
- 哪些字段是基础字段

---

## 9.6.1 scene 信息

建议至少包含：

- sceneCode
- sceneName
- 可选的 sceneType

它的意义不只是展示，也有利于：

- 多场景统一引擎
- 日志解释
- 本地缓存组织

---

## 9.6.2 eventSchema 摘要

建议快照里不要原封不动塞整套复杂事件定义， 但应保留运行时真正需要的摘要。

例如：

- 主 eventType
- requiredFields
- 可选字段清单
- 可选字段类型提示

为什么需要这些？

### 1）运行时校验

至少可以在引擎里做一些轻量级检查。

### 2）仿真统一

仿真页也可以复用这份摘要来辅助输入和校验。

### 3）表达式依赖边界

比如 `amount` 到底是不是合法变量来源。

---

### 9.7 为什么快照里要单独列出 variables.baseFields

这一点很多人第一次做时不会专门建。 但我强烈建议保留。

例如：

```json
"variables": {
  "baseFields": ["eventId", "eventTime", "userId", "deviceId", "ip", "amount", "result"]
}
```

这样做的好处有几个：

### 1）定义变量边界

让引擎和控制平台都知道：

- 哪些变量来自原始事件
- 哪些变量来自 feature
- 哪些变量来自 derived feature

### 2）便于依赖检查

比如某条规则引用了 `merchantRiskLevel`，但它不在 baseFields，也不在 features 中，那发布时就该报错。

### 3）便于表达式引擎上下文构建

后面构建 `EvalContext` 时，你可以非常清楚地知道上下文来源。

所以我建议：

> **快照里显式声明 baseFields，是一个很实用的工程细节。**

---

### 9.8 stream feature 在快照中应该怎么表示

这一部分是你后面 Flink 引擎实现的关键基础。

stream feature 不是一个“变量名”就够了， 它必须包含足够多的执行语义。

建议至少包含下面这些字段：

- `code`
- `name`
- `sourceEventTypes`
- `entityType`
- `entityKeyExpr`
- `aggType`
- `valueExpr`
- `filterExpr`
- `windowType`
- `windowSize`
- `windowSlide`
- `includeCurrentEvent`
- `ttl`
- `valueType`

---

## 9.8.1 `entityKeyExpr`

非常关键。 它告诉 Flink：

- 这个特征围绕谁聚合
- state key 应该怎么取

例如：

- `userId`
- `deviceId`
- `ip`

---

## 9.8.2 `aggType`

建议先限制在平台支持的固定集合里，例如：

- COUNT
- SUM
- MAX
- LATEST
- DISTINCT\_COUNT

这样 Flink 执行器就能模板化实现，而不是任意脚本化。

---

## 9.8.3 `valueExpr`

例如：

- COUNT 时可以是 `1`
- SUM 时可以是 `amount`
- DISTINCT\_COUNT 时可以是 `userId`

它表示：

- 聚合时取哪个值

---

## 9.8.4 `filterExpr`

例如：

- `result == 'SUCCESS'`

这允许你在模板内做受控动态化。

---

## 9.8.5 `windowType / windowSize / windowSlide`

这些字段是流式特征执行的核心语义。 建议限制在平台支持的范围内，例如：

- windowType：SLIDING / TUMBLING / NONE
- windowSize： `5m`、 `30m`、 `1h`
- windowSlide： `1m`、 `5m`

注意：
这些字段虽然可以动态配置，但你平台必须明确只支持有限模型。 不能任意开放成“随便写 Flink window 程序”。

---

## 9.8.6 `includeCurrentEvent`

这个字段很有价值。 因为有些系统要的是：

- 当前事件之前的历史值

有些要的是：

- 包含当前事件后的实时值

这个差异会直接影响规则判断结果。

所以建议在快照里明确表达。

---

## 9.8.7 `ttl`

它告诉引擎：

- 状态理论上可保留多久
- 清理策略大致如何做

这既影响状态大小，也影响恢复策略。

---

### 9.9 lookup feature 在快照中应该怎么表示

lookup feature 代表：

- 不依赖 Flink state
- 依赖外部在线数据源
- 通常是 Redis / KV / 字典

建议字段至少有：

- `code`
- `name`
- `lookupType`
- `keyExpr`
- `sourceRef`
- `defaultValue`
- `valueType`
- 可选 `cacheHint`
- 可选 `timeoutMs`

---

## 9.9.1 `lookupType`

建议做成明确枚举：

- REDIS\_SET
- REDIS\_HASH
- REDIS\_STRING
- DICT

不要让运行时去猜。

---

## 9.9.2 `keyExpr`

例如：

- `deviceId`
- `userId`

它定义了 lookup key 如何从事件上下文中取出。

---

## 9.9.3 `sourceRef`

例如：

- `blacklist:device`
- `profile:user:risk`

这个字段不是要把完整 Redis key 全写死，而是给 lookup executor 一个明确来源描述。

---

## 9.9.4 `defaultValue`

非常重要。 因为热路径中你必须决定：

- 查不到时怎么办
- 超时时怎么办

如果这个默认值不在快照中明确，很容易让执行逻辑分散在代码里。

所以建议把默认值作为配置的一部分下发。

---

### 9.10 derived feature 在快照中应该怎么表示

derived feature 是后面 rule 执行的重要中间层。

它通常基于：

- base field
- stream feature
- lookup feature

计算出新的变量。

建议字段：

- `code`
- `name`
- `engineType`
- `expr`
- `dependsOn`
- `valueType`
- 可选 `orderNo`

---

## 9.10.1 为什么一定要有 `dependsOn`

这是非常关键的字段。

因为 derived feature 本质上有依赖关系。 例如：

```latex
trade_burst_flag = user_trade_cnt_5m >= 3 && amount >= 5000
```

那它 clearly depends on：

- `user_trade_cnt_5m`
- `amount`

把 dependsOn 明确写入快照，有几个好处：

### 1）引擎知道上下文必须先准备哪些变量

### 2）可以做依赖拓扑排序

### 3）可以减少运行时猜测

### 4）便于日志解释和调试

所以我建议：

> **dependsOn 不要只在发布时内部用，应该作为快照结构的一部分保留下来。**

---

## 9.10.2 `engineType`

建议支持：

- AVIATOR
- DSL
- GROOVY

但不要什么都默认 GROOVY。 后面你会发现这对性能、安全、可维护性都有巨大影响。

---

### 9.11 rule 在快照中应该怎么表示

rule 是平台逻辑判断的核心对象。

建议字段：

- `code`
- `name`
- `engineType`
- `priority`
- `whenExpr`
- `dependsOn`
- `hitAction`
- `score`
- `hitReasonTemplate`
- 可选 `enabled`

---

## 9.11.1 `whenExpr`

它表示规则命中的条件表达式。 这是 rule 的核心。

---

## 9.11.2 `priority`

在 FIRST\_HIT 模式下尤其重要。 即便 policy 里还有 ruleOrder，rule 本身保留 priority 也有价值：

- 便于审计
- 便于调试
- 便于 fallback 排序

---

## 9.11.3 `hitAction`

例如：

- PASS
- REVIEW
- REJECT
- TAG\_ONLY

建议每条规则明确产出自己的动作建议， 最终是否作为最终动作，再由 policy 决定。

---

## 9.11.4 `score`

为 SCORE\_CARD 模式预留。 哪怕一期先主做 FIRST\_HIT，也建议结构上留出来。

---

## 9.11.5 `hitReasonTemplate`

这是一个很实用的字段。 例如：

- `用户5分钟交易次数={user_trade_cnt_5m}, amount={amount}`

这样命中日志就能更可解释。

注意：
这类模板建议保持简单，不要做成任意脚本化模板系统。

---

### 9.12 policy 在快照中应该怎么表示

很多人第一次设计时，容易把 policy 做得过轻，只放一个 `decisionMode`。 其实还不够。

建议 policy 至少包含：

- `policyCode`
- `decisionMode`
- `defaultAction`
- `ruleOrder`
- 可选 `scoreBands`

---

## 9.12.1 `ruleOrder`

即便每条 rule 有 priority，我仍然建议在 policy 中显式放 `ruleOrder`。 原因是：

### 1）执行顺序是策略语义，不只是规则属性

### 2）发布时已经可以把中间关联表展开好

### 3）引擎按这个顺序直接执行最简单

也就是说：

- rule priority 是对象自身属性
- policy ruleOrder 是运行时执行顺序

两个都保留，工程上更稳。

---

## 9.12.2 `scoreBands`

如果你支持评分卡，可以设计成：

```json
"scoreBands": [
  {"min": 0, "max": 39, "action": "PASS"},
  {"min": 40, "max": 79, "action": "REVIEW"},
  {"min": 80, "max": 999999, "action": "REJECT"}
]
```

这样引擎端就很容易统一实现。

---

### 9.13 为什么快照里建议增加 runtimeHints

这是一个非常实用、但不是所有人都会第一时间想到的设计点。

例如：

```json
"runtimeHints": {
  "requiredStreamFeatures": ["user_trade_cnt_5m"],
  "requiredLookupFeatures": ["device_in_blacklist"],
  "requiredDerivedFeatures": ["high_amt_flag"]
}
```

它的价值在于：

### 1）帮助 Flink 快速知道当前场景需要准备什么

而不是每次都从 rule/feature 全量遍历推导。

### 2）帮助做本地缓存构建

例如：

- 先编译哪类表达式
- 哪些 lookup executor 要准备

### 3）帮助调试与观测

你可以很清楚知道：

- 某个场景的运行时依赖有哪些

### 4）帮助仿真页复用

仿真页同样可以利用 hints 快速构建上下文。

所以 runtimeHints 虽然不是绝对必需，但我强烈建议加上。

---

### 9.14 变量依赖关系怎么表达才合理

这一段非常关键，因为它直接关系到：

- 发布校验
- 执行顺序
- 本地编译缓存
- 日志解释

我建议你至少在以下对象上保留 `dependsOn`：

- derived feature
- rule

有时 policy 也可以保留汇总依赖，但不是必须。

---

## 9.14.1 依赖表达的最小方案

最简单实用的方式就是：

```json
"dependsOn": ["amount", "user_trade_cnt_5m"]
```

这已经能解决很多问题。

---

## 9.14.2 要不要建复杂依赖图结构

可以，但一期不必太重。

例如你也可以设计：

```json
"dependencyGraph": {
  "high_amt_flag": ["amount"],
  "trade_burst_flag": ["user_trade_cnt_5m", "amount"],
  "R002": ["trade_burst_flag"]
}
```

但对个人项目来说，通常保留到发布器内部即可。 快照里把关键对象的 `dependsOn` 写清楚就够用了。

---

### 9.15 快照中要不要放“编译结果”

答案是：

> **不要把不可序列化的编译结果直接放进快照。**

这是非常重要的设计边界。

不建议放入快照或 Broadcast State 的内容包括：

- Groovy 编译后的 Class
- GroovyClassLoader
- Aviator 的 compiled object（如果不可稳定序列化）
- 各种本地执行器实例

为什么？

### 1）这些对象很可能不可序列化

### 2）容易和 Flink checkpoint 恢复冲突

### 3）容易引入 classloader 问题

### 4）不利于版本切换与恢复

正确做法是：

- 快照保存“原始表达式与元信息”
- Flink 收到快照后，在本地 transient cache 中编译

也就是：

> **快照负责“描述执行逻辑”，本地缓存负责“准备运行对象”。**

### 9.18 Flink 端为什么会更喜欢这种 POJO 结构

因为后面 Flink 拿到快照后，很自然就能构建：

```java
class CompiledSceneRuntime {
    private SceneSnapshot snapshot;
    private Map<String, StreamFeatureExecutor> streamExecutors;
    private Map<String, LookupInvoker> lookupInvokers;
    private Map<String, ExpressionInvoker> derivedInvokers;
    private Map<String, RuleInvoker> ruleInvokers;
    private PolicyRuntime policyRuntime;
}
```

而如果你的快照本身只是乱糟糟的大 JSON， 那这一步会非常痛苦。

所以：

> **好的快照结构，能够自然过渡到好的本地运行时结构。**

这也是第 9 章真正的重要性。

### 9.19 一些容易踩坑的快照设计错误

下面我列一些非常常见的坑。

---

## 9.19.1 坑 1：快照直接照搬数据库结构

例如：

- rules 里还是 policyRuleRef 的样子
- feature 还得再查子表
- 变量依赖全没展开

这种快照本质上还是设计态，不是真正运行态。

---

## 9.19.2 坑 2：快照里缺少依赖信息

如果没有 dependsOn，引擎往往只能：

- 临时解析表达式
- 动态推断上下文
- 或者每次全量准备所有变量

这样实现要么低效，要么复杂。

---

## 9.19.3 坑 3：快照里塞入不可序列化编译对象

这会直接给 Broadcast State 和 checkpoint 恢复带来麻烦。

---

## 9.19.4 坑 4：快照没有版本元信息

这样你后面：

- 无法追溯
- 无法回滚
- 无法对比新旧版本

---

## 9.19.5 坑 5：快照过于追求“通用 DSL”，失去可控性

比如试图让快照支持：

- 任意状态机脚本
- 任意外部调用
- 任意自定义窗口逻辑

这会让引擎失控。 你应该做的是：

- 对 stream feature 用模板化描述
- 对 derived/rule 用表达式/Groovy
- 不把底层流计算语义完全脚本化

---

### 9.20 本章小结

这一章本质上回答的是：

> **运行时快照到底应该设计成什么结构，才能真正支撑 Flink 执行。**

我们把重点收一下。

#### 1）快照不是配置导出，而是运行时执行计划

它应该面向执行端，而不是面向后台页面。

#### 2）快照必须满足几个原则

- 自包含
- 去关系化
- 可恢复
- 可追溯
- 面向执行
- 边界清晰

#### 3）快照中建议显式包含这些部分

- 元信息
- scene / eventSchema
- variables.baseFields
- streamFeatures
- lookupFeatures
- derivedFeatures
- rules
- policy
- runtimeHints

#### 4）变量依赖关系应显式保留

尤其是：

- derived feature 的 dependsOn
- rule 的 dependsOn

#### 5）快照不应直接放不可序列化编译对象

编译结果应该进入 Flink 本地 transient cache，而不是 Broadcast State。

#### 6）Java POJO 设计要为 Flink 本地运行时结构服务

好的快照结构会自然过渡到：

- CompiledSceneRuntime
- RuleInvoker
- FeatureExecutor
- PolicyRuntime
