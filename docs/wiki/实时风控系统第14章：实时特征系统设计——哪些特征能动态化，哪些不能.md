## 第14章：实时特征系统设计——哪些特征能动态化，哪些不能

### 14.1 这一章解决什么问题

到了这一章，整个实时风控平台已经进入到真正的“引擎核心设计”阶段了。

前面几章我们已经讲清楚了：

- 控制平台如何管理设计态配置
- 为什么发布时必须生成运行时快照
- Flink 在系统中的角色是什么
- Flink 如何接收、存储和切换快照

但真正让风控平台变得“像个平台而不是几条规则”的关键，并不是规则本身，而是**特征系统**。

因为风控的很多判断都不是基于原始事件字段，而是基于这样一类信息：

- 用户 5 分钟交易次数
- 设备 1 小时关联账号数
- IP 10 分钟注册次数
- 当前用户风险等级
- 当前设备是否命中黑名单
- 本次事件是否属于“高额且高频”

这些都不是单纯的 event 字段，而是**特征 Feature**。

所以这一章的核心任务就是回答下面这些问题：

1. 什么是实时风控系统里的“特征”
2. 为什么特征系统比规则系统更难
3. 特征应该分成哪些类型
4. 哪些特征适合动态化，哪些不适合
5. Flink 到底能支持什么程度的动态特征
6. 为什么流式特征不能完全交给 Groovy 随便定义
7. 为什么要把“特征平台化能力”和“底层状态执行模型”分开
8. 修改特征时为什么要考虑 version、warm-up、兼容性
9. 你这个项目里应该把特征系统做到什么边界最合理

这一章实际上是后面第 15 章状态设计、第 16 章规则与策略执行、第 17/18 章表达式与 Groovy 执行器设计的前提。

你可以把这一章理解为：

> 它决定了控制平台到底能开放多大的“特征配置能力”，以及 Flink 引擎到底如何安全、稳定、可解释地执行这些特征。

---

### 14.2 为什么说特征系统比规则系统更核心

很多人一开始做风控平台，会天然把注意力放在“规则中心”上，觉得规则才是主角。

比如：

- 规则表达式怎么写
- 阈值怎么调
- 命中后返回什么动作
- 是否支持评分卡

这些当然重要，但如果把整个风控平台比作一个人，那么：

- **规则系统更像大脑中的判断逻辑**
- **特征系统更像感官和记忆系统**

因为没有特征，规则就几乎只能做静态判断：

- `amount > 5000`
- `channel == 'APP'`
- `province != lastProvince`

这种能力很有限。

而真正有价值的风控判断，往往依赖的是：

- 历史累积
- 实时行为序列
- 跨事件聚合
- 名单命中
- 在线画像
- 上下文推导

这些能力的承载体，都是特征。

所以风控平台的本质不是“规则配置平台”，而更准确地说是：

> **一个以特征构建为基础、以规则/策略执行为出口的脉流实时风控平台。**

这也是为什么成熟风控系统里，规则引擎往往不算最难，最难的是：

- 特征怎么定义
- 特征怎么计算
- 特征怎么解释
- 特征怎么热更新
- 特征怎么兼容版本
- 特征怎么保证性能和可恢复

---

### 14.3 什么是风控特征

在这个项目里，我们可以给“特征”一个工程上非常实用的定义：

> **特征是对当前事件、相关实体、历史行为、外部画像或名单信息的一种结构化变量表达，它能被规则与策略消费。**

特征的作用是把复杂上下文，变成规则可以直接使用的变量。

例如下面这些都可以视为特征：

- `amount`
- `user_trade_cnt_5m`
- `device_bind_user_cnt_1h`
- `ip_register_cnt_10m`
- `device_in_blacklist`
- `user_risk_level`
- `high_amt_flag`
- `trade_burst_flag`

如果进一步抽象，特征就是：

- **规则的输入**
- **上下文的变量化表达**
- **控制平台与运行时之间的桥梁之一**

---

### 14.4 为什么特征不能“一锅炖”

如果不做分类，所有东西都叫 feature，看起来好像统一了，实际上会导致执行边界混乱。

因为下面这几类 feature 的运行语义完全不同：

#### 1）原始字段类

例如：

- amount
- userId
- deviceId
- ip

它们直接来自事件。

#### 2）流式聚合类

例如：

- user\_trade\_cnt\_5m
- user\_trade\_amt\_sum\_30m
- device\_bind\_user\_cnt\_1h

它们依赖 Flink State、Window、Timer。

#### 3）外部查询类

例如：

- device\_in\_blacklist
- user\_risk\_level
- ip\_proxy\_flag

它们依赖 Redis 或外部存储查询。

#### 4）派生计算类

例如：

- high\_amt\_flag = amount >= 5000
- trade\_burst\_flag = user\_trade\_cnt\_5m >= 3 && amount >= 5000

它们依赖表达式/Groovy 对已有上下文进一步推导。

这四类 feature 的：

- 计算方式
- 热更新方式
- 性能开销
- 可动态化程度
- 可解释方式
- 风险点

都不一样。

所以如果你想把风控平台做得真正可控，必须先接受一个事实：

> **“特征”是统一概念，但不是统一执行模型。**

---

### 14.5 特征系统的推荐分层模型

我建议你把风控特征系统分成四层来理解。

---

#### 第一层：基础字段 Base Fields

这是事件自身提供的字段。

例如：

- eventId
- eventTime
- userId
- deviceId
- ip
- amount
- channel
- result

这层本质上不是“计算得来”的，但在规则上下文里，它们也会以变量形式存在。

它们的特点是：

- 无需计算
- 直接从事件提取
- 是后续特征和规则的基础依赖

---

#### 第二层：流式特征 Stream Features

这是由 Flink 状态计算得到的特征。

例如：

- 用户 5 分钟交易次数
- 用户 30 分钟交易金额和
- 设备 1 小时关联账号数
- IP 10 分钟注册次数
- 最近一次交易时间

它们的特点是：

- 依赖时间窗口或状态
- 依赖 keyBy / state / timer
- 最能体现 Flink 价值
- 最不适合完全交给任意脚本去定义

---

#### 第三层：外部查询特征 Lookup Features

这是运行时通过 Redis / 外部 KV 查询得到的特征。

例如：

- 设备是否在黑名单
- 用户风险等级
- IP 是否代理节点
- 商户风险标签

它们的特点是：

- 不一定需要 Flink 自己维护状态
- 适合做低延迟 lookup
- 通常要有默认值与超时策略
- 通常适合模板化动态配置

---

#### 第四层：派生特征 Derived Features

这是在基础字段、流式特征、lookup 特征基础上再次推导出来的特征。

例如：

- `high_amt_flag = amount >= 5000`
- `trade_burst_flag = user_trade_cnt_5m >= 3 && amount >= 5000`
- `high_risk_device_trade = device_in_blacklist == true && amount > 1000`

它们的特点是：

- 语义上仍然是 feature，不是最终 rule
- 特别适合表达式引擎
- 少量复杂逻辑可以考虑 Groovy
- 是平台灵活性的重要来源

---

### 14.6 为什么这四层划分非常重要

因为它直接决定了控制平台和 Flink 引擎的职责边界：

| 层级               | 主要来源        | 核心执行方式     | 是否适合高度动态化 |
| :--------------- | :---------- | :--------- | :-------- |
| Base Fields      | Event 本身    | 直接提取       | 不涉及       |
| Stream Features  | Flink State | 模板化状态执行器   | 适合“受控动态化” |
| Lookup Features  | Redis/KV    | Lookup 执行器 | 适合动态化     |
| Derived Features | 表达式/Groovy  | 解释执行/编译执行  | 非常适合动态化   |

这张表是你后面设计整个特征系统边界时最重要的依据之一。

请你记住这句话：

> **越靠近底层状态语义的特征，越不适合完全自由化；越靠近变量推导的特征，越适合动态化。**

---

### 14.7 什么叫“特征动态化”

“特征动态化”这个词很容易被理解得太宽。这里要讲清楚。

在这个项目里，所谓特征动态化，不是指用户可以随便写任何代码来定义特征，而是指：

> **控制平台能够通过配置方式定义特征的行为，并在发布后让 Flink/执行引擎按配置生效，而不需要改 Java 代码重新发版。**

这个定义有两个核心限定：

#### 限定 1：是“通过配置定义行为”

而不是靠手工改代码。

#### 限定 2：是“在平台允许的执行模型内”

而不是允许任意脚本接管底层状态机。

所以动态化不是无限自由，而是：

> **在可控语义边界内的配置化执行。**

---

### 14.8 哪些特征适合动态化

这一部分是整章最关键的设计边界。

---

#### 14.8.1 基础字段类：天然可用，不涉及动态化难题

例如：

- amount
- deviceId
- ip
- channel

这些字段直接由事件 schema 决定。它们不需要“动态计算”，只需要：

- 在控制平台中做字段定义
- 在表达式变量表中注册
- 在运行时正确提取即可

所以这类特征几乎不构成动态化风险。

---

#### 14.8.2 Lookup 特征：非常适合动态化

例如：

- `device_in_blacklist`
- `user_risk_level`
- `ip_proxy_flag`

这种特征通常只需要配置：

- lookup 类型（Redis Set / Hash / String）
- key 表达式（如 deviceId、userId）
- 默认值
- timeout
- 缓存 TTL

这类特征之所以适合动态化，是因为：

- 它不改变 Flink 底层状态模型
- 它只是在运行时做一次受控查询
- 执行语义清晰
- 容易解释和调试

所以 lookup feature 是平台最应该优先开放的一类动态特征能力。

---

#### 14.8.3 Derived 特征：非常适合动态化

这是控制平台灵活性的主战场之一。

例如：

- `high_amt_flag = amount >= 5000`
- `trade_burst_flag = user_trade_cnt_5m >= 3 && amount >= 5000`
- `risky_user_device_flag = user_risk_level in ['M','H'] && device_in_blacklist == true`

这类特征的特点是：

- 依赖上下文变量
- 不直接维护 Flink state
- 纯函数式、可解释
- 很适合表达式引擎

所以 derived feature 应当是控制平台开放程度最高的一类 feature。

如果平台要体现“低代码”或“规则可配置能力”，很大一部分灵活性其实应该由 derived feature 提供，而不是一上来就全开 Groovy。

---

#### 14.8.4 Stream 特征：适合“模板化动态化”

这类是最需要谨慎的。

例如：

- `user_trade_cnt_5m`
- `user_trade_amt_sum_30m`
- `device_bind_user_cnt_1h`
- `ip_register_cnt_10m`

它们适合动态化，但只能在受控模板内动态化。

可以开放的配置项包括：

- `entityKeyExpr`
- `aggType`
- `filterExpr`
- `valueExpr`
- `windowSize`
- `windowSlide`
- `includeCurrentEvent`
- `ttl`

也就是说：

> **你可以让用户配置“要按什么模板计算”，但不要让用户自由编写“如何自己实现一个状态机”。**

这就是我说的“模板化动态化”。

---

### 14.9 哪些特征不适合前期动态化

这部分同样非常重要。

---

#### 14.9.1 任意自定义状态机类特征

例如让用户通过脚本自由定义：

- 创建什么 state
- state 如何演化
- timer 怎么触发
- 水位线怎么处理
- 乱序怎么补偿

这种能力理论上很强，但工程上极其危险。

原因有：

- 无法有效校验
- 无法保证性能
- 无法保证可恢复
- 无法保证安全
- 无法做统一解释
- 容易让引擎退化为“在线脚本容器”

对你的项目来说，前期强烈不建议做。

---

#### 14.9.2 任意外部 IO 特征

例如让 feature 脚本直接：

- 查数据库
- 发 HTTP
- 调 RPC
- 读文件

这在实时风控热路径里会产生非常多问题：

- 延迟不可控
- 超时不可控
- 依赖管理复杂
- 重试语义不清晰
- 容易拖垮 Flink task

正确做法应该是：

- lookup feature 只开放平台支持的几种受控源
- 严格限制外部访问方式

---

#### 14.9.3 任意 CEP/复杂状态序列脚本

像这类场景：

- 登录失败 → 改密 → 提现
- 短时多次切换设备后大额交易
- 多步骤攻击路径识别

它们非常有价值，但不适合一开始作为完全动态自由脚本能力开放。

更合理的做法是：

- 一期先用 count/sum/max/latest/distinct 这类模板跑通
- 二期如果要做序列特征，再单独设计受控 CEP 模板或专用规则类型

---

### 14.10 风控特征系统的设计目标到底是什么

做特征系统不是为了“定义尽可能多的 feature”，而是为了在平台化和可控性之间取得平衡。

一个成熟的风控特征系统，至少应该满足下面几个目标。

---

#### 14.10.1 可表达

能表达核心风控场景下常见的特征：

- 次数类
- 金额类
- 最近值类
- 去重计数类
- 名单命中类
- 画像查询类
- 派生布尔类

---

#### 14.10.2 可执行

控制平台定义出来的特征，Flink 端真的能执行，
而不是只停留在配置页面层面。

---

#### 14.10.3 可解释

当你看到一条规则命中时，能够追溯：

- feature 是什么
- 它怎么来的
- 当前值是多少
- 它依赖哪些字段和其他 feature

---

#### 14.10.4 可版本化

特征定义是会变化的。系统必须支持：

- 版本演进
- 新旧兼容
- warm-up
- 回滚

---

#### 14.10.5 可控制

平台必须明确知道：

- 哪类特征允许动态配置
- 哪类只能按模板来
- 哪类暂不开放

不能把平台做成一个“无限制脚本执行器”。

---

#### 14.10.6 可观测

至少要能知道：

- 某个 feature 是否计算成功
- 哪类 feature 耗时高
- lookup 是否超时
- 某个特征 state 是否过大

---

### 14.11 Stream Feature 应该如何定义能力边界

这一节是全章里最重要的工程边界之一。

---

#### 14.11.1 Stream Feature 的本质

流式特征本质上是：

> **围绕某个实体 Entity，在某种时间语义下，对一类事件进行聚合、去重、最近值提取或其他状态演化后，形成的变量。**

它至少涉及下面几个核心元素：

- 输入事件类型
- 维度实体（user/device/ip...）
- entity key 提取方式
- filter 条件
- 聚合方式
- 窗口语义
- 状态 TTL
- 是否包含当前事件

这意味着：

- stream feature 一定涉及 state
- state 一定涉及 keyBy
- keyBy + time 语义 + state 结构，构成了底层执行模型

因此它不能像 derived feature 那样完全脚本化。

---

#### 14.11.2 一期推荐开放的 stream feature 类型

对你 6 个月项目来说，我建议一期只开放这几类，已经足够强：

1. `COUNT`
2. `SUM`
3. `MAX`
4. `LATEST`
5. `DISTINCT_COUNT`（可先简单精确版）

如果这五类做扎实，已经足够支撑：

- 登录风控
- 注册反作弊
- 交易风控

这比一上来卷复杂 CEP 更务实也更完整。

---

#### 14.11.3 Stream Feature 推荐开放的配置项

比如你可以允许用户配置：

- `sourceEventTypes`
- `entityType`
- `entityKeyExpr`
- `aggType`
- `filterExpr`
- `valueExpr`
- `windowType`
- `windowSize`
- `windowSlide`
- `ttl`
- `includeCurrentEvent`

但要注意，这里的表达式也只是：

- key 提取表达式
- filter 布尔表达式
- value 投影表达式

而不是让用户去写自己的状态机实现。

---

#### 14.11.4 为什么这叫“模板化执行器”

因为底层真正执行时，不是 Groovy 任意控制 Flink，而是平台内部有一套固定的执行器模板：

- CountFeatureExecutor
- SumFeatureExecutor
- MaxFeatureExecutor
- LatestFeatureExecutor
- DistinctCountFeatureExecutor

控制平台配置出来的是“参数”，而不是“新引擎”。

也就是说：

> **用户配置的是“特征意图”，引擎执行的是“内建模板”。**

这是平台可控的关键。

---

### 14.12 Lookup Feature 的设计重点

Lookup feature 往往是最容易做出平台价值的一类能力。

它非常适合风控，因为很多关键上下文来自外部存储：

- 名单
- 画像
- 风险标签
- 设备标签
- 区域标签

这类 feature 的设计重点通常包括：

- 查询源类型
- key 提取方式
- 默认值
- timeout
- 缓存 TTL
- 缓存穿透处理
- 降级策略

例如：

```latex
feature_code = device_in_blacklist
lookup_type = REDIS_SET
key_expr = deviceId
default_value = false
```

再比如：

```latex
feature_code = user_risk_level
lookup_type = REDIS_STRING
key_expr = userId
default_value = "L"
```

这类能力应当高度模板化，因为：

- 运行语义清晰
- 易于发布
- 易于解释
- 易于在 Flink 中统一调度

---

### 14.13 Derived Feature 的设计重点

Derived feature 在我看来，是风控平台里非常被低估、但实际上特别重要的一层。

它的价值在于：

- 把复杂的上下文先收敛成几个中间变量
- 让 rule 更短、更清晰、更可复用
- 降低规则表达式的复杂度
- 提升命中原因解释的可读性

例如，你完全可以不直接写规则：

```latex
amount >= 5000 && user_trade_cnt_5m >= 3 && user_risk_level in ['M','H']
```

而是先定义：

```latex
high_amt_flag = amount >= 5000
trade_burst_flag = user_trade_cnt_5m >= 3 && high_amt_flag
high_risk_user_flag = user_risk_level in ['M','H']
```

然后规则只写：

```latex
trade_burst_flag && high_risk_user_flag
```

这样会让整套配置：

- 更模块化
- 更可解释
- 更容易调试

所以在你的平台里，derived feature 应该是一个一等公民，而不是 rule 的附属物。

---

### 14.14 Feature 与 Rule 的边界到底是什么

这是一个非常容易混淆的问题。

---

#### Feature 的语义

Feature 更偏向：

- 描述状态
- 表达变量
- 中间推导
- 提供输入

例如：

- `device_in_blacklist`
- `high_amt_flag`
- `user_trade_cnt_5m`

---

#### Rule 的语义

Rule 更偏向：

- 进行判定
- 输出命中
- 提供动作建议或分值

例如：

- `if high_amt_flag && trade_burst_flag then REVIEW`
- `if device_in_blacklist then REJECT`

---

#### 为什么要分开

因为如果把所有中间逻辑都堆到 rule 里，会导致：

- rule 太复杂
- 可解释性变差
- 复用性变差
- 仿真展示困难

所以合理做法是：

> **Feature 负责把上下文变成变量，Rule 负责对变量做最终判断。**

这个分层对于控制平台建模、日志展示、仿真测试都很重要。

---

### 14.15 为什么特征版本管理比规则版本管理更复杂

规则修改通常是：

- 改个阈值
- 改个表达式
- 改个优先级

这种变化，一般不影响底层状态存储语义。

但特征不一样。

例如：

- 把 `user_trade_cnt_5m` 改成 `user_trade_cnt_10m`
- 把 `SUM(amount)` 改成 `COUNT(*)`
- 把按 `userId` 聚合改成按 `deviceId` 聚合
- 把 1 小时窗口改成 24 小时窗口

这些变化不是简单参数调整，而是在改变**特征语义本身**。

这会带来一个关键问题：

> 老状态还能不能复用？

很多时候答案是：不能直接安全复用。

所以特征的版本管理必须更谨慎。

---

### 14.16 特征变更应该如何分类

我建议你把 feature 变更分成三类理解。

---

#### 14.16.1 参数级变更

例如：

- lookup default value 改了
- derived expr 微调
- 命名描述改了
- cache TTL 改了

这类变更通常可以视为轻量变更。

---

#### 14.16.2 逻辑级变更

例如：

- derived feature 表达式语义明显变化
- lookup keyExpr 改了
- filterExpr 改了

这类会影响运行结果，但不一定破坏底层 state 结构。

---

#### 14.16.3 语义级变更

例如：

- entityKey 改了
- aggType 改了
- window 改了
- state 组织方式变了

这类变更应当高度谨慎，很多时候不应原地修改，而应：

- 新建新 feature code
- 或新建新 version
- 让新版本 warm-up

---

### 14.17 为什么 Stream Feature 经常要“新建而不是原地改”

假设你原来有：

- `user_trade_cnt_5m`

现在想改成：

- `user_trade_cnt_10m`

这看起来只是一个窗口值改动，但实际上线上会面临很多问题：

- 已有 state 是按 5 分钟语义累积的
- 现在改成 10 分钟后，历史状态怎么解释？
- 改完立刻生效，那前 10 分钟数据够不够？
- 新规则引用它时，是不是要经历预热过程？

所以更合理的做法往往不是“原地覆盖定义”，而是：

1. 新建 `user_trade_cnt_10m`
2. 发布新版本
3. 让它在 Flink 中逐步 warm-up
4. 新规则改引用新 feature
5. 旧 feature 逐步下线

这就是成熟系统里的“特征演进”思路。

---

### 14.18 什么是特征 warm-up，为什么它重要

Warm-up 可以理解为：

> **一个新的状态型特征从上线到具备完整历史语义之间的过渡阶段。**

举例：

- 你刚发布了一个 `user_trade_amt_sum_24h`
- 但新版本刚生效时，它不可能瞬间拥有过去 24 小时完整状态

那么在最初一段时间内：

- 它的值是不完整的
- 它可能只能反映“版本生效后的累计值”

所以你需要在平台设计上承认这一点：

- 新流式特征可能存在预热期
- 某些版本切换不能假设所有特征立刻完整
- 风控策略上有时要避免依赖刚上线的长窗口特征做强拦截

这就是为什么特征系统不能只讲配置，还必须讲运行语义。

---

### 14.19 特征系统在快照中的表达应该包含哪些信息

为了让 Flink 能真正执行特征，快照中至少要包含下面这些内容。

---

#### 对所有 feature 的通用信息

- featureCode
- featureName
- featureType
- valueType
- sceneCode
- description

---

#### 对 stream feature 的关键信息

- sourceEventTypes
- entityType
- entityKeyExpr
- aggType
- filterExpr
- valueExpr
- windowType
- windowSize
- windowSlide
- ttl
- includeCurrentEvent

---

#### 对 lookup feature 的关键信息

- lookupType
- keyExpr

---

#### 对 derived feature 的关键信息

- engineType

这些信息足够让 Flink 端构建一个明确的 feature execution plan。

---

### 14.20 Flink 端对 Feature 的使用方式应该是什么

到这里我们可以先给出一个很关键的执行原则：

> **Flink 不是逐条“理解 UI 配置”，而是根据快照中的 FeatureSpec 走统一的 feature 执行计划。**

也就是说，Flink 端看到的不是后台表，而是：

- 一组 StreamFeatureSpec
- 一组 LookupFeatureSpec
- 一组 DerivedFeatureSpec

然后引擎做的事情是：

1. 先提取 base fields
2. 再执行 stream feature
3. 再执行 lookup feature
4. 再执行 derived feature
5. 最终得到完整上下文供 rule 使用

这是一种明确的“分层上下文构建模型”。

---

### 14.21 为什么 Derived Feature 很适合表达式，而 Stream Feature 不适合完全交给 Groovy

这是你后面会反复关注的重点，我在这一章先把核心逻辑讲透。

---

#### Derived Feature 的本质是纯计算

它通常只是：

- 输入一个 context
- 输出一个值

例如：

```latex
amount >= 5000
user_trade_cnt_5m >= 3 && amount >= 5000
```

这类逻辑：

- 无状态
- 无副作用
- 可解释
- 容易编译和缓存

所以非常适合：

- DSL
- Aviator
- 少量 Groovy

---

#### Stream Feature 的本质是状态演化

它不是简单“给 context 算个值”，而是：

- 事件来了
- 要按 key 更新 state
- 可能要注册 timer
- 可能要清理 bucket
- 可能要合并窗口范围

这类逻辑的关键不是表达式本身，而是底层状态模型。

所以让 Groovy 随便控制 stream feature，容易出的问题包括：

- state 结构不可控
- checkpoint 恢复不稳定
- 性能难估算
- 脚本安全难保证
- 线上排障困难

所以正确边界是：

> **Derived Feature 可以主要靠表达式/Groovy，Stream Feature 必须主要靠平台模板执行器。**

---

### 14.22 你这个项目里，特征系统最合理的能力边界

如果结合你的目标：

- 6 个月内做成完整开源项目
- 要能用于求职加分
- 要体现 Flink + 风控 + 平台化能力

那么我建议特征系统这样分阶段。

---

#### 第一阶段：必须做的

1. Base Fields
2. Stream Feature：COUNT / SUM / MAX / LATEST
3. Lookup Feature：Redis Set / Redis String / Redis Hash
4. Derived Feature：表达式引擎

这一阶段已经足够强。

---

#### 第二阶段：增强能力

1. Stream Feature：DISTINCT\_COUNT
2. Derived Feature：支持 Groovy
3. Lookup Feature：增加本地 cache 与 timeout 控制
4. Feature dependency 图展示
5. Feature warm-up 状态提示

---

#### 第三阶段：高级能力

1. CEP/序列特征
2. 更复杂的近似 distinct（HLL / RoaringBitmap）
3. 高级画像融合
4. Feature 回放验证
5. 更完整的 feature version 迁移机制

---

### 14.23 平台实现时应该如何让特征“既灵活又不失控”

这是这一章最后的一个核心方法论。

我建议你采用下面这个原则：

> **让用户配置“想要什么特征”，而不是让用户自己定义“Flink 如何工作”。**

更具体一点说：

#### 对用户开放的

- feature code
- feature type
- entity key
- filter expr
- value expr
- window size
- lookup source
- derived expr

#### 对平台内建的

- 状态结构
- bucket 管理
- timer 清理逻辑
- checkpoint 恢复语义
- lookup 调用框架
- 执行顺序
- 错误处理与降级

这就是一个成熟平台应该有的“内核/配置”分离。

### 14.24 本章小结

这一章核心是在回答：

> **实时风控平台里的特征系统到底应该怎么设计，哪些特征能动态化，哪些不能。**

把重点收一下：

#### 1）特征比规则更基础

规则是判断逻辑，特征是上下文变量。没有特征系统，规则系统就很难有深度。

#### 2）特征必须分层

建议至少分为：

- Base Fields
- Stream Features
- Lookup Features
- Derived Features

#### 3）并不是所有特征都适合完全动态化

- Lookup / Derived 非常适合动态化
- Stream Feature 适合模板化动态化
- 任意状态机/任意外部 IO 不适合前期开放

#### 4）Flink 能很好承载平台化特征，但前提是执行模型要受控

- 状态型特征交给模板执行器
- 派生特征交给表达式/Groovy
- 不要让 Groovy 随意接管底层状态逻辑

#### 5）特征变更比规则变更更复杂

尤其是 stream feature，一旦改变窗口、聚合方式、entity key，往往就是语义级变更，应通过新 version / 新 code + warm-up 演进，而不是简单原地覆盖。

#### 6）特征系统的目标不是“无限灵活”，而是“在可控边界内足够灵活”

这才是真正可落地的平台化设计。
