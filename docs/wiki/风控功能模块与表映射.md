## 1. 核心结论

这篇文档只保留最重要的信息，帮助你快速理解 `docs/sql/pulsix-risk.sql` 的主线。

只记住这 5 句话即可：

- `scene_def` 是全局根对象，其他核心设计态对象都围绕 `scene_code` 组织。
- 主链路是：`场景 -> 事件 -> 特征/名单 -> 规则 -> 策略 -> scene_release -> 日志/仿真`。
- `scene_release` 不是普通配置表，而是**发布编译产物表**。
- `decision_log` / `rule_hit_log` 是运行结果表，不参与规则执行。
- 如果当前只想支撑 `kernel + engine` 开发，最该关注的是 `scene_release` 和主链路核心表。

---

## 2. 一张图看懂主链路

```text
scene_def
  -> event_schema / event_field_def / event_sample
  -> feature_def + feature_*_conf
  -> list_set / list_item
  -> rule_def
  -> policy_def + policy_rule_ref + policy_score_band
  -> scene_release
  -> simulation_* / decision_log / rule_hit_log / replay_job / risk_audit_log
```

可以把它理解为 6 层：

- 输入层：`event_schema`、`event_field_def`、`event_sample`、`ingest_*`
- 变量层：`feature_def`、`feature_*_conf`、`list_*`、`entity_type_def`
- 判断层：`rule_def`
- 收敛层：`policy_*`
- 发布层：`scene_release`
- 追溯治理层：`simulation_*`、`decision_log`、`rule_hit_log`、`risk_audit_log`、`replay_job`

## 4. 各层表的最小理解

### 4.1 场景与事件层

对应表：

- `scene_def`
- `event_schema`
- `event_field_def`
- `event_sample`
- `ingest_source`
- `ingest_mapping_def`

作用：

- `scene_def`：定义一个风控场景是什么。
- `event_schema`：定义场景接收哪些标准事件。
- `event_field_def`：定义标准事件字段、类型和校验规则。
- `event_sample`：保存样例报文，方便联调和仿真。
- `ingest_source` / `ingest_mapping_def`：定义原始接入如何映射为标准事件。

一句话：这一层解决“输入长什么样”。

### 4.2 特征与名单层

对应表：

- `entity_type_def`
- `feature_def`
- `feature_stream_conf`
- `feature_lookup_conf`
- `feature_derived_conf`
- `list_set`
- `list_item`

作用：

- `feature_def`：定义平台里有哪些变量。
- `feature_stream_conf`：定义流式特征怎么算。
- `feature_lookup_conf`：定义外部 lookup 怎么取。
- `feature_derived_conf`：定义派生特征怎么推导。
- `list_set` / `list_item`：定义黑白名单等外部辅助判断数据。
- `entity_type_def`：定义按谁聚合，例如用户、设备、IP。

一句话：这一层解决“规则执行时有哪些上下文变量”。

### 4.3 规则层

对应表：

- `rule_def`

作用：

- 定义单条规则表达式、优先级、动作、风险分、命中原因模板、启停状态。

一句话：这一层解决“单条规则怎么判断”。

### 4.4 策略层

对应表：

- `policy_def`
- `policy_rule_ref`
- `policy_score_band`

作用：

- `policy_def`：定义策略模式和默认动作。
- `policy_rule_ref`：定义策略由哪些规则组成、顺序如何。
- `policy_score_band`：为 `SCORE_CARD` 提供分段映射。

一句话：这一层解决“多条规则最后如何收敛成一个决策”。

### 4.5 发布层

对应表：

- `scene_release`

作用：

- 保存运行时快照 JSON、版本号、checksum、发布状态、生效时间、回滚来源等。

一句话：这是设计态和运行态之间最重要的桥。

### 4.6 日志与治理层

对应表：

- `simulation_case`
- `simulation_report`
- `decision_log`
- `rule_hit_log`
- `ingest_error_log`
- `risk_audit_log`
- `replay_job`
- `risk_metric_snapshot`

作用：

- `simulation_*`：验证版本发布结果。
- `decision_log` / `rule_hit_log`：追溯线上决策链路。
- `ingest_error_log`：记录接入异常。
- `risk_audit_log`：记录配置和发布操作。
- `replay_job`：支持回放。
- `risk_metric_snapshot`：保存指标快照。

一句话：这一层解决“发布后如何追溯、治理和优化”。

## 5. 当前阶段最该关心哪些表

如果你当前的开发重心是 `pulsix-kernel + pulsix-engine`，建议把重点收敛到下面这些表：

- `scene_release`
- `scene_def`
- `event_schema`
- `event_field_def`
- `feature_def`
- `feature_stream_conf`
- `feature_lookup_conf`
- `feature_derived_conf`
- `rule_def`
- `policy_def`
- `policy_rule_ref`

原因很简单：

- `kernel` 需要这些表对应的语义来定义 `SceneSnapshot` 和执行链路。
- `engine` 最终只直接消费 `scene_release`，但理解快照来自哪里仍需要知道主链路上游结构。
- 其他表当前可以后置，不影响你先验证执行内核。

## 7. 最小逻辑外键关系

当前 SQL 虽然可能没有全部做成物理外键，但脑子里要把它们理解成下面这些逻辑关系：

- `event_schema.scene_code -> scene_def.scene_code`
- `event_field_def.(scene_code, event_code) -> event_schema.(scene_code, event_code)`
- `feature_def.scene_code -> scene_def.scene_code`
- `feature_*_conf.(scene_code, feature_code) -> feature_def.(scene_code, feature_code)`
- `list_set.scene_code -> scene_def.scene_code`
- `list_item.(scene_code, list_code) -> list_set.(scene_code, list_code)`
- `rule_def.scene_code -> scene_def.scene_code`
- `policy_def.scene_code -> scene_def.scene_code`
- `policy_rule_ref.(scene_code, policy_code) -> policy_def.(scene_code, policy_code)`
- `policy_rule_ref.(scene_code, rule_code) -> rule_def.(scene_code, rule_code)`
- `scene_release.scene_code -> scene_def.scene_code`
- `decision_log.(scene_code, version_no) -> scene_release.(scene_code, version_no)`
- `rule_hit_log.decision_id -> decision_log.id`

---

## 8. 只记住这些就够了

如果你只想快速抓重点，记住下面这些即可：

- `scene_def` 是根。
- `event_*` 定义输入。
- `feature_*` 和 `list_*` 定义上下文变量。
- `rule_def` 定义单条判断。
- `policy_*` 定义多条规则如何收敛。
- `scene_release` 是发布编译产物。
- `decision_log` / `rule_hit_log` 是结果追溯，不参与执行。

一句话总结：

> `pulsix-risk.sql` 的真正主线，不是“很多表”，而是“场景驱动下，从事件定义到快照发布，再到结果追溯的一条完整链路”。
