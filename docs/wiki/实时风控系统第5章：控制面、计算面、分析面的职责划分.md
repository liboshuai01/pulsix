## 5.1 这一章解决什么问题

上一章我们已经把系统总体架构搭出来了，知道这套实时风控平台大体上分成：

- 控制面
- 计算面
- 分析面
- 基础设施层

但如果只停留在这个层面，系统还是容易在真正落地时变得混乱。因为你很快会遇到几个实际问题：

- 控制平台到底应该做到什么程度？
- Flink 引擎到底负责什么，不负责什么？
- 日志、报表、监控到底是平台的哪一部分？
- 为什么成熟系统一定强调“编辑态”和“运行态”的分离？
- 为什么一定要有发布、版本、回滚、审计，而不能“改完表直接生效”？

如果这些职责边界不清楚，后面几乎所有设计都会出问题：

- 控制平台可能变成一个“半个执行引擎”
- Flink 可能变成一个“边执行边拼配置的脚本容器”
- 分析层可能反向污染热路径
- 配置改动可能造成线上逻辑不一致
- 日志和指标可能做出来了，但无法解释和回溯

所以这一章的核心目标，是把系统进一步拆清楚：

> **控制面负责定义和编译决策逻辑，计算面负责实时执行决策逻辑，分析面负责观测、追溯与优化决策逻辑。**

这一章看似是在讲“职责划分”，实际上是在给你建立整个系统的**运行秩序**。

---

## 5.2 先建立一个核心认知：这是三个不同世界

你可以把实时风控平台理解成三个并行但协作的世界：

1. **设计世界**：人编辑规则、特征、策略、名单
2. **执行世界**：机器实时处理事件并给出决策
3. **观测世界**：人和系统一起回看结果、分析效果、监控稳定性

分别对应：

- **控制面**：设计世界
- **计算面**：执行世界
- **分析面**：观测世界

这三个世界最大的区别，不是“模块名称不同”，而是它们面对的问题完全不同。

### 控制面面对的问题

- 怎么定义业务逻辑
- 怎么配置规则、特征、名单、策略
- 怎么保证配置正确
- 怎么形成可发布的版本
- 怎么回滚和审计

### 计算面面对的问题

- 怎么低延迟处理实时事件
- 怎么维护状态
- 怎么在不重启作业的情况下切换新版本
- 怎么保证容错和恢复
- 怎么在高吞吐下稳定执行

### 分析面面对的问题

- 为什么命中了这条规则
- 哪个版本效果更好
- 哪条规则误杀多
- 当前延迟是否抖动
- Kafka lag / Checkpoint / Backpressure 是否异常

所以，从架构哲学上你一定要记住：

> **控制面不是运行时引擎，计算面不是配置后台，分析面不是热路径的一部分。**

这是后面很多设计成立的前提。

---

## 5.3 控制面到底负责什么

控制面可以理解为整个平台的“设计态中枢 + 发布编译中心”。

它不是简单的后台管理系统，而是决定：

- 平台允许运行哪些场景
- 每个场景的事件模型是什么
- 哪些特征可以被使用
- 规则和策略如何组织
- 哪个版本什么时候生效
- 仿真如何执行
- 出问题如何回滚

### 5.3.1 控制面的第一职责：元数据管理

控制面首先负责管理整套平台的设计态对象。

核心包括：

- 场景 Scene
- 事件模型 Event Schema
- 特征定义 Feature Definition
- 名单定义 List Definition
- 规则定义 Rule Definition
- 策略定义 Policy Definition

这部分的本质，不是“做页面”，而是为平台建立一套**结构化、可校验、可编译**的配置体系。

为什么一定要结构化？
因为后面你要做：

- 变量依赖分析
- 规则表达式校验
- 发布编译
- 仿真执行
- 版本管理

如果一开始就把所有内容都塞进大 JSON，后面这些能力会非常难做。

---

### 5.3.2 控制面的第二职责：配置校验

控制面不能只是“保存用户输入”，它必须承担编译前的校验职责。

至少要做这些检查：

- 规则表达式是否合法
- 派生特征依赖的变量是否存在
- 规则依赖的 feature 是否已定义
- 是否出现循环依赖
- 名单引用是否有效
- 策略中引用的 rule 是否存在
- 同名 code 是否冲突
- Groovy 是否通过语法检查
- 是否使用了危险 API

这是一个非常关键的原则：

> **错误应该尽量在发布前暴露，而不是等到 Flink 线上执行时才报错。**

也就是说，控制平台本质上是“编译期校验器”。

---

### 5.3.3 控制面的第三职责：依赖分析

在实时风控平台里，规则和特征不是孤立的。

例如：

- 规则 `R002` 依赖 `trade_burst_flag`
- `trade_burst_flag` 又依赖 `user_trade_cnt_5m` 和 `amount`
- `user_trade_cnt_5m` 是一个流式特征

那么平台就应该在发布前知道：

- 当前规则最终依赖哪些变量
- 哪些是基础字段
- 哪些是 stream feature
- 哪些是 lookup feature
- 哪些是 derived feature

这件事非常重要，因为它直接决定后面：

- Snapshot 怎么生成
- Flink 运行时要准备哪些上下文
- 仿真页要怎么组织输入和输出

所以控制面不只是“录配置”，而是要形成**依赖图**和**执行提示**。

---

### 5.3.4 控制面的第四职责：发布编译

这是控制面最核心、最容易被低估的职责。

发布不是把某几张表的状态标成“启用”，
而是要把设计态配置编译成一份完整的运行态快照。

也就是说：

- 编辑态：多张关系表
- 运行态：一份自包含的 Snapshot

控制平台发布时要完成：

1. 加载场景下所有相关配置
2. 做校验和依赖分析
3. 按运行时需要展开结构
4. 生成版本号、checksum、生效时间
5. 产出一份完整快照
6. 写入发布记录表
7. 写入发布记录表，并由 MySQL CDC 同步给 Flink

你一定要记住这一点：

> **控制平台不是给 Flink 提供“配置库”，而是给 Flink 提供“运行计划”。**

---

### 5.3.5 控制面的第五职责：版本管理与回滚

成熟系统一定有版本，不只是为了好看，而是为了保证运行态可控。

控制平台至少要管理：

- 草稿版本
- 已发布版本
- 当前生效版本
- 历史版本
- 回滚来源关系

为什么要这样？

因为线上出问题时，你真正需要的不是“去改表”，而是：

- 快速定位当前运行的是哪个版本
- 把系统切回上一个稳定版本
- 保证切换后的逻辑可追溯

如果没有版本，线上配置变化就是一团糊涂账。

---

### 5.3.6 控制面的第六职责：仿真测试

仿真测试不是锦上添花，而是控制面的核心可信能力。

仿真测试页应该支持：

- 输入一条事件 JSON
- 指定某个场景和某个版本
- 模拟执行规则和策略
- 展示特征值、命中规则、最终动作、耗时

为什么这件事必须属于控制面？
因为它服务的是：

- 发布前验证
- 策略调试
- 回归测试
- 问题复盘

而不是实时线上处理。

同时还有一个很重要的工程原则：

> **仿真必须尽量复用线上规则执行核心。**

不然你会出现：

- 仿真命中了
- 线上没命中

这种情况对平台可信度伤害非常大。

---

### 5.3.7 控制面的第七职责：权限与审计

真实风控系统里，配置改动本身就是高风险操作。

所以控制面还必须具备：

- 用户管理
- 角色管理
- 菜单权限
- 发布权限
- 操作审计

审计至少要记录：

- 谁改了规则
- 改前是什么，改后是什么
- 谁发布了版本
- 回滚了哪个版本
- 发布时间和说明是什么

对你的项目来说，不一定要做企业级审批流，
但至少要做基础 RBAC + 审计日志。

---

## 5.4 控制面不应该负责什么

讲完“该做什么”，我们也要讲清楚“不该做什么”。

### 5.4.1 不应该负责逐条实时决策

控制平台不是在线决策执行器。

它可以做仿真，但不应该承担生产高并发逐条事件处理。

### 5.4.2 不应该直接维护流式中间状态

例如：

- 用户5分钟交易次数
- 设备1小时关联账号数

这些是 Flink 的职责，不是 Spring Boot 后台的职责。

### 5.4.3 不应该让 Flink 每次执行时直接查几十张设计态表

这会让：

- 热路径延迟高
- 逻辑不一致
- 配置变更不可控

### 5.4.4 不应该承担大规模在线名单主查询压力

控制平台可以管理名单，但运行时大名单查询应该交给 Redis 等在线存储。

### 5.4.5 不应该把“保存”当成“发布”

这是一个非常常见但很危险的设计错误。

正确做法应该是：

- 保存：只是编辑草稿
- 发布：才会生成运行时版本并影响线上

---

## 5.5 计算面到底负责什么

计算面就是运行态实时决策执行引擎。对你这个项目来说，它的核心就是 Flink Job。

计算面面对的不是页面，不是用户操作，而是：

- 实时事件流
- 配置广播流
- 外部 lookup 数据
- 流式状态
- 输出决策流

它的核心目标非常明确：

> **在高并发、低延迟条件下，稳定执行当前生效版本的决策逻辑。**

---

### 5.5.1 计算面的第一职责：消费实时事件

例如：

- login 事件
- register 事件
- trade 事件

这些事件通常来自 Kafka，也可以有其他入口，但最终在实时引擎里都应被统一成标准事件对象。

计算面首先做的是：

- 读取事件
- 解析标准字段
- 标准化时间
- 校验最基本字段
- 找到 sceneCode / eventCode
- 进行字段映射、格式转换、公共字段补齐
- 将不合法事件送入错误流 / DLQ

这是整条热路径的起点。

---

### 5.5.2 计算面的第二职责：消费配置快照并热切换

控制面发布版本后，Flink 端要能接收到新快照。

计算面要完成的事包括：

- 接收配置广播流
- 更新 Broadcast State
- 在本地编译表达式 / Groovy
- 构建运行时缓存
- 切换当前场景版本
- 在必要时回退旧版本

这里有个非常重要的边界：

> **Flink 端应该消费的是 Snapshot，而不是控制平台的多表关系数据。**

也就是说，计算面负责“加载运行计划”，而不是“自己拼装运行计划”。

---

### 5.5.3 计算面的第三职责：维护流式状态

这是 Flink 在整个系统中的核心价值之一。

计算面要维护：

- user 维度 count / sum / max / latest
- device 维度关联统计
- ip 维度频次统计
- distinct\_count 相关状态
- 中间聚合桶
- timer 清理信息

这些内容通常放在：

- Keyed State
- MapState
- ValueState
- Timer

这类能力是控制面和普通规则引擎都无法替代的。

---

### 5.5.4 计算面的第四职责：构建实时上下文

规则执行前，计算面必须把上下文准备完整。

一个完整上下文通常包括：

- 基础事件字段
- 实时流式特征
- Redis 查询得到的名单/画像
- 派生特征结果
- 运行时版本信息

这一步可以理解为：

> **把原始事件补齐成一个可执行规则的 EvalContext。**

没有这个上下文，规则根本无法稳定执行。

---

### 5.5.5 计算面的第五职责：执行规则与策略

上下文准备好之后，计算面要按照当前快照中的执行计划：

1. 算派生特征
2. 执行规则
3. 汇总命中结果
4. 按策略收敛最终动作

例如：

- FIRST\_HIT：按优先级命中即返回
- SCORE\_CARD：累计分数决定最终动作

这一层才是我们通常口头说的“规则引擎执行”，
但你现在应该清楚，它只是计算面的一个步骤，而不是全部。

---

### 5.5.6 计算面的第六职责：输出决策和运行日志

计算面最终要输出：

- 决策结果
- 规则命中明细
- 版本号
- 特征快照
- 耗时指标
- 异常信息

这些结果一般不会在热路径同步写复杂数据库，而是：

- 发往 Kafka decision topic / log topic
- 后续异步落 MySQL / Doris / ClickHouse

这样做的原因是：

> **热路径只做实时执行，不让复杂日志存储反向拖慢决策。**

---

### 5.5.7 计算面的第七职责：容错与恢复

Flink 不是只负责“跑起来”，它还要负责：

- Checkpoint
- 恢复后重建状态
- Broadcast State 恢复
- 本地编译缓存重建
- 版本切换容错

这也是为什么你不能把运行时设计成一堆不可恢复、不可序列化的随意对象。

---

## 5.6 计算面不应该负责什么

### 5.6.1 不应该负责配置编辑

Flink 不负责：

- 规则编辑
- 页面表单保存
- 权限控制
- 发布审批

### 5.6.2 不应该负责设计态语义校验

例如：

- 表达式依赖分析
- 循环依赖检查
- code 唯一性校验

这些都应该在控制面解决。

### 5.6.3 不应该实时查几十张管理表拼装逻辑

这一点一定要坚守。

如果 Flink 直接去查：

- rule\_def
- feature\_def
- policy\_rule\_ref
- list\_def
- scene\_config

会导致：

- 运行时耦合过重
- 一致性差
- 发布边界消失
- 性能不可控

### 5.6.4 不应该承担长期分析报表

Flink 可以做实时指标输出，但不是你的主分析系统。

### 5.6.5 不应该开放成“任意脚本平台”

尤其不要让运行时脚本控制：

- keyBy
- 状态结构
- timer 模型
- 任意网络/文件调用

计算面必须是**受控的执行引擎**，而不是无限制脚本容器。

---

## 5.7 分析面到底负责什么

分析面的作用，简单说就是：

> **让系统结果可看、问题可查、效果可评估。**

很多人做项目时容易觉得分析面是“次要功能”，但对风控系统来说，分析面实际上是业务可运营性的关键。

如果没有分析面，业务就无法回答：

- 为什么这条交易被拒绝？
- 哪条规则今天命中率异常高？
- 新版本上线后拒绝率为什么上升？
- 哪个场景 P95 延迟明显变差？
- 哪条规则误杀了大量正常用户？

所以分析面一定不能省略。

---

### 5.7.1 分析面的第一职责：决策日志查询

决策日志是分析面的基础。

至少应该支持按这些条件查询：

- traceId
- eventId
- sceneCode
- userId
- 时间范围
- 决策结果
- 版本号

并能查看：

- 输入事件
- 命中规则列表
- 命中原因
- 特征快照
- 最终动作
- 决策耗时

这一点非常重要，因为这是问题追查和业务复盘的第一入口。

---

### 5.7.2 分析面的第二职责：规则与策略效果分析

除了逐条查日志，分析面还应该支持统计视角：

- 某规则今日命中次数
- 某规则拒绝贡献占比
- 某场景拒绝率趋势
- 某版本上线前后命中变化
- 某规则是否过于敏感导致误杀率上升

这让平台从“执行系统”升级为“可运营系统”。

---

### 5.7.3 分析面的第三职责：可观测性与监控

实时风控平台的稳定性也需要被观测。

分析面通常要展示：

- QPS / TPS
- 决策耗时 P50/P95/P99
- Kafka lag
- Flink Checkpoint 成功率
- Backpressure
- Redis 调用耗时
- 脚本执行异常数
- 配置切换次数与成功率

这些指标不仅服务运维，也服务架构优化。

---

### 5.7.4 分析面的第四职责：版本对比与问题定位

当你有多个发布版本后，分析面就要承担另一类重要职责：

- 当前运行版本是什么
- 某次异常是从哪个版本开始出现的
- 回滚之后指标是否恢复
- 某条规则是哪个版本引入的

这就是为什么分析面必须和版本体系打通。

---

### 5.7.5 分析面的第五职责：服务策略优化

成熟系统里，分析面最终是服务于策略优化的。

例如：

- 哪些规则命中很多但实际价值很低
- 哪些名单已经老化
- 哪些特征计算开销很大但收益不高
- 哪些场景应该拆分或合并

换句话说，分析面不仅“看结果”，更要帮助你“改策略”。

---

## 5.8 分析面不应该负责什么

### 5.8.1 不应该反向参与热路径决策

日志查询系统、报表系统不能反过来卡住实时决策。

### 5.8.2 不应该成为配置主存储

报表库、日志库不是规则配置中心。

### 5.8.3 不应该承担流式状态计算

分析层可以消费结果，不负责维护在线计算状态。

### 5.8.4 不应该与控制面、计算面强耦合部署

分析面最好通过日志流和指标流与系统其他部分解耦，而不是把查询和统计逻辑硬塞进热路径。

---

## 5.9 为什么“编辑态配置”和“运行态执行”必须分离

这一节非常关键，因为它其实决定了你后面为什么必须设计 Snapshot。

很多新手最容易犯的一个架构错误就是：

- 后台配置直接改数据库
- Flink 或服务实时去查这些表
- 觉得这样最“简单”

但真实系统里，这种方式问题非常多。

---

### 5.9.1 原因一：编辑过程天然是不完整的

用户在后台改配置时，往往是分步骤完成的：

- 先改 rule
- 再改 policy
- 再改 feature
- 最后才点发布

如果运行时直接读设计态表，就可能读到半成品：

- 规则改了，但策略还没改
- feature code 改了，但规则还引用旧 code
- 名单绑定关系还没完成

这会导致线上逻辑短时间内出现不一致。

---

### 5.9.2 原因二：设计态强调灵活，运行态强调稳定

设计态允许：

- 草稿
- 编辑中
- 暂存
- 校验失败
- 试验配置

运行态则要求：

- 完整
- 可恢复
- 可追溯
- 版本冻结
- 可快速切换

这两者本质上就是不同的数据世界。

---

### 5.9.3 原因三：运行态必须可重复执行

如果你想回答：

- 3 月 7 日 10:05 那条交易为什么被拒绝？

你必须知道：

- 当时执行的是哪个版本
- 当时规则和策略长什么样
- 当时依赖哪些特征和名单

如果运行时直接依赖一堆持续变化的设计态表，就很难做到严格追溯。

---

### 5.9.4 原因四：分离后才能支持版本、回滚和仿真一致性

只有当运行态是一份独立快照时，下面这些能力才自然成立：

- 发布新版本
- 快速回滚到旧版本
- 使用指定版本做仿真
- 比较两个版本效果差异

所以：

> **编辑态和运行态分离，不是“设计得更复杂”，而是让系统真正可控。**

---

## 5.10 设计态配置与运行态快照的关系

可以用一句话概括两者的关系：

> **设计态是“怎么定义逻辑”，运行态是“怎么执行逻辑”。**

设计态一般对应多张表：

- scene
- feature\_def
- rule\_def
- policy\_def
- list\_ref

运行态则对应：

- 一个版本号
- 一份快照 JSON / POJO
- 一组编译后的运行时对象

你可以把它理解成：

- 设计态 = 源代码 / 工程文件
- 运行态 = 编译产物 / 发布包

控制平台做的就是“编译发布”，Flink 做的是“加载执行”。

---

## 5.11 为什么成熟系统一定有发布、回滚、版本

这不是为了像“大厂平台”，而是实时风控系统天然需要这些能力。

### 5.11.1 版本解决“可定位”问题

你必须知道线上当前到底跑的是哪一版逻辑。

### 5.11.2 发布解决“生效边界”问题

不是保存就生效，而是经过校验和编译之后，明确在某个时刻生效。

### 5.11.3 回滚解决“风险兜底”问题

风控最怕误杀，一旦上线后出现大量误拦截，必须能迅速切回稳定版本。

### 5.11.4 版本还能解决“分析对比”问题

例如：

- v11 的拒绝率是多少
- v12 上线后命中规则发生了什么变化

如果没有版本体系，这些问题都很难回答。

---

## 5.12 三个面的协作关系到底是什么

可以把三者协作总结成一条链：

1. **控制面定义并发布快照**
2. **计算面加载快照并执行决策**
3. **分析面消费结果并提供查询与监控**
4. **分析结论再反哺控制面优化规则和策略**

这实际上形成了一个闭环：

> 设计 → 发布 → 执行 → 观测 → 优化 → 再发布

这就是成熟实时风控平台真正的运行方式。

---

## 5.13 一个非常实用的职责分工表

下面这张表你后面做系统设计时可以经常拿来对照。

| 能力               | 控制面     | 计算面      | 分析面    |
| :--------------- | :------ | :------- | :----- |
| 场景/事件/特征/规则/策略配置 | 负责      | 不负责      | 不负责    |
| 表达式/Groovy校验     | 负责      | 只做兜底编译校验 | 不负责    |
| 发布快照生成           | 负责      | 不负责      | 不负责    |
| 版本管理 / 回滚        | 负责      | 执行切换     | 展示结果   |
| 实时事件处理           | 不负责     | 负责       | 不负责    |
| 流式状态维护           | 不负责     | 负责       | 不负责    |
| Redis lookup     | 配置定义    | 执行查询     | 可做统计   |
| 规则 / 策略执行        | 仿真可复用核心 | 负责实时执行   | 展示命中结果 |
| 决策日志查询           | 提供查询接口  | 产生日志     | 负责展示分析 |
| 指标监控 / 告警        | 管理入口    | 输出指标     | 汇总展示   |
| 误杀分析 / 效果评估      | 配置优化来源  | 提供原始结果   | 主要负责   |

这张表本质上就是整个系统的职责地图。

## 5.14 典型错误架构示例

为了让边界感更清楚，我再给你几个常见反例。

### 反例 1：Flink 直接查后台规则表

问题：

- 热路径依赖管理库
- 配置不完整时容易出错
- 无法做稳定版本切换

### 反例 2：保存配置即线上生效

问题：

- 没有发布边界
- 审计与回滚困难
- 编辑中间态污染线上

### 反例 3：日志同步写 MySQL

问题：

- 决策延迟被数据库拖慢
- 高峰期容易放大抖动

### 反例 4：允许 Groovy 控制任意状态和外部调用

问题：

- 安全风险高
- 性能不可控
- Flink 恢复和调试复杂度高

### 反例 5：分析报表直接跑在线事务库

问题：

- 查询和管理、热路径耦合
- 高峰期容易相互影响

## 5.15 从代码组织上，三个面应该怎么映射

为了后面落地时更自然，你现在就可以先建立一个代码层映射。

### 控制面模块

例如：

- `pulsix-server`
- `pulsix-kernel`（仿真可依赖）
- `compiler-core`

负责：

- Controller / Service / Domain / Compiler / Simulation

### 计算面模块

例如：

- `pulsix-engine`
- `feature-core`
- `runtime-core`

负责：

- Event Source
- Broadcast State
- Feature Executor
- Rule Invoker
- Policy Evaluator
- Sink

### 分析面模块

可以先并入控制面，后续扩展：

- log query
- metrics dashboard
- audit display

对个人项目来说，这样做最务实：

- 逻辑分层清晰
- 物理服务不必过多

## 5.17 本章小结

这一章最核心的内容，可以浓缩成下面几条。

### 1）控制面、计算面、分析面，本质上是三个不同世界

- 控制面负责定义和编译
- 计算面负责实时执行
- 分析面负责观测和优化

### 2）控制面不是普通后台，而是发布编译中心

它负责：

- 元数据管理
- 校验
- 依赖分析
- Snapshot 编译
- 发布/回滚
- 仿真
- 审计

### 3）计算面不是规则页面的延伸，而是运行态引擎

它负责：

- 事件消费
- 状态维护
- 上下文构建
- 规则/策略执行
- 结果输出
- 恢复与容错

### 4）分析面不是附属功能，而是平台可运营性的关键

它负责：

- 决策日志
- 效果分析
- 指标监控
- 版本对比
- 问题定位

### 5）设计态和运行态必须分离

只有分离，才能真正支持：

- 稳定发布
- 版本冻结
- 快速回滚
- 仿真一致
- 可追溯

### 6）成熟实时风控平台一定是闭环系统

不是“配规则 + 执行”，而是：

**设计 → 发布 → 执行 → 观测 → 优化 → 再发布**
