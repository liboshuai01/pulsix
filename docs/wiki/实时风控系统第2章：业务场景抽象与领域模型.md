## 2.1 这一章解决什么问题

在开始设计实时风控系统之前，必须先把业务语言抽象成平台语言。否则很容易出现两个问题：

- 业务场景讲得很热闹，但系统设计出来只是零散页面和零散规则
- 技术实现很多，但核心对象边界不清楚，后面越做越乱

这一章要解决的核心问题是：

1. 登录风控、注册反作弊、交易风控这些场景，本质上到底有什么共性
2. 实时风控系统里，最核心的领域对象到底有哪些
3. 这些对象之间是什么关系
4. 如何把业务描述翻译成系统可以执行的结构化模型
5. 你的开源项目应该围绕什么样的领域模型来搭建

如果说第 1 章解决的是“你在做什么系统”，那么这一章解决的是：

> 你要用什么语言、什么对象、什么边界去描述这个系统。

---

## 2.2 为什么必须做“业务抽象”

很多技术项目失败，不是因为代码不会写，而是因为一开始没有把业务问题抽象成稳定的系统模型。

例如业务同学可能会说：

- 最近 10 分钟内这个设备注册了很多账号
- 这个用户异地登录很可疑
- 这笔交易金额很大而且行为异常
- 这个 IP 像代理机房
- 这个账号和很多风险设备有关联

如果你直接按照这些自然语言堆实现，系统会变成：

- 每来一个场景就加一套表
- 每来一个需求就写一段 if-else
- 每来一个策略就新增一个特殊接口

最后系统就会越来越不可维护。

真正成熟的做法是，把这些业务表达统一抽象成一组稳定的领域概念。例如：

- 行为 -> Event
- 行为主体 -> Entity
- 风险观测值 -> Feature
- 高风险集合 -> List
- 判断逻辑 -> Rule
- 多条规则组合 -> Policy
- 最终输出 -> Decision

这样你不管是做登录风控、注册反作弊还是交易决策，本质上都可以落在同一套系统骨架上。

---

## 2.3 三个典型业务场景的共性

你准备做的项目，最适合聚焦以下三类场景：

1. 登录风控
2. 注册反作弊
3. 交易/提现风控

表面看起来它们差异很大，但从系统视角看，它们其实有非常强的共性。

### 2.3.1 登录风控

典型问题：

- 短时间内多次登录失败
- 同账号短时间跨多个设备登录
- 同设备登录多个账号
- 异地/IP 风险异常
- 账号疑似被撞库或盗号

典型动作：

- 允许登录
- 二次验证
- 限制登录
- 拒绝登录
- 告警

### 2.3.2 注册反作弊

典型问题：

- 同设备批量注册
- 同 IP 高频注册
- 同手机号段异常集中注册
- 黑产设备/代理 IP 命中
- 批量薅新人权益

典型动作：

- 正常注册
- 要求验证码
- 延迟通过
- 拒绝注册
- 打可疑标签

### 2.3.3 交易/提现风控

典型问题：

- 大额交易
- 短时高频交易
- 跨设备/异地交易
- 用户行为和历史画像不一致
- 命中高风险名单

典型动作：

- 放行
- 审核
- 拒绝
- 冻结
- 限额

---

## 2.4 三类场景的统一抽象

虽然三种场景具体规则不同，但它们可以抽象成同一个决策公式：

> 当某个事件发生时，系统基于当前事件、实时特征、历史画像、名单和策略，产出一个处置动作。

可以统一表示为：

```latex
Decision = Policy(Event + StreamFeatures + LookupFeatures + Context)
```

再展开一点：

```latex
原始事件 -> 上下文补充 -> 特征计算 -> 规则判断 -> 策略收敛 -> 输出动作
```

这个抽象非常重要，因为它意味着：

- 登录风控不是特殊系统
- 注册风控也不是特殊系统
- 交易风控也不是特殊系统

它们只是：

- 事件模型不同
- 关注实体不同
- 特征定义不同
- 规则内容不同
- 动作类型不同

但底层决策框架是一致的。

这就是平台化的基础。

---

## 2.5 实时风控系统中的核心领域对象

下面进入本章最关键的部分：领域模型。

我建议你把实时风控系统的核心对象分成 8 个一级概念：

1. Scene：场景
2. Event：事件
3. Entity：实体
4. Feature：特征
5. List：名单
6. Rule：规则
7. Policy：策略
8. Decision：决策

这 8 个对象就是你整个系统的骨架。

后续不管是数据库表、控制平台页面、Flink 快照、引擎执行链路，都会围绕它们展开。

---

## 2.6 Scene：场景

### 2.6.1 什么是 Scene

Scene 可以理解为：

> 某一类风控业务问题的决策容器。

例如：

- LOGIN\_RISK：登录风控
- REGISTER\_ANTI\_FRAUD：注册反作弊
- TRADE\_RISK：交易风控

Scene 不是一条规则，也不是一个事件，而是一组决策逻辑的组织边界。

### 2.6.2 Scene 的作用

它主要解决三个问题：

1. 把不同业务场景隔离开
2. 让事件模型、特征、规则、策略归属明确
3. 让发布、回滚、日志和监控都能按场景组织

### 2.6.3 为什么 Scene 很重要

如果没有 Scene，你后面会遇到很多混乱：

- 登录和交易规则混在一起
- 不同类型事件的字段差异很大，难以统一
- 策略版本不好管理
- 决策日志难以定位业务语义

所以我建议你整个项目都围绕 Scene 做一级组织。

例如：

- 场景下面有事件模型
- 场景下面有特征定义
- 场景下面有规则和策略
- 场景发布时生成该场景的运行时快照

---

## 2.7 Event：事件

### 2.7.1 什么是 Event

Event 是实时风控系统的输入起点。

它代表：

> 某个时间点上发生的一次业务行为或状态变化。

例如：

- 用户登录
- 用户注册
- 用户发起支付
- 用户提现
- 用户修改密码
- 设备绑定账号

### 2.7.2 Event 的核心特征

一个事件至少应包含：

- 事件 ID
- 场景 code
- 事件编码
- 事件时间
- 主体标识
- 设备/IP 等上下文字段
- 业务字段

例如一条交易事件：

```json
{
  "eventId": "E10001",
  "sceneCode": "TRADE_RISK",
  "eventCode": "TRADE_EVENT",
  "eventTime": "2026-03-07T10:00:00",
  "userId": "U1001",
  "deviceId": "D9001",
  "ip": "1.2.3.4",
  "amount": 6800,
  "currency": "CNY",
  "result": "SUCCESS"
}
```

### 2.7.3 为什么 Event 不是普通 DTO

在实时风控里，Event 不只是传输对象，它是：

- Flink 的输入流元素
- 特征计算的原始素材
- 规则执行的上下文源头
- 决策日志的主线索

所以事件模型必须规范化，不能每个场景随便传一份 JSON。

### 2.7.4 事件模型设计建议

建议事件模型采用“公共字段 + 扩展字段”的方式：

- 公共字段：eventId、sceneCode、eventCode、eventTime、traceId、userId、deviceId、ip
- 扩展字段：amount、channel、merchantId、loginResult 等

这样可以既保证统一性，又支持不同业务场景扩展。

### 2.7.5 为什么还需要“事件接入与标准化预处理”

但这里要特别提醒你一件事：

> **业务平台上传的原始事件，并不一定天然就是风控平台可直接消费的 `RiskEvent`。**

真实项目里，上游事件经常会出现下面这些问题：

- 字段名不一致，例如 `uid` / `user_id` / `userId` 混用
- 时间格式不一致，例如时间戳、字符串时间、时区表达混杂
- 数值字段被序列化成字符串
- `traceId`、`sceneCode`、`eventCode` 等公共字段缺失
- 一些业务字段需要映射到统一 `ext` 或标准字段

所以，除了定义事件模型本身，你还应该具备一层轻量的“事件接入与标准化预处理”能力，至少负责：

- 原始字段到标准字段的映射
- 基础类型转换
- 公共字段补齐与默认值填充
- 必填字段与场景合法性校验
- 不合法事件打入死信或错误流

它不是一个重型离线 ETL 平台，但它必须是实时风控平台的一期输入层能力。因为如果没有这层，事件模型往往只停留在设计态，无法稳定落到真正运行链路里。

## 2.8 Entity：实体

### 2.8.1 什么是 Entity

Entity 是风控系统观察和聚合的核心对象。

可以理解为：

> 事件背后的行为主体或关联主体。

常见实体包括：

- user：用户
- device：设备
- ip：IP
- account：账号
- card：银行卡
- merchant：商户
- mobile：手机号

### 2.8.2 为什么 Entity 很重要

因为大多数特征都不是针对“事件本身”统计，而是针对某个实体在一段时间内的行为统计。

例如：

- 用户 5 分钟交易次数 -> 以 user 为实体
- 设备 1 小时关联账号数 -> 以 device 为实体
- IP 10 分钟注册次数 -> 以 ip 为实体

也就是说，Feature 本质上往往是：

> 某个实体在某个观察窗口内的风险观测值。

### 2.8.3 一个事件可以关联多个实体

例如一笔交易事件：

- 主实体：userId
- 次实体：deviceId
- 网络实体：ip
- 金融实体：cardNo
- 商户实体：merchantId

所以在领域建模时，不要把 Entity 理解为只有一个 user。

成熟系统里，一个事件往往是一个多实体上下文。

这件事会直接影响你后面 Flink 特征设计：

- 特征的 key 可能是 userId
- 也可能是 deviceId
- 也可能是 ip

---

## 2.9 Feature：特征

### 2.9.1 什么是 Feature

Feature 是风控系统里最核心的对象之一。

可以理解为：

> 为了描述风险而对事件、实体、历史行为、画像等信息做出的结构化观测值。

例如：

- user\_trade\_cnt\_5m：用户 5 分钟交易次数
- device\_bind\_user\_cnt\_1h：设备 1 小时关联用户数
- ip\_register\_cnt\_10m：IP 10 分钟注册次数
- device\_in\_blacklist：设备是否命中黑名单
- user\_risk\_level：用户风险等级

### 2.9.2 为什么 Feature 是平台核心

因为规则几乎都不会直接基于原始事件字段判断，而是基于特征判断。

例如下面这句话：

> 最近 30 分钟内，用户在不同设备上连续发起大额交易

如果不抽象特征，你只能写一堆业务代码。 但如果抽象成特征，就可以转化为：

- `user_trade_cnt_30m`
- `user_distinct_device_cnt_30m`
- `user_trade_amt_sum_30m`
- `high_amt_flag`

然后规则只需要组合这些特征即可。

### 2.9.3 Feature 的三大类型

后续你整个系统最好使用这三个分类：

#### 1）Stream Feature

由 Flink 基于实时事件和状态计算出来的特征，例如：

- 用户 5 分钟交易次数
- 设备 1 小时关联账号数
- IP 10 分钟注册次数

#### 2）Lookup Feature

通过 Redis / 外部存储查询得到的特征，例如：

- 是否黑名单命中
- 用户风险等级
- 设备画像标签

#### 3）Derived Feature

由基础字段、流式特征、查询特征再计算出来的特征，例如：

- high\_amt\_flag = amount >= 5000
- burst\_trade\_flag = user\_trade\_cnt\_5m >= 3 && amount >= 5000

这个分类非常关键，它会直接决定：

- 哪些东西由 Flink State 负责
- 哪些东西由 Redis 查询负责
- 哪些东西由表达式/Groovy 负责

## 2.10 List：名单

### 2.10.1 什么是 List

List 是实时风控系统里非常常见的一种对象。

它本质上是：

> 一组高风险或高信任对象的集合。

最常见的是：

- 黑名单
- 白名单
- 灰名单
- 可疑设备名单
- 高风险 IP 段名单
- 特殊商户名单

### 2.10.2 名单和普通特征的区别

名单也可以被看作特征的一种来源，但它和普通流式特征不同：

- 名单更像“外部知识”或“业务标签”
- 它通常不需要复杂流式计算
- 更适合放在 Redis、KV、Bitmap 里

例如：

- `device_in_blacklist`
- `user_in_vip_whitelist`

### 2.10.3 名单在风控中的价值

名单系统非常适合处理强规则场景，例如：

- 命中黑名单直接拒绝
- 命中白名单直接放过某些校验
- 命中灰名单提高审查等级

所以在领域上，List 既可以独立管理，也可以在运行时被封装成 lookup feature 使用。

---

## 2.11 Rule：规则

### 2.11.1 什么是 Rule

Rule 是风控系统中的最小判断单元。

可以理解为：

> 在一组上下文变量之上，对风险条件进行判定并给出单条处置结果的逻辑单元。

例如：

- 如果设备命中黑名单，则拒绝
- 如果用户 5 分钟交易次数 >= 3 且 amount >= 5000，则审核
- 如果设备 1 小时关联账号数 >= 4 且用户风险等级为高，则拒绝

### 2.11.2 Rule 的本质

规则本质上包括两部分：

1. 条件
2. 命中后的结果

例如：

```latex
when: device_in_blacklist == true
action: REJECT
```

或者：

```latex
when: user_trade_cnt_5m >= 3 && amount >= 5000
action: REVIEW
```

### 2.11.3 Rule 的边界

一条 Rule 负责“判断”，但不一定负责“最终决策”。

因为：

- 多条规则可能同时命中
- 规则之间可能有优先级
- 有的场景需要评分卡模式

所以 Rule 是最小逻辑单元，不是最终收敛单元。

---

## 2.12 Policy：策略

### 2.12.1 什么是 Policy

Policy 是对多条规则的组织、编排和收敛。

可以理解为：

> 一个场景下，系统如何组织多条规则并产生最终决策的方式。

### 2.12.2 为什么需要 Policy

如果只有 Rule，没有 Policy，就会遇到很多问题：

- 两条规则都命中，以谁为准
- 是命中即返回，还是累计评分
- 是否有默认动作
- 是否要分层执行

这些都属于策略层，而不是规则层。

### 2.12.3 常见策略模式

建议你项目一期至少支持两种：

#### 1）FIRST\_HIT

按优先级执行，命中第一条规则就返回。

适合：

- 黑白名单
- 强拦截规则
- 简单风控链路

#### 2）SCORE\_CARD

多条规则命中后累计分数，再按阈值决定动作。

适合：

- 复杂交易风险
- 综合评分场景
- 多个弱信号组合判断

### 2.12.4 Policy 的价值

Policy 决定了系统是不是“平台化”。

因为平台不是简单存一堆规则，而是要定义：

- 规则如何执行
- 规则如何组合
- 最终如何输出动作

---

## 2.13 Decision：决策

### 2.13.1 什么是 Decision

Decision 是系统最终输出的结果。

可以理解为：

> 系统在某次事件处理过程中，基于当前场景、上下文、特征、规则和策略，给出的最终处置结果。

### 2.13.2 Decision 包含什么

一条完整决策通常不只是 PASS/REJECT，而是应该包含：

- 决策结果 action
- 决策版本 version
- 命中的规则列表
- 规则命中原因
- 特征快照
- 耗时 latency
- 事件 ID / traceId

例如：

```json
{
  "traceId": "T10001",
  "eventId": "E10001",
  "sceneCode": "TRADE_RISK",
  "version": 12,
  "action": "REJECT",
  "hitRules": ["R001", "R003"],
  "reason": "设备命中黑名单且设备关联多个高风险账号",
  "latencyMs": 38
}
```

### 2.13.3 为什么 Decision 很重要

因为系统最终要对业务侧有输出，也要对分析系统有留痕。

所以 Decision 是：

- 下游业务动作的输入
- 决策日志的核心记录
- 效果分析的基础数据

## 2.14 领域对象之间的关系

现在把这些对象串起来。

一个场景下的核心关系可以抽象成：

```latex
Scene
 ├── EventSchema
 ├── Features
 │    ├── StreamFeatures
 │    ├── LookupFeatures
 │    └── DerivedFeatures
 ├── Lists
 ├── Rules
 ├── Policy
 └── Decisions
```

进一步从执行过程来看：

```latex
Event -> Entity -> Feature -> Rule -> Policy -> Decision
```

这里每个箭头的含义是：

- Event 提供原始上下文
- Entity 决定统计和观察维度
- Feature 描述风险观测值
- Rule 在特征上下文上做判断
- Policy 组合多条 Rule 并收敛结果
- Decision 作为最终输出

这条链路就是你整个系统设计的主骨架。

## 2.15 如何把业务语言翻译成平台语言

这一步很重要，因为它关系到你之后控制平台如何设计。

举几个例子。

### 例子 1：登录风控

业务语言：

> 最近 10 分钟内，同一个账号登录失败超过 5 次，则拒绝登录。

平台语言：

- Scene：LOGIN\_RISK
- Event：login
- Entity：user
- Stream Feature：user\_login\_fail\_cnt\_10m
- Rule：user\_login\_fail\_cnt\_10m > 5
- Action：REJECT
- Policy：FIRST\_HIT

### 例子 2：注册反作弊

业务语言：

> 同设备 1 小时注册超过 3 个账号，且 IP 位于代理机房，则要求验证码。

平台语言：

- Scene：REGISTER\_ANTI\_FRAUD
- Event：register
- Entity：device
- Stream Feature：device\_register\_user\_cnt\_1h
- Lookup Feature：ip\_proxy\_flag
- Rule：device\_register\_user\_cnt\_1h > 3 && ip\_proxy\_flag == true
- Action：VERIFY

### 例子 3：交易风控

业务语言：

> 5 分钟内交易超过 3 笔且金额大于 5000，并且设备风险等级为高，则人工审核。

平台语言：

- Scene：TRADE\_RISK
- Event：trade
- Entity：user / device
- Stream Feature：user\_trade\_cnt\_5m
- Derived Feature：high\_amt\_flag
- Lookup Feature：device\_risk\_level
- Rule：user\_trade\_cnt\_5m >= 3 && high\_amt\_flag == true && device\_risk\_level == 'H'
- Action：REVIEW

你会发现，一旦翻译成平台语言，后续很多事情就都能结构化：

- 后台可以配
- 快照可以生成
- Flink 可以执行
- 日志可以解释

## 2.16 你的项目推荐采用的领域分层

为了让系统后续更容易实现，我建议你把领域模型再按执行链路分成四层。

### 2.16.1 输入层

这一层负责定义系统收到什么。

对象包括：

- Scene
- EventSchema
- Event
- Entity

它解决的问题是：

- 事件从哪里来
- 数据长什么样
- 哪些字段必须有
- 主要观察对象是谁

### 2.16.2 上下文层

这一层负责构建规则执行所需的上下文。

对象包括：

- StreamFeature
- LookupFeature
- DerivedFeature
- List
- Profile

它解决的问题是：

- 规则需要的值从哪里来
- 哪些由 Flink 计算
- 哪些由 Redis 查询
- 哪些由表达式/Groovy 派生

### 2.16.3 决策层

这一层负责根据上下文做判断与收敛。

对象包括：

- Rule
- Policy
- Action

它解决的问题是：

- 哪些条件构成风险
- 命中规则后怎么办
- 多条规则如何合并

### 2.16.4 输出层

这一层负责形成可执行、可追溯的结果。

对象包括：

- Decision
- DecisionLog
- HitDetail
- Metrics

它解决的问题是：

- 最终输出给谁
- 如何追踪原因
- 如何做统计分析

这个四层结构会非常适合你后续：

- 设计控制平台页面
- 设计快照结构
- 设计 Flink 执行链路
- 设计数据库表

## 2.18 领域模型对后续系统设计的直接影响

这一章不是纯理论，它会直接决定你后面每一部分怎么设计。

### 影响 1：数据库表怎么建

因为你已经知道核心对象是：

- scene
- event\_schema
- feature\_def
- list\_set / list\_item
- rule\_def
- policy\_def
- scene\_release
- decision\_log

所以你的表结构会自然稳定下来。

### 影响 2：控制平台页面怎么设计

页面不应该按技术组件堆，而应该按领域对象设计：

- 场景中心
- 事件中心
- 特征中心
- 名单中心
- 规则中心
- 策略中心
- 发布中心
- 日志中心

### 影响 3：快照如何组织

因为运行时需要的是：

- 某个 Scene 的完整执行计划

所以快照应该包含：

- scene
- features
- rules
- policy

而不是数据库表的原始结构。

### 影响 4：Flink 执行链路怎么组织

Flink 的本质就是在执行：

```latex
Event -> Feature -> Rule -> Policy -> Decision
```

所以你后面思考 Flink 时，不会迷失在 API 细节中，而是始终围绕这条主链路来实现。

## 2.19 一套推荐的领域对象命名规范

为了让你的开源项目更专业，我建议一开始就统一命名方式。

### 场景命名

- `LOGIN_RISK`
- `REGISTER_ANTI_FRAUD`
- `TRADE_RISK`

### 事件命名

- `login`
- `register`
- `trade`
- `withdraw`

### 特征命名

建议采用：

```latex
{entity}_{metric}_{window}
```

例如：

- `user_login_fail_cnt_10m`
- `device_bind_user_cnt_1h`
- `ip_register_cnt_10m`
- `user_trade_amt_sum_30m`

### 规则命名

- `R_LOGIN_001`
- `R_REGISTER_002`
- `R_TRADE_003`

### 策略命名

- `LOGIN_RISK_POLICY`
- `REGISTER_POLICY`
- `TRADE_RISK_POLICY`

### 决策动作命名

- `PASS`
- `REVIEW`
- `REJECT`
- `VERIFY`
- `LIMIT`
- `TAG`

统一命名的价值很大，因为它会让：

- 快照更清晰
- 日志更容易读
- 代码更统一
- 面试展示更专业

## 2.20 本章小结

这一章的核心，其实就一句话：

> 你不是在做一堆页面，也不是在做一堆规则，而是在搭建一套围绕 Scene、Event、Entity、Feature、List、Rule、Policy、Decision 展开的实时决策领域模型。

把要点收一下：

### 1）三大典型场景虽然不同，但底层抽象一致

- 登录风控
- 注册反作弊
- 交易风控

它们都可以抽象成：

```latex
事件输入 -> 上下文构建 -> 特征计算 -> 规则判断 -> 策略收敛 -> 决策输出
```

### 2）实时风控系统的骨架对象有 8 个

- Scene
- Event
- Entity
- Feature
- List
- Rule
- Policy
- Decision

### 3）Feature 是平台化的关键

因为原始业务语言只有在抽象成 Feature 后，规则才真正可配置、可复用、可执行。

### 4）Policy 是“平台感”的关键

Rule 只是单条判断，Policy 才负责组织和收敛。

### 5）领域模型会直接决定后续所有设计

包括：

- 数据库表
- 控制平台页面
- 快照结构
- Flink 执行链路
- 日志模型
