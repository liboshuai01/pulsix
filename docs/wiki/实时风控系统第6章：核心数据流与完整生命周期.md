## 6.1 这一章解决什么问题

前面几章，我们已经把这套实时风控平台的定位、领域模型、能力边界、总体架构和职责分层讲清楚了。

但如果你现在就开始写代码，马上还会遇到一个更“工程化”的问题：

> 这套系统里的数据，到底是怎么流动的？

也就是说，我们不仅要知道：

- 有控制面、计算面、分析面
- 有 Event、Feature、Rule、Policy、Decision
- 有 Spring Boot、Flink、Kafka、Redis、MySQL

还必须进一步搞清楚：

- 一条事件从进入系统到输出决策，完整生命周期是什么
- 配置数据是怎么从控制平台进入 Flink 的
- 实时特征是在哪里算、在哪里存、什么时候被使用的
- 决策日志和运行日志怎么从热路径流到查询系统
- Kafka 中应该有哪些 Topic
- Redis 中到底放什么
- MySQL 中到底放什么
- Flink State 中到底放什么
- 哪些是实时热路径，哪些是异步查询路径，哪些是离线分析路径

如果这一章不先讲透，后面你在设计快照、Flink 执行链路、Redis Key、日志落库时就会非常容易混乱。

所以这一章的目标，就是把整套系统真正串起来，让你第一次从“数据流和生命周期”的角度，看清楚这套实时风控平台是怎么运转的。

---

## 6.2 从静态架构图，进入动态生命周期图

前一章我们看到的，更多是一张“静态结构图”：

- 控制面在左边
- Flink 引擎在中间
- Kafka / Redis / MySQL 在下面支撑
- 分析面在另一边做查询和监控

这很重要，但还不够。

因为一个系统真正的复杂度，不在于你画了多少框，而在于：

> **这些框之间的数据，究竟是怎么在时间上流动的。**

所以从这一章开始，你要开始建立一种新的系统视角：

> **不要只看“有哪些模块”，而要看“数据从哪里来、经过哪里、如何变化、最终流向哪里”。**

在实时风控系统里，这种动态视角尤其重要。因为这个系统本质上不是静态配置系统，而是一个：

- 持续接收事件
- 持续接收配置变更
- 持续维护状态
- 持续输出决策
- 持续积累日志
- 持续反哺策略优化

的在线决策系统。

所以后面你应该用“流”的方式来理解它，而不是只用“表”和“页面”的方式来理解它。

---

## 6.3 一套实时风控系统里，通常有哪几类核心数据流

如果把这套系统里所有最关键的数据都抽象出来，一般可以分成 5 类流：

1. **原始事件流**
2. **配置发布流**
3. **特征流 / 特征结果流**
4. **决策结果流**
5. **日志与指标流**

这 5 类流的职责不同、实时性不同、承载介质也不同。

我们逐类讲。

---

### 6.3.1 原始事件流

原始事件流是整个系统的起点。

它承载的是业务侧真正发生的行为事件，例如：

- 登录事件
- 注册事件
- 交易事件
- 提现事件
- 改密事件
- 绑卡事件

它们来自：

- 前端 / App 埋点通过 `HTTP / Beacon -> Nginx -> pulsix-ingest`
- 业务后端通过 `Pulsix SDK(Netty) -> pulsix-ingest`
- API 推送 / 模拟器
- CDC 订阅后转换成事件

这类流的特点是：

- 吞吐高
- 连续不断
- 强实时
- 是决策触发源
- 格式必须标准化

这句话不要轻描淡写带过。因为在真实平台里，业务侧上传的原始事件往往还要先经过一层轻量预处理，至少包括：

- 字段映射
- 时间 / 金额 / 枚举格式转换
- 公共字段补齐
- 必填字段校验
- 脏数据分流到错误流 / DLQ

只有这样，后面的特征、规则、策略执行才是真正建立在一份稳定的标准事件之上。

你后面所有实时特征、规则、策略，最终都围绕这类流展开。

---

### 6.3.2 配置发布流

配置发布流不是业务事件，而是控制平台发布后的运行时配置变更流。

它承载的是：

- 新的场景运行快照
- 新版本策略
- 新的规则定义集合
- 新的特征执行计划
- 生效时间、版本号、checksum 等控制信息

这类流的特点是：

- 吞吐低
- 但非常关键
- 需要版本语义
- 需要广播到所有 Flink 并行实例
- 需要支持恢复和重建本地编译缓存

配置流本质上是：

> **控制面把设计态逻辑推送给计算面的通道。**

---

### 6.3.3 特征流 / 特征结果流

这一类最容易被误解。

因为很多人一提到特征，就以为一定要有“独立的 feature topic”。

实际上，特征流有两种存在方式：

#### 方式 A：内联计算，不单独出流

Flink 在处理某条事件时，直接：

- 更新状态
- 得到当前特征值
- 放进 EvalContext
- 继续执行规则

这种方式下，特征结果只是运行时上下文的一部分，并不一定要发到外部 Topic。

#### 方式 B：物化输出，单独成流

某些特征可能会被：

- 多场景复用
- 多系统消费
- 落 Redis/ES/Feature Cache

这种情况下，就可以把特征结果输出成单独流，例如：

- `feature.materialized`
- `feature.snapshot`

对于你的项目来说，一期不需要把所有特征都做成独立流。更合理的做法是：

- 关键决策链路中的特征，优先在 Flink 内联使用
- 需要查询复用的热点特征，再按需物化到 Redis 或 Kafka

---

### 6.3.4 决策结果流

决策结果流承载的是每条事件最终产出的决策结果。

例如：

- PASS
- REVIEW
- REJECT
- LIMIT
- TAG\_ONLY

除此之外，结果流通常还会带一些附属信息：

- eventId / traceId
- sceneCode
- version
- hitRules
- finalScore
- latencyMs
- riskTags

这类流的特点是：

- 实时性高
- 需要对接下游业务系统
- 也是日志和分析系统的重要输入

它一般会被发送到：

- 下游业务消费 Topic
- 决策日志落库链路
- 告警与监控系统

---

### 6.3.5 日志与指标流

日志和指标虽然也是“流”，但和决策流不完全一样。

它更多承载：

- 决策明细日志
- 命中规则日志
- 异常日志
- 引擎指标
- 审计与排障信息

这类数据的特点是：

- 一部分需要准实时
- 一部分允许异步
- 主要用于分析、排障、审计、监控
- 不应该反向影响热路径

所以这一类流通常要和主决策路径解耦。

---

## 6.4 一条事件的完整生命周期：从进入系统到输出决策

现在我们从“单条事件”的视角，把整个系统串一遍。

假设有一条交易事件：

```json
{
  "eventId": "E202603070001",
  "sceneCode": "TRADE_RISK",
  "eventType": "trade",
  "eventTime": "2026-03-07T10:00:00",
  "userId": "U1001",
  "deviceId": "D9001",
  "ip": "1.2.3.4",
  "amount": 6800,
  "result": "SUCCESS"
}
```

下面是它典型的生命周期。

---

### 阶段 1：事件产生

业务系统中，一个真实交易发生。

此时它可能由：

- 交易服务直接发送到 Kafka
- HTTP 接入服务接收到请求后转发到 Kafka
- 某条 CDC 变更经过转换后变成标准事件

到这一步，事件进入了平台的“事件流世界”。

---

### 阶段 2：事件标准化与接入

为了让后续平台可以统一处理，事件需要被组织成统一模型，至少包含：

- eventId
- sceneCode
- eventType
- eventTime
- entity fields
- ext

这一阶段可能做的事情包括：

- 字段映射
- 类型转换
- 时间格式规范化
- traceId 补充
- 幂等标识生成
- 基础合法性校验

注意，这里做的是“轻量标准化”，不是复杂业务判断。

---

### 阶段 3：事件进入 Flink 引擎

`pulsix-ingest` 完成鉴权、标准化和公共字段补齐后，把事件写入 Kafka `pulsix.event.standard`。

Flink Job 从 Kafka `pulsix.event.standard` 消费到这条事件。

此时引擎会先根据：

- `sceneCode`
- `eventType`

定位当前应该使用哪套运行时快照。

也就是说：

> **同一条事件并不是“被代码写死地处理”，而是被“当前生效的场景快照”驱动处理。**

---

### 阶段 4：加载当前生效快照

Flink 在 Broadcast State 中找到：

- `TRADE_RISK` 场景当前生效的版本
- 对应的运行时快照 `snapshot_v12`

并从本地 transient cache 拿到：

- 已编译的表达式
- 已编译的 Groovy Invoker
- stream feature 执行计划
- policy 执行计划

到这一步，Flink 已经知道这条事件应该怎么处理。

---

### 阶段 5：构建基础上下文

引擎先把事件本身转成基础上下文，例如：

```json
{
  "eventId": "E202603070001",
  "sceneCode": "TRADE_RISK",
  "eventType": "trade",
  "eventTime": "2026-03-07T10:00:00",
  "userId": "U1001",
  "deviceId": "D9001",
  "ip": "1.2.3.4",
  "amount": 6800,
  "result": "SUCCESS"
}
```

这个阶段的上下文还只是“原始上下文”，还不是规则最终要用的完整上下文。

---

### 阶段 6：计算流式特征

接下来，Flink 根据当前快照里的 stream feature 执行计划，计算当前事件所需的实时特征。

例如：

- `user_trade_cnt_5m`
- `user_trade_amt_sum_30m`
- `device_bind_user_cnt_1h`

这些特征依赖：

- Keyed State
- Timer
- 桶化状态
- TTL / 清理逻辑

例如得到：

```json
{
  "user_trade_cnt_5m": 3,
  "user_trade_amt_sum_30m": 18800,
  "device_bind_user_cnt_1h": 4
}
```

这一步是 Flink 在整个系统里最核心的作用之一。

---

### 阶段 7：查名单 / 查画像 / 查外部 lookup 特征

光有流式特征还不够，风控还需要外部上下文。

于是引擎再按需访问 Redis 或其他在线存储，拿到 lookup 特征，例如：

- `device_in_blacklist = false`
- `user_risk_level = H`
- `ip_geo_risk_tag = HIGH`

注意：

- 小配置可以 Broadcast
- 大名单不应该全量广播
- 大部分 lookup 适合 Redis + 本地缓存

---

### 阶段 8：计算派生特征

在基础字段、流式特征、lookup 特征都准备好以后，Flink 可以继续计算派生特征，例如：

- `high_amt_flag = amount >= 5000`
- `trade_burst_flag = user_trade_cnt_5m >= 3 && amount >= 5000`

这部分通常更适合用：

- 表达式引擎
- 少量 Groovy

因为它本质上是“上下文上的逻辑变换”，而不是“底层状态演化”。

---

### 阶段 9：执行规则与策略

到这一步，完整的 `EvalContext` 基本已经准备好了。

比如：

```json
{
  "amount": 6800,
  "userTradeCnt5m": 3,
  "deviceBindUserCnt1h": 4,
  "deviceInBlacklist": false,
  "userRiskLevel": "H",
  "highAmtFlag": true,
  "tradeBurstFlag": true
}
```

然后开始：

1. 执行规则
2. 收集命中规则
3. 进入策略收敛
4. 计算最终动作

例如：

- R001：黑名单设备直接拒绝 -> 未命中
- R002：高金额且高频交易 -> 命中
- R003：高风险用户多账号设备 -> 命中

策略模式如果是 `FIRST_HIT`，可能取第一条高优先级命中规则；
如果是 `SCORE_CARD`，可能累计分数后得到 `REJECT`。

---

### 阶段 10：输出决策结果

引擎最终生成 `DecisionResult`，内容可能包括：

- finalAction
- hitRules
- score
- version
- hitReasons
- latencyMs
- traceId

这个结果会被发送到：

- 决策结果 Topic
- 业务消费方
- 日志落库链路

---

### 阶段 11：异步记录日志与指标

为了不阻塞热路径，日志和监控数据通常异步处理。

例如：

- 决策明细日志 -> Kafka -> MySQL / Doris
- 指标 -> Prometheus
- 异常日志 -> 告警系统

这一步非常重要，因为它决定了：

> **系统是否既能低延迟执行，又能后续查清楚“为什么这么决策”。**

---

### 阶段 12：结果被分析面消费，形成策略优化闭环

最后，控制平台和分析系统会消费这些决策与日志数据，供：

- 查询
- 报表
- 规则命中分析
- 版本效果评估
- 策略优化

至此，一条事件的生命周期才真正闭环。

---

## 6.5 不只是事件有生命周期，配置也有完整生命周期

除了业务事件，这套系统里另一个非常关键的生命周期是：

> **配置生命周期**

也就是一条规则/策略从“被编辑”到“被 Flink 真正执行”的全过程。

这个生命周期和事件生命周期一样重要。

---

### 阶段 1：策略人员编辑配置

在控制平台中编辑：

- 场景
- 特征
- 名单
- 规则
- 策略

这些配置此时都还是“设计态数据”。

---

### 阶段 2：控制平台做校验与依赖分析

在发布前，控制平台要检查：

- 表达式是否正确
- Groovy 是否通过校验
- 特征引用是否存在
- 变量依赖是否闭合
- 是否有循环依赖
- 规则和策略绑定是否完整

---

### 阶段 3：编译为运行时快照

控制平台将多张设计态表编译为一份运行时快照：

- scene info
- stream features
- lookup features
- derived features
- rules
- policy
- runtime hints
- version/checksum/effectiveFrom

---

### 阶段 4：写入发布表

例如写入：

- `scene_release`
- version = 12
- snapshot\_json = {...}

这一步意味着：

> **从草稿态正式进入了版本态。**

---

### 阶段 5：通过配置流传播给 Flink

当前系统通过：

- Flink 直接订阅 MySQL CDC（读取 `scene_release` 的快照 + binlog）

把这个新版本发送给 Flink。

---

### 阶段 6：Flink 接收并准备切换

Flink 收到新快照后会：

- 解析
- 检查版本
- 构建可序列化快照对象
- 本地编译表达式/Groovy
- 生成新的运行时缓存

如果本地编译失败，应该：

- 保留旧版本继续运行
- 记录错误
- 不让整个任务直接崩掉

---

### 阶段 7：新版本生效

在满足：

- 版本合法
- 编译成功
- 生效时间到达

之后，Flink 会切换到新快照处理后续事件。

---

### 阶段 8：历史版本保留，用于回滚与追溯

运行态永远不能只保留“当前最新版本”。

还必须保留：

- 上一版本
- 历史版本
- 发布记录
- 回滚关系

因为：

- 线上结果要追溯
- 新版本要能回滚
- 仿真要可指定版本

---

## 6.6 Kafka 中通常应该有哪些 Topic

这一节非常重要，因为 Topic 设计基本就是系统数据流的骨架。

对于你的项目，我建议先按“够用、清晰、可扩展”的原则设计，不要一开始铺太多 Topic。

---

### 6.6.1 事件类 Topic

建议至少按场景或事件域拆分。

例如：

- `pulsix.event.login`
- `pulsix.event.register`
- `pulsix.event.trade`

如果你想先简单一点，一期建议直接统一一个标准事件 Topic：

- `pulsix.event.standard`

然后根据 `sceneCode` / `eventType` 在 Flink 内部分流。

#### 建议

对个人项目来说，一期推荐：

- 一个主标准事件 Topic： `pulsix.event.standard`
- 每条消息带 `sceneCode`
- 异常事件统一进入 `pulsix.event.dlq` 或 `error topic`

这样更容易起步。

后面需要再拆分即可。

---

### 6.6.2 配置发布通道（仅 MySQL CDC）

最简单的方案，是让 Flink 直接通过 MySQL CDC 读取发布表，例如：

- `scene_release`

这时不需要 Kafka 参与，只需要正确配置：

- MySQL 地址、端口、用户名、密码
- Flink CDC `server-id`
- MySQL binlog
- 复制读取权限

当前系统不保留额外的配置 Topic。

MySQL 配置快照进入 Flink 的方式只有一条：

- `scene_release -> MySQL CDC -> Flink Broadcast State`

这个可选 Topic 的特点是：

- 吞吐小
- 但语义关键
- 必须能广播到 Flink 全部并行实例

---

### 6.6.3 决策结果 Topic

建议：

- `pulsix.decision.result`

承载最终对下游可消费的结果。

适合包含：

- sceneCode
- eventId
- finalAction
- score
- version
- hitRules 摘要

---

### 6.6.4 决策日志 Topic

建议：

- `pulsix.decision.log`

这个 Topic 更偏内部分析使用，可以比结果 Topic 更详细。

例如包含：

- 原始事件摘要
- feature snapshot
- hit rules detail
- hit reasons
- engine latency
- exception info

---

### 6.6.5 死信 / 异常 Topic

建议：

- `pulsix.engine.deadletter`

用来承载：

- 反序列化失败事件
- 非法数据
- 执行异常但不想直接丢弃的数据

这对排障非常有帮助。

---

### 6.6.6 可选的特征物化 Topic

如果后面你想把某些特征独立输出，可以增加：

- `pulsix.feature.materialized`

但一期不是必须。

---

## 6.7 MySQL 中存什么

MySQL 在这套系统里不是热路径数据存储，而是“设计态与管理态存储中心”。

你可以把 MySQL 中的数据分成 4 类。

---

### 6.7.1 设计态元数据

包括：

- scene
- event\_schema
- feature\_def
- list\_set / list\_item（管理态）
- rule\_def
- policy\_def

这些数据的特点是：

- 结构化
- 需要事务一致性
- 需要被后台编辑与查询
- 不适合直接成为 Flink 运行输入

---

### 6.7.2 发布态数据

包括：

- scene\_release
- snapshot\_json
- version
- checksum
- published\_at
- published\_by
- rollback\_from\_version

这部分是控制面和计算面之间最关键的桥梁数据。

---

### 6.7.3 管理与审计数据

包括：

- sys\_user
- sys\_role
- sys\_menu
- sys\_audit\_log
- release\_record

这些数据支撑平台运营和管理。

---

### 6.7.4 查询型日志数据

你一期可以先把部分决策日志落 MySQL，特别是：

- decision\_log
- rule\_hit\_log
- simulation\_report

但要记住：

> **MySQL 更适合“查询型、管理型、相对中低频”的日志，不适合高吞吐热路径同步写。**

所以建议通过 Kafka 异步落库。

---

## 6.8 Redis 中存什么

Redis 在你的系统里主要承担“在线 lookup 与热点辅助存储”的职责。

可以把 Redis 中的数据分成 4 类。

---

### 6.8.1 名单数据

例如：

- 黑名单设备
- 高风险 IP
- 白名单用户
- 内部员工账号

可能使用的数据结构：

- Set
- String
- Hash
- Bitmap
- Bloom（后续）

这类数据特点是：

- 查询频繁
- 延迟要求低
- 数据量可能较大
- 变更相对可控

---

### 6.8.2 在线画像数据

例如：

- 用户风险等级
- 历史标签
- 用户渠道属性
- 设备画像
- 商户画像

这类数据通常来自离线系统、近线计算或外部同步。

---

### 6.8.3 热点特征缓存 / 物化特征

例如某些可复用、可查询的特征：

- 最近一次交易时间
- 用户近 24 小时交易金额汇总
- 设备 1 小时关联账号数（如做物化）

不是所有特征都适合放 Redis，只有：

- 热点查询频繁
- 可接受近实时一致性
- 有明显复用价值

的特征，才值得物化。

---

### 6.8.4 辅助型短态缓存

例如：

- 幂等去重标记
- 轻量级结果缓存
- 查询级本地缓存的共享补充

但要注意：

> **Redis 不是 Flink State 的替代品。**

窗口型、强时序、强恢复语义的状态，不要试图全部搬到 Redis。

---

## 6.9 Flink State 中存什么

这是本章非常关键的一部分。

很多人一说到“状态”，只会笼统说“用 RocksDB 存状态”。
但真正做风控引擎时，你必须非常清楚：

> **到底哪些数据应该进 Flink State，为什么要进，按什么 key 组织。**

---

### 6.9.1 Broadcast State：运行时快照

Broadcast State 里最核心的是：

- 场景 -> 当前快照
- 场景 -> 当前版本
- 场景 -> 可序列化配置对象

例如：

- `TRADE_RISK -> SceneSnapshot(v12)`
- `LOGIN_RISK -> SceneSnapshot(v8)`

注意：

- Broadcast State 适合放可恢复配置
- 不适合直接放编译后的 Groovy Class

编译结果应该放 task 本地 transient cache。

---

### 6.9.2 Keyed State：流式特征中间态

这是 Flink 在风控系统中的核心价值所在。

例如：

- 用户 5 分钟交易次数 -> 计数桶状态
- 用户 30 分钟交易金额和 -> sum 桶状态
- 设备 1 小时关联账号数 -> distinct member state
- 最近一次交易时间 -> latest value state

这些状态通常按：

- `sceneCode + featureCode + entityKey`

来组织。

例如：

- `(TRADE_RISK, user_trade_cnt_5m, U1001)`
- `(REGISTER_ANTI_FRAUD, ip_register_cnt_10m, 1.2.3.4)`

---

### 6.9.3 Timer 相关状态

为了清理窗口桶、过期 distinct 成员、触发延迟处理等，Flink 还会维护：

- 定时器注册信息
- timer 对应的辅助状态

这些也属于运行态中非常关键的中间数据。

---

### 6.9.4 中间组装状态（可选）

如果你采用“事件拆成多个特征分支，再按 eventId 聚合回收”的严格引擎模式，那么还会有：

- eventId -> 已返回的 feature results
- eventId -> 等待中的 feature count
- eventId -> 超时清理 timer

这类状态属于“事件级中间装配状态”。

一期项目如果你采用更简单的 inline 执行模型，这类状态可以少一些甚至不需要。

---

### 6.9.5 为什么这些状态要放在 Flink，而不是 Redis

因为它们通常具备下面这些特征：

- 依赖事件时间或处理时间
- 依赖窗口语义
- 需要 checkpoint 恢复
- 需要与 operator 生命周期一致
- 是高频更新中间态

这些特征非常适合 Flink State，而不适合直接搬去 Redis。

---

## 6.10 哪些数据是实时热路径，哪些是异步/查询路径

这是系统设计里一个非常重要的划分。

因为只要你把热路径和查询路径混在一起，实时性能就很容易被拖垮。

---

### 6.10.1 热路径是什么

热路径就是：

> **一条业务事件从进入引擎到产出决策结果的那条最短实时链路。**

对你的系统来说，热路径通常是：

```latex
业务事件 -> `pulsix-ingest` -> Kafka `pulsix.event.standard` -> Flink
-> 当前快照 -> State 特征计算 -> Redis lookup
-> 派生特征 -> 规则执行 -> 策略收敛
-> 输出 decision result
```

热路径的关键目标是：

- 低延迟
- 稳定
- 可恢复
- 少阻塞

所以热路径上不应该做太重的事情。

---

### 6.10.2 近实时异步路径是什么

这类路径通常紧跟热路径之后，但不阻塞热路径。

例如：

- 决策结果写日志 Topic
- 决策日志异步落 MySQL
- 指标异步上报
- 命中明细异步下沉分析库

这类路径的目标是：

- 尽快记录
- 不影响主决策链路
- 供查询与分析使用

---

### 6.10.3 查询/分析路径是什么

这类路径面向的是：

- 后台查询
- 报表
- Dashboard
- 规则命中统计
- 审计和回溯

它通常基于：

- MySQL
- Doris / ClickHouse（后续）
- Prometheus / Grafana

它不是面向“每条实时事件立刻返回”的。

---

### 6.10.4 设计态路径是什么

还有一类经常被忽略的路径，就是控制平台的设计态路径：

- 配置编辑
- 配置校验
- 依赖分析
- 快照编译
- 发布/回滚

这条路径面向的是“人”，不是“每条实时事件”。

所以它的优化目标不是低延迟，而是：

- 正确性
- 可维护性
- 可追溯性

---

## 6.11 用一张时序思维图，理解整套系统的数据流

下面我用一个文本时序图，把整套系统串起来。

```latex
策略人员 -> 控制平台: 编辑规则/特征/策略
控制平台 -> MySQL: 保存设计态配置
策略人员 -> 控制平台: 点击发布
控制平台 -> Compiler: 校验 + 依赖分析 + 编译快照
Compiler -> MySQL: 写入 scene_release
MySQL/CDC -> Flink(config): 读取 snapshot v12
Flink -> BroadcastState: 更新场景快照

业务系统 -> Kafka(event): 发送交易事件
Flink -> Kafka(event): 消费事件
Flink -> BroadcastState: 读取 TRADE_RISK 当前快照
Flink -> KeyedState: 计算 user_trade_cnt_5m 等特征
Flink -> Redis: 查询名单/画像
Flink -> RuleEngine: 执行表达式/Groovy
Flink -> PolicyEngine: 汇总最终动作
Flink -> Kafka(result): 输出决策结果
Flink -> Kafka(log): 输出决策日志
Kafka(log) -> MySQL/Doris: 异步落库
控制平台 -> MySQL/Doris: 查询日志、统计报表
策略人员 -> 控制平台: 观察效果并调优策略
```

你要逐渐习惯用这种方式看系统。

因为后面所有具体技术实现，本质上都只是这张时序图的某一段代码化。

## 6.12 生命周期视角下，哪些地方最容易出问题

这一节我们讲一些特别关键的边界和坑。

---

### 6.12.1 把编辑态当运行态

这是最常见的问题之一。

表现为：

- Flink 直接查设计态表
- 发布和草稿没有边界
- 线上逻辑不可追溯

这会直接破坏运行时稳定性。

---

### 6.12.2 把热路径和查询路径混在一起

例如：

- 决策时同步写多张 MySQL 表
- 决策时做复杂统计查询
- 决策时查后台管理表

这会明显拖慢实时决策延迟。

---

### 6.12.3 把所有特征都物化出去

不是所有特征都值得：

- 发 Topic
- 写 Redis
- 落外部系统

很多特征只需要在当前事件上下文中使用，内联计算即可。

过度物化会导致：

- 系统复杂度升高
- 数据一致性更难处理
- 存储和同步成本上升

---

### 6.12.4 把 Redis 当作 Flink State 替代品

Redis 很强，但它不适合替代：

- 窗口态
- timer 语义
- checkpoint 恢复语义
- 高更新频率中间态

这类数据更适合 Flink State。

---

### 6.12.5 日志链路设计得过重

决策日志非常重要，但不要在热路径里同步做太多。

正确思路是：

- 热路径产出最关键结果
- 详细日志异步输出
- 查询系统按需汇总

---

### 6.12.6 配置流没有版本语义

如果配置流里只是“某个 rule 改了”，而不是“某个场景发布了 v12 快照”，
后面版本一致性会非常差。

所以配置流最好以“场景快照版本”为单位传播。

---

## 6.13 从本章开始，你应该形成的系统直觉

这一章讲完以后，你应该开始建立几个非常重要的直觉。

---

### 直觉 1

这套系统不是“数据库 + 页面 + Flink”三个东西的拼接，
而是一套由多种数据流贯穿起来的在线决策系统。

---

### 直觉 2

最重要的不是“某张表怎么设计”，
而是“某类数据处于什么生命周期阶段”。

比如：

- 设计态 -> 存 MySQL 多表
- 发布态 -> 存 snapshot
- 运行态 -> 存 Broadcast State + Keyed State + Redis
- 分析态 -> 存决策日志与指标库

---

### 直觉 3

热路径必须短、稳、可恢复。

凡是不直接服务“当前这条事件决策”的重逻辑，
都要考虑移出热路径。

---

### 直觉 4

事件流和配置流同等重要。

很多人只盯着 Kafka `pulsix.event.standard`，但实际上：

- 事件流决定“发生了什么”
- 配置流决定“该怎么处理它”

这两条流在 Flink 中相遇，才形成真正的实时决策。

---

## 6.14 本章小结

这一章我们第一次从“动态流转”的角度，把整套系统串起来了。

把重点收一下：

### 1）系统里至少有 5 类核心数据流

- 原始事件流
- 配置发布流
- 特征流 / 特征结果流
- 决策结果流
- 日志与指标流

### 2）业务事件有完整生命周期

从事件产生、标准化、进入 Flink、读取快照、计算特征、查 lookup、执行规则、输出决策，到日志下沉和分析闭环。

### 3）配置也有完整生命周期

从编辑、校验、编译快照、发布、传播、Flink 切换，到历史版本保留与回滚。

### 4）不同存储承载不同生命周期的数据

- MySQL：设计态 / 管理态 / 发布态 / 查询型日志
- Redis：名单 / 在线画像 / 热点 lookup
- Flink State：流式中间状态 / Broadcast 快照 / timer 状态
- Kafka：事件流、结果流、日志流；配置流可按需接入 Kafka

### 5）必须区分热路径与查询路径

热路径负责低延迟决策，查询与分析链路必须异步解耦。
