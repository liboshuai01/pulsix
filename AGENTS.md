# AGENTS.md

## 项目定位

本仓库是**实时风控平台**，不是普通管理后台，不是 demo，不是单一规则引擎，也不是“在通用后台上加几条表达式”的项目。

平台核心链路固定为：

`事件 -> 特征 -> 规则 -> 策略 -> 决策 -> 日志 -> 发布 / 回滚`

后续所有 AI 助手、自动化工具、脚本生成器进入本仓库后，都必须围绕这条主链路工作。任何不服务于该链路、或破坏该链路边界的改动，均视为错误改动。

本仓库的目标是构建一套由以下部分组成的完整风控系统：

- `Spring Boot` 控制面
- `Flink` 计算面
- `Redis / Kafka / MySQL` 基础设施
- 可被仿真与线上复用的共享执行内核

禁止将本项目按“通用 CRUD 后台”思路理解和扩展。风控平台的正确性、可解释性、可回放性、可发布性、可回滚性，优先级高于页面数量、接口数量和代码堆叠速度。

---

## 总体架构边界

### 1. 设计态与运行态必须分离

控制面维护的是**设计态对象**；计算面执行的是**运行态快照**。

- 设计态对象：场景、事件模型、特征定义、名单、规则、策略、发布单、版本元数据
- 运行态对象：发布后生成的 `SceneSnapshot`、运行时事件、特征值、命中结果、决策结果、审计日志

**Flink 只能消费运行态快照，不能直接读取设计态多表拼装执行。**

### 2. 控制面、计算面、分析面必须分离

- 控制面负责“定义、校验、编译、发布、回滚、仿真”
- 计算面负责“消费、计算、执行、输出”
- 分析面负责“日志、命中、指标、回放、版本效果分析”

禁止把分析逻辑塞进计算主链路，也禁止把计算逻辑回灌成控制面临时脚本。

### 3. 共享执行内核必须纯净

共享执行内核用于保证**仿真与线上一致**。因此它必须满足：

- 纯输入 / 纯输出
- 可重复执行
- 无外部 IO
- 无框架副作用
- 不绑定 Spring MVC、MyBatis、Flink Runtime、远程服务 SDK

共享执行内核应放在可复用边界内，不得被实现为某个控制面 Service 的私有逻辑，也不得被实现为某个 Flink Operator 的临时代码。

### 4. 发布必须版本化

本平台上线方式不是“改表即生效”，而是“**编译快照 -> 生成版本 -> 发布 -> 引擎切换 -> 可回滚**”。

任何绕过版本、绕过快照、绕过发布记录的做法，都破坏风控平台的可追溯性，禁止采用。

## 控制面 / 计算面 / 分析面的职责划分

### 控制面

控制面主要对应：

- `pulsix-server`
- `pulsix-module-risk`
- `pulsix-ui`
- 与控制面直接相关的 `pulsix-module-system`、`pulsix-module-infra`

控制面职责只包括：

- 场景、事件、实体、特征、名单、规则、策略等设计态对象管理
- 表达式校验、依赖分析、引用检查、冲突检查
- 运行态快照编译
- 版本发布、版本切换、版本回滚
- 仿真、回放入口
- 控制台与 API 输出

控制面**不负责**：

- 承担实时主决策吞吐
- 在请求链路中模拟 Flink 状态语义
- 直接替代引擎执行线上实时链路

### 计算面

计算面主要对应：

- `pulsix-engine`
- 与执行复用相关的 `pulsix-framework/pulsix-kernel`

计算面职责只包括：

- 消费事件流
- 消费快照版本流
- 维护流式状态
- 计算流式特征
- 执行规则与策略
- 产出决策结果、命中明细、运行日志

计算面**不负责**：

- 维护设计态多表
- 直接调用控制面 Service/Mapper
- 临时拼接后台管理库表作为运行时事实源
- 在规则执行阶段做外部 IO

### 分析面

分析面职责包括：

- 决策日志查询
- 命中明细查询
- 版本效果对比
- 回放 / 复盘 / 审计
- 指标与监控展示

分析面**不负责**：

- 回写引擎热路径状态
- 侵入规则执行链路
- 通过报表逻辑影响线上判定结果

## 模块边界要求

仓库模块边界按以下原则执行：

- `pulsix-framework/pulsix-kernel`：共享执行内核、运行时上下文、稳定契约、纯执行逻辑
- `pulsix-engine`：Flink 作业、状态管理、流式特征、快照切换、实时决策链路
- `pulsix-module-risk`：风控控制面领域对象、发布中心、仿真入口、控制面 API
- `pulsix-server`：Spring Boot 启动与聚合，不承载风控领域绕行逻辑
- `pulsix-ui`：控制台，不得创造后端未定义的风控语义
- `pulsix-module-system` / `pulsix-module-infra`：基础支撑，不得反向定义风控领域模型

任何新代码都必须先判断自己属于哪个平面、哪个模块、哪个边界；不允许“先放进去再说”。

## 明确禁止事项

以下行为一律禁止：

- 禁止让 `Flink` 直接读取设计态多表并在运行时拼装规则
- 禁止让 `pulsix-engine` 直接依赖控制面 `Mapper`、`Entity`、后台管理 `Service`
- 禁止把流式特征实现成无边界的临时脚本、随意 Groovy、随意 SQL 字符串
- 禁止在规则层、策略层、共享执行内核中发起外部 IO
- 禁止把 Redis / MySQL / HTTP 查询直接塞进规则表达式求值过程
- 禁止通过“改配置表即生效”替代版本化发布
- 禁止跳过测试与验证，直接堆代码
- 禁止一次任务同时大改多个平面、多个模块、多个链路
- 禁止未更新文档事实源就先改架构
- 禁止未更新版本矩阵就擅自升级依赖、运行时、镜像、框架版本
- 禁止把通用后台字段、页面表单模型直接当作风控运行时模型
- 禁止提交“看起来能跑”“理论上可行”但未验证的结果
- 禁止伪造验证结果、伪造测试结论、伪造接口契约
- 禁止因为脚手架默认值、上游模板、最新版本推荐而改动核心边界
- 禁止进行无任务目标支持的全仓格式化、全仓重构、批量重命名

如任务确实要求突破上述任一边界，必须先补充或更新 `ADR` 与 `contracts`，再进入实现。

## 版本矩阵引用方式

版本矩阵唯一事实源为：

- `docs/adr/0001-version-matrix.md`

所有 AI 助手必须遵守以下规则：

- 依赖版本、运行时版本、镜像版本、工具链版本，统一以该文件为准
- 当 `README`、`pom.xml`、`package.json`、脚本、注释、示例命令与该文件冲突时，以该文件为准
- 任何版本升级都不是“顺手修改”，而是**架构变更**
- 版本升级流程固定为：**先更新 ADR，再更新代码 / 构建 / 部署文件，再执行验证**
- “使用最新版本”“与上游模板保持一致”不是合法升级理由

## 文档事实源与优先级

本仓库不是“代码即唯一事实源”的项目。风控平台必须以文档化约束先行。

事实源优先级如下：

### 1. 需求与验收事实源

- `docs/requirements/requirement-matrix.md`

用于定义：

- 当前阶段要解决什么问题
- 哪些能力已进入范围
- 哪些能力暂不进入范围
- 每项任务的验收口径

### 2. 架构与边界事实源

- `docs/adr/*.md`

用于定义：

- 架构边界
- 版本矩阵
- 快照模型
- 特征边界
- 执行上下文边界
- 关键技术决策

### 3. 契约事实源

- `docs/contracts/scene-snapshot.schema.json`
- `docs/contracts/risk-event.schema.json`
- `docs/contracts/decision-result.schema.json`
- 以及后续新增的 contract 文件

用于定义：

- 运行时输入输出
- 快照结构
- 事件结构
- 决策结果结构
- 兼容性边界

### 4. 说明性设计文档

- `docs/方案/*`

这些文档用于帮助理解设计意图和章节化蓝图，**可以指导实现，但不能覆盖 requirement matrix、ADR、contracts 的正式约束**。

### 5. 示例与进度文件

- `docs/examples/*`
- `docs/progress/*`

这些文件只用于示例和过程沟通，**不是正式事实源**。

### 执行规则

- 如果事实源缺失或仍是占位文件，禁止 AI 自行脑补完整业务语义
- 如果任务会改变边界、契约、版本或发布方式，必须先更新事实源，再写实现
- 如果代码与已接受的事实源冲突，默认视为代码落后于设计，而不是文档自动失效

## 开发工作流要求

每次任务必须遵守以下工作流：

### 1. 先 `plan`

在动手前，必须先完成以下动作：

- 识别任务属于控制面、计算面、分析面还是共享内核
- 明确本次只改哪些模块，不改哪些模块
- 明确是否涉及 requirement matrix、ADR、contracts 变更
- 明确验证方式和验证命令
- 将任务拆成最小可交付步骤

没有 plan，不得直接开始大段编码。

### 2. 再 `code`

编码时必须遵守：

- 优先最小改动
- 优先补齐边界和契约，再补实现
- 优先单模块闭环，再扩展跨模块协作
- 优先主链路正确性，再考虑便捷写法
- 优先复用共享执行内核，不做控制面 / 计算面双份实现

### 3. 最后 `verify`

所有任务在交付前必须完成验证。未验证的内容必须明确标记为“未验证”，不得包装成完成态。

## 每次任务的输出格式要求

每次 AI 助手完成任务后，输出必须至少包含以下内容：

### 1. Scope

说明：

- 本次任务目标
- 所属平面
- 触达的主链路位置
- 明确未触达的边界

### 2. Changes

说明：

- 修改了哪些文件
- 修改了哪些模块
- 是否改动了 requirement matrix / ADR / contracts
- 是否影响发布、回滚、快照、日志、决策结果

### 3. Verify

说明：

- 实际执行了哪些命令
- 实际验证了哪些行为
- 哪些验证通过
- 哪些验证未做，以及原因

### 4. Risk

说明：

- 剩余风险
- 未覆盖场景
- 兼容性注意事项
- 是否需要后续补测试、补文档、补迁移

### 5. Next

说明：

- 推荐的下一步最小任务
- 是否建议继续做下一个小闭环

禁止只给“已完成”或“大概好了”式输出。

## 必须执行的验证要求

验证不是可选项，而是交付的一部分。

### 文档类改动

至少验证：

- 文档路径正确
- 文档引用未失效
- 与 `docs/adr/0001-version-matrix.md` 不冲突
- 与已有 contracts 命名一致

### 控制面改动

至少验证：

- 相关模块可编译
- 新增 / 修改字段与 contract、snapshot 模型一致
- 发布 / 回滚语义未被绕开
- 关键服务存在单元测试或最小集成验证

### 共享执行内核改动

至少验证：

- 相同输入得到相同输出
- 不依赖外部 IO
- 不依赖 Spring 容器状态
- 规则 / 策略执行可被单元测试直接覆盖

### 计算面改动

至少验证：

- 快照加载 / 切换路径正确
- 事件输入到决策输出链路可最小化验证
- 流式特征状态更新符合预期
- 决策结果和日志字段与 contract 一致

### 契约或 Schema 改动

至少验证：

- 示例数据可通过新契约解释
- 字段命名、必填性、版本兼容性已说明
- 变更是否影响历史版本读取已说明

### SQL / 基础设施改动

至少验证：

- DDL / 配置变更范围明确
- 对控制面、计算面、分析面影响已说明
- 不会把运行时热路径重新拉回设计态表查询

### 通用规则

- 优先执行最小范围验证，再执行更大范围验证
- 不能运行验证时，必须写明阻塞原因与未验证范围
- 未经验证，不得声称“可用”“完成”“可上线”

## handoff 要求

交接信息必须足够让下一位 AI 助手或工程师继续工作，而不是重新猜测上下文。

每次 handoff 必须明确：

- 本次任务解决的问题是什么
- 改动落在哪个平面、哪个模块、哪条链路
- 哪些文件是关键入口
- 哪些文件是配套变更
- 实际执行过的验证命令与结果
- 未完成项与已知风险
- 是否涉及快照契约、事件契约、决策结果契约变更
- 是否涉及发布、回滚、灰度、兼容性或迁移要求
- 明确说明“本次刻意没有改什么”

如果任务跨越多个模块，必须按模块分开说明，不得一句话笼统带过。

---

## 适用于 Codex / 通用 AI coding agent 的行为约束

所有 AI coding agent 必须遵守以下行为规则：

- 进入仓库后先读本文件，再读与任务相关的 requirement matrix、ADR、contracts
- 不得把自己默认训练语料中的“后台项目经验”直接套到本仓库
- 不得因为看到现成脚手架代码，就假定风控域边界可以随意借用
- 不得为了“快速完成”而绕开发布、快照、契约、版本边界
- 不得进行无关文件修改
- 不得进行无根据的全仓搜索替换、全仓升级、全仓格式化
- 不得伪造测试执行、伪造运行结果、伪造接口返回
- 不得在未确认事实源前发明字段、事件、规则语义、状态语义
- 不得把外部 IO 塞进规则求值过程
- 不得把 Flink 作业写成“能查后台库就查后台库”的实现
- 不得把 stream feature 做成无生命周期、无状态边界、无契约的脚本黑盒
- 必须优先保证可解释、可审计、可回放、可回滚
- 必须把改动控制在最小必要范围
- 必须在输出中显式说明验证情况
- 必须在边界不清时先补文档，再补代码

## AI 助手进入仓库后的第一步动作清单

1. 阅读根目录 `AGENTS.md`
2. 判断任务属于控制面、计算面、分析面还是共享执行内核
3. 阅读对应事实源：
   - `docs/requirements/requirement-matrix.md`
   - `docs/adr/0001-version-matrix.md`
   - 相关 `docs/adr/*.md`
   - 相关 `docs/contracts/*`
4. 明确本次只改哪些模块，明确不改哪些模块
5. 检查任务是否涉及：
   - 快照结构
   - 事件契约
   - 决策结果契约
   - 发布 / 回滚语义
   - 版本矩阵
6. 先写 plan，明确验证方式，再开始编码
7. 编码后必须执行 verify，并在 handoff 中写清结果

未完成以上清单前，不得开始大规模实现。
